## Shopify 在 Cyber Monday 登入系統掛了

2025 年 11 月，黑色星期五購物季的最高峰——Cyber Monday。

這一天全球 Shopify 平台預計處理 **142 億美元**的銷售額。對每一個 Shopify 商家來說，這是一年中最重要的一天，可能是全年收入的 20–30%。

那天早上流量開始飆升，Shopify 的登入認證系統掛了。

商家打開瀏覽器，看到登入失敗。後台進不去。訂單看不到。庫存管理不了。客服系統也依賴後台，全部失聯。

奇怪的是，**顧客的購物頁面和結帳流程還能用**。商品可以買，但商家不知道有哪些訂單進來。

這個對比讓損失更精準：顧客的錢進來了，但商家完全失控。

## Barclays 在發薪日銀行系統當機

2025 年 1 月 31 日到 2 月 2 日，英國 Barclays 銀行的核心系統出現重大故障。

這個時間點是英國的月底發薪日。數百萬英國人在這幾天領薪水、付房租、轉帳給家人。

故障讓用戶：
- 看到帳戶餘額顯示為 0 或完全看不到
- 無法完成轉帳
- 刷卡被拒絕
- 無法繳交各種定期帳單

Barclays 最終為這次故障支付了 **750 萬英鎊的補償**，每位受影響的客戶最多賠 100 英鎊。

更重要的是無形的損害：信任。一個讓你在最需要它的時候失去它的銀行，你還信任它嗎？

## 為什麼是最壞的時機？

兩個事件有一個共同點：**在系統壓力最大的時間點，失效了一個平常看起來很穩定的元件。**

Shopify 的問題是登入認證系統。平常跑得好好的，因為平常沒有幾十萬商家在同一個時間登入。

Barclays 的問題是核心交易系統。平常跑得好好的，因為平常沒有幾百萬人在同一天發薪日同時查餘額、轉帳、繳帳單。

這不是巧合，這是統計必然：**系統在最高負載的時候，暴露的是在低負載時看不見的脆弱點。**

## 壓力測試測的是對的東西嗎

Shopify 的案例暴露了壓力測試的一個常見盲點：

**我們測了主要流程，但忘記測邊緣元件在高峰時的行為。**

電商平台的壓力測試通常會測：商品瀏覽、加入購物車、結帳、付款。這些都是顧客的流程，也是最顯眼的流程。

但商家的後台登入、管理介面、訂單處理——這些在流量低的時候平常跑，沒有人想到它們在 Cyber Monday 那天也會有幾十萬次同時存取。

**問題不是壓力測試沒做，是測試的流量模型不完整。**

一個完整的壓力測試流量模型應該包含：
- 顧客的使用流程（Browse → Cart → Checkout）
- 商家的後台流程（Login → Dashboard → Order Management）
- 系統的內部流程（Authentication service、Session management、Notification queue）

每一條路徑在高峰時都有它自己的峰值，不一定跟主要流程同步。

## 發布時機選擇：QA 能介入的地方

這兩個事故都有一個值得討論的問題：**為什麼這些變更沒有設定「不能在這個時間點上線」的限制？**

Barclays 在發薪日前後的系統變更，應該有人問過：「這個時間窗口安全嗎？」

但可能沒有人問，或是問了但沒有人有權力說不。

QA 通常被定義為「測試是否符合規格」，但**發布時機評估也是品質管理的一環**。

QA 可以在發布計畫中加入時機評估的維度：

| 問題 | 說明 |
|------|------|
| 這次上線有沒有已知的高風險時間點衝突？ | 節日、發薪日、重大活動、其他系統的維護窗口 |
| 如果上線後出問題，回滾需要多少時間？ | 這段時間是否在業務可接受範圍內？ |
| 這次變更影響的元件在高峰期的負載是否已測試？ | 不只是主要流程，包含所有相關元件 |
| 有沒有 feature flag 可以快速關閉新功能？ | 不需要回滾整個版本 |

這些問題不是在阻止上線，是在確保上線的時機和準備都是對的。

## 高風險時間窗口的應對策略

如果你的產品有明確的高峰期（電商的節日、金融的發薪日、遊戲的賽季更新），可以建立一個「凍結窗口」制度：

```
高風險時間窗口定義（範例）
────────────────────────
🔴 完全凍結（絕對不上線）
   - 月底前後 3 天（發薪日）
   - 重大節日前 48 小時
   - 行銷活動執行期間

🟡 需要額外審批
   - 月中（相對低風險）
   - 非核心功能的小改版

🟢 正常流程
   - 低流量時段
   - 純 bugfix（有完整 rollback plan）
```

這個框架讓「什麼時候可以上線」有明確的答案，而不是每次都靠個人判斷。

對 QA 來說，把這個框架帶進 release checklist，是讓品質保護從「測試結果」延伸到「上線判斷」最直接的方式。
