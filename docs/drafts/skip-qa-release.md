## 一行 config 讓付款掛了 {#一行-config}

Sprint 快結束，RD 說：「這個只改一行 config，把付款超時時間從 30 秒改成 60 秒，不用測。」

PM 說好，直接上 production。

隔天早上客服開始收到投訴：付款頁面轉圈圈，等一分鐘才顯示失敗。

原因是那個 timeout 值同時被付款 SDK 和前端 UI 的 loading state 共用。60 秒的 timeout 讓使用者以為系統當掉，紛紛重試——每次重試都對第三方付款服務發了新的請求，導致重複扣款。

一行 config，三個連鎖反應。QA 沒有看過，也沒有人問過 QA。

## 為什麼大家覺得跳過 QA 合理 {#為什麼合理}

每次跳過 QA 測試上線，都有一套聽起來很合理的理由：

**「改動很小」**
只改一行、只加一個欄位、只換一個文案。小改動感覺風險低，測試顯得浪費時間。

**「時間不夠」**
Sprint 最後一天才做完，下次 Sprint 要進新功能，現在不上就要等兩週。

**「這是 hotfix」**
線上有 bug，要快速修，沒時間走正常流程。合理。但「hotfix 可以跳過測試」這個規則開始被套用到所有緊急的事情上。

**「功能開關關著」**
新功能先上 code，開關關著，等下次再開。開關開的時候誰在測？

這些理由每一個單獨看都說得通。問題是它們開始被當成通行證，讓跳過 QA 變成一個「正常選項」。

## Hotfix 心態怎麼蔓延到正常功能 {#hotfix-蔓延}

Hotfix 本來是例外。例外需要快速，快速需要跳過一些步驟，跳過步驟就接受了更高的風險。

這在真正的緊急情況下是合理的取捨。

但當「緊急」的定義開始鬆動——deadline 很近算緊急、PM 說要快算緊急、RD 覺得改動小算緊急——hotfix 心態就從例外變成了預設。

工程團隊開始形成一個默契：**重要的功能才走完整測試，「小的」可以跳**。

但沒有人定義過什麼叫小。

## QA 不在決策鏈裡的根本問題 {#決策鏈}

上線的決定通常是這樣做的：

```
RD 說「好了」→ PM 說「上吧」→ 上線
```

QA 在哪裡？

多數情況下，QA 是事後知道——收到上線通知、在 staging 環境看到新版本、或者從客服的 bug 單裡發現。

這不是 QA 的問題，是流程設計的問題。

**QA 沒有在上線決策的那一刻被諮詢**，意味著 QA 的判斷從來不是「要不要上線」的輸入，只是「上線後有沒有問題」的後備。

這個結構讓 QA sign-off 成為可選的，不是必要的。只要 RD 和 PM 都覺得 OK，上線就發生了。

## Retro 說了也沒用的原因 {#retro-失效}

幾乎每個跑 Scrum 的團隊，在 retro 裡都說過：「要讓 QA 測完才能上線。」

然後下一個 Sprint 又有人說「這個小改動不用測」。

Retro 解決不了這個問題，原因很具體：

**Retro 是對話，不是授權**。說了「要讓 QA 測完才上線」，誰有權力在下次上線前擋住那個沒測的 PR？沒有人。

**行動項目沒有 owner**。「QA 要參與上線決策」是誰的工作？通常沒有人被指定負責建立這個流程，這句話就只是一句話。

**deadline 壓力高過 retro 共識**。承諾是在低壓力的回顧會議裡做的，但真正的壓力在 Sprint 末、在上線前一小時。那個時刻，retro 的共識贏不過「時間不夠」。

## 能實際改變的是什麼 {#能改變什麼}

只有一個地方可以真正改變這件事：**讓 QA sign-off 變成上線的必要條件，不是選項**。

具體來說：

**在 release checklist 加入強制欄位**

```
上線前確認清單
──────────────
□ PR 已 Code Review
□ CI 通過
□ QA sign-off（必填，跳過需填寫原因並由 PM + RD 雙簽）
□ Staging 驗證
□ Rollback plan 確認
```

sign-off 不是 QA 簽一個名字，是 QA 確認「我知道這個改動做了什麼，我認為可以上線」。

**定義「可以跳過 QA」的條件**

不是所有東西都需要完整測試。把可以跳過的條件寫清楚：

- 純文案修改（有截圖對照）
- 已上線功能的 typo fix
- 純後端邏輯，有完整單元測試覆蓋，無前端影響

其他情況預設需要 QA。

**讓 QA 在 PR 流程就參與，而不是等到上線前**

PR 被 merge 的時候 QA 收到通知、在 staging 就可以看到變更。不用等到上線前才知道有什麼東西要測。這讓「QA 沒時間」的理由失效——因為 QA 從 merge 那天就開始看了。

---

跳過 QA 的代價不是「可能有 bug」，而是「確認 bug 存在的人不在上線決策的現場」。

把 QA 放進決策鏈，不是增加流程，是讓已知風險被看見。
