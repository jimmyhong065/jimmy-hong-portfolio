## 我以為我說清楚了

有一次 sprint review，PO 問我：「這個功能測得怎麼樣？」

我說：「我跑了登入、帳號建立、還有幾個邊界值，regression 那邊有一個 edge case 在 iOS 16 以下會有問題，不過 happy path 都是綠的，主要流程 coverage 大概 80%，另外有一個 flaky 的 test 我還在看是不是環境問題——」

PO 打斷我：「所以可以上線嗎？」

我愣了一秒。

「可以，但那個 iOS 16 的問題要確認一下影響範圍。」

她點點頭，繼續下一個議題。

我走出去才意識到：她要的答案是五個字，我給了她一段話，裡面一半的詞她聽不懂，重點在最後才出現。

## 五個讓人聽不懂的原因

那次 sprint review 讓我開始回頭想，我在溝通上到底哪裡有問題。整理下來大概是這五件事。

### 一、沒有先說結論

我習慣從「我做了什麼」開始說，然後說「我發現了什麼」，最後才說「所以結論是什麼」。

問題是大多數人問問題的時候，要的是結論，不是過程。

PO 問「這個功能測得怎麼樣」，她要的是「可以上線 / 不可以上線 / 有條件上線」，不是測試過程的報告。

我把答案藏在最後，她必須聽完整段才知道重點在哪——如果她還有耐心聽完的話。

### 二、QA 用語對 RD/PO 來說像外語

「regression」「edge case」「happy path」「flaky test」「coverage」——這些詞對 QA 來說是日常語言，對 PO 來說可能是模糊的技術術語，對剛入職的 RD 來說也不一定全懂。

我在說「edge case 在 iOS 16 以下有問題」的時候，PO 腦子裡可能在想：「edge case 是什麼？iOS 16 以下有多少用戶？嚴不嚴重？」但她沒有問出來，因為我還在繼續說下去。

用對方聽得懂的語言說話，不是降格，是有效溝通。

### 三、跳躍式思考，對方跟不上

我腦子的運作方式是：A → D，然後回頭補 B，再跳到 F，然後說「所以 C 才是問題」。

在我腦子裡這是有邏輯的，因為我知道 A 和 F 的連接是什麼。但對方沒有這條連接線，他只看到我跳來跳去。

最常出現的場景是開會討論 bug：「這個問題在帳號登入那邊，對，就是上次那個 OAuth 的情況，不過主要問題其實是 API timeout 的處理，因為前端那邊沒有 retry，所以才會——」

會議室裡有人開始滑手機了。

### 四、現場被問到，腦子當機

PO 突然問：「這個 bug 會影響多少用戶？」

我知道我需要先查資料，但現場沒有數字，我大腦瞬間空白，開始說「呃，我需要看一下，應該是……應該不多，但我不確定……」

說不清楚，又不想沉默，就開始說一些模糊的話填空。結果比沉默更糟，因為我說的東西沒有根據，反而讓對方更不確定。

### 五、說話太快，容易插話

這個我自己知道，但很難控制。

講到有想法的地方就想馬上說出來，有時候對方還沒說完我就插進去。

結果有兩個：一是對方覺得我沒在聽，二是我說的東西因為還沒想完就說出口，說出來的不是最好的版本。

## 怎麼調整

這些問題我沒有全部解決，但有幾個調整讓溝通明顯變好了。

### 先說結論，再說根據

任何問題，先給結論。

> ❌ 「我跑了登入、帳號建立、邊界值……coverage 80%……有一個 iOS 16 的問題……可以上線，但要確認一下。」
>
> ✅ 「可以上線，但 iOS 16 以下有一個已知問題，影響範圍需要確認後決定是否 hotfix。」

說完結論，對方如果想知道細節，他會問。不想知道，他也拿到了他要的答案。

這個原則叫 BLUF（Bottom Line Up Front）。我把它貼在我的測試筆記本封面。

### 翻譯 QA 術語

我現在對自己有一個要求：說 QA 術語之前，先問「這個詞對面的人聽得懂嗎？」

幾個常用的翻譯：

| QA 說的 | 對 RD/PO 說的 |
|---|---|
| edge case | 少見但存在的使用情境 |
| regression | 這個改動有沒有影響到舊功能 |
| happy path | 最標準的使用流程 |
| flaky test | 自動化測試有時通有時不通，環境問題 |
| coverage | 這個功能我們測了幾個情境 |

不是每個詞都要翻譯，但如果對方不是 QA，我會有意識地少用縮寫和術語。

### 說話之前先在腦子裡排一下順序

跳躍式思考很難根治，但可以在說話之前用三秒鐘想一下：「我要說的事情，哪個是 1，哪個是 2，哪個是 3？」

尤其是在重要的討論前，我會先用一句話在腦子裡組好：「問題是 X，原因是 Y，建議是 Z。」

說出來的順序就是這樣，不插其他東西。

### 當機了就說「我需要查一下再回你」

現場被問到沒有答案，不要硬撐。

「我需要查一下，今天下班前給你回覆」比「呃……應該……我不確定……」好得多。

一開始我以為說「我不知道」會讓人覺得我沒準備，但實際上「我知道我不確定，我去確認再給你答案」讓人更信任你，不是更不信任。

### 插話：等對方說完那句話

我給自己一個規則：對方的句子沒有說完，不開口。

不是等到對方說完整段，就只是那句話。

這個規則執行起來比聽起來難，因為有時候對方說的話會讓我立刻想到什麼，那個想法又會消失。

我的做法是：想到的東西先在紙上記一個關鍵字，等對方說完再說。大多數時候記下來就夠了，不需要插話打斷整個對話。

## 結尾

這些問題我現在還是會犯，只是頻率少了。

比較根本的改變是：我開始把「對方聽不聽得懂」當成我的責任，而不是對方的問題。

以前我覺得「我說得很清楚啊，他們聽不懂是他們的問題」。後來我發現，我說的東西在我腦子裡是清楚的，但從我嘴巴出來傳到對方耳朵裡，中間有很多失真。消除那個失真，是說話的人要做的事。

QA 的工作本來就是讓品質問題變得可見、可理解、可行動。把這個邏輯用在溝通上也一樣：讓你的觀察變得可見、讓你的判斷可理解、讓你的建議可行動。

說清楚，也是 QA 工作的一部分。
