## 那張被關掉的 ticket

那天我在測印花兌換功能。

流程很簡單：用印花在商店兌換植物。我有 520 枚印花，開始兌換：

1. 買石蒜 →150 枚，成功，剩 370 枚
2. 買彩虹安 →150 枚，成功，剩 220 枚
3. 買海芋 →110 枚，**失敗**，App 顯示「印花不足」，但我明明還有 220 枚

有規律的：連續兌換兩顆植物，第三顆一定失敗。

我開了一個 ticket，把這個流程寫下來，然後順手補充：「另外，領取獎勵頁快速連點領取按鈕也有問題，可能會多扣。」

接下來的事情讓我理解了什麼叫做「看不懂的 bug report」。

RD Matt 問我：「那你說的多扣是按哪裡會多扣？」

我回答：「兌換確認 dialog 上的『兌換』按鈕。」

Matt：「那你說的多扣是兌換確認 dialog 的按鈕按很多次也會多扣？」

我再想了一下，回：「我剛有在兌換確認 dialog 試過多按幾次，不會多扣印花。」

這時候我才意識到：我自己也不確定最初觀察到的「多扣」是在哪裡、怎麼觸發的。我只是把一個還沒驗證的觀察直接貼進 ticket 裡。

最後這張 ticket 的主要問題（第三顆印花不足）在 Android 上確認修復了，但那個「多扣」的問題因為我描述不清，整個調查過程花了好幾個來回。

## 哪裡出問題

回頭看那張 ticket，我犯了三個錯：

### 錯誤一：兩個問題塞進同一張 ticket

「第三顆印花不足」和「快速連點可能多扣」是兩件完全不同的事。

前者是計算邏輯的問題，後者是 UI 防重複點擊的問題。把兩個問題混在一起，RD 看 ticket 的時候，不知道重點是哪個，兩個都需要時間去理解，最後往往兩個都處理得不乾淨。

一張 ticket 一個問題。這個原則在那次事件之後我一直記著。

### 錯誤二：「多扣」這個詞沒有意義

我說「可能會多扣」，但我沒有說：

- 在哪個頁面？
- 哪個按鈕？
- 多扣多少？
- 我實際有沒有看到多扣，還是只是猜測？

「多扣」是我腦子裡的詮釋，不是 App 顯示的事實。

RD 看到「多扣」，他需要先猜我的意思，再去找對應的程式邏輯，再去重現。每猜一次錯，就是一個來回。

### 錯誤三：現象沒有用數字說清楚

主要 bug 的關鍵其實很明確：

> 剩餘 220 枚印花，兌換需要 110 枚，App 卻顯示「印花不足」。

這兩個數字（220 和 110）是整個 bug 的核心。220 > 110，不可能不足，這是不需要解釋的邏輯。

但我在 ticket 裡只寫了「顯示印花不足」，沒有把「剩餘數量」和「所需數量」並排列出來。RD 需要自己去想：到底是真的不足，還是計算錯了？

描述現象要包含數字，不要只有 App 顯示的文字。

## 好的 bug report 長什麼樣

好的 bug report 讓 RD 在還沒重現之前，就已經知道「這個 bug 在哪、是什麼、多嚴重」。

### 標題：說現象，不說感受

❌ 印花兌換有問題  
❌ 印花扣錯了  
✅ 【印花商店】連續兌換第三顆植物時，顯示「印花不足」但餘額足夠

好的標題包含三個元素：**功能模組 + 觸發條件 + 具體現象**。

RD 光看標題就能開始猜可能的程式邏輯，而不是要進去讀完才知道方向。

### 前置條件：說清楚起點

複現步驟從哪裡開始，不是從「打開 App」，而是從「讓 bug 能出現的最低條件」。

> 前置條件：帳號持有至少 410 枚印花（確保前兩次兌換成功後，第三次仍有足夠餘額）

前置條件寫清楚，RD 才能直接準備好環境，不用猜要多少測試資料。

### 複現步驟：numbered，每步一個動作

```
1. 進入印花商店
2. 選擇石蒜（需 150 枚），點擊兌換確認
3. 確認兌換成功，餘額由 520 變為 370
4. 選擇彩虹安（需 150 枚），點擊兌換確認
5. 確認兌換成功，餘額由 370 變為 220
6. 選擇海芋（需 110 枚），點擊兌換確認
```

每步一個動作。不要把兩個動作合併，不要用「然後」連接。RD 重現的時候，可以一步一步對照，找到在第幾步出了問題。

### 實際結果 vs 預期結果：用數字，不用形容詞

| | 說法 |
|---|---|
| ❌ 實際結果 | 顯示印花不足，兌換失敗 |
| ✅ 實際結果 | 步驟 6 點擊兌換後，App 顯示「印花不足」，兌換失敗。此時帳號餘額為 220 枚，海芋兌換所需為 110 枚。|
| ❌ 預期結果 | 應該要可以兌換成功 |
| ✅ 預期結果 | 220 > 110，應兌換成功，餘額扣至 110 枚 |

「印花不足」是 App 說的話，不是 bug 本身。bug 是「App 說不足，但數字告訴我足夠」。把兩個數字並排，RD 一眼看懂矛盾在哪。

### 環境：版本 + 平台

```
版本：v3.5.2
平台：iOS 17.4 / iPhone 15 Pro
是否可在其他平台重現：Android 待確認
```

同一個 bug 在不同平台可能行為不同。如果你只在 iOS 測，就說只在 iOS 確認過，不要幫 Android 下結論。

## 重寫那張 ticket

按照上面的原則，那張 ticket 應該拆成兩張，分開處理。

**Ticket 1：計算邏輯問題**

> **標題**：【印花商店】連續兌換第三顆植物時，顯示「印花不足」但餘額足夠
>
> **前置條件**：帳號持有 520 枚印花
>
> **複現步驟**：
> 1. 進入印花商店
> 2. 兌換石蒜（150 枚），成功，餘額 → 370
> 3. 兌換彩虹安（150 枚），成功，餘額 → 220
> 4. 選擇海芋（110 枚），點擊兌換確認
>
> **實際結果**：App 顯示「印花不足」，兌換失敗。帳號餘額 220 枚，海芋需 110 枚。
>
> **預期結果**：220 > 110，兌換應成功，餘額扣至 110 枚。
>
> **環境**：v3.5.2 / iOS 17.4

**Ticket 2：防重複點擊問題**

這個問題我後來另外開了一張 ticket，標題是：

> 抽到轉換印花的樹時，在領獎流程裡一直按領取按鈕，印花數量會狂加

標題已經比「多扣」好很多——有功能模組、有觸發動作、有具體現象。

但完整的 ticket 還需要補：

> **前置條件**：帳號持有印花 X 枚（記下初始值）
>
> **複現步驟**：
> 1. 抽到可轉換為印花的樹
> 2. 進入領獎流程
> 3. 在「領取」按鈕出現後，連續快速點擊 N 次
>
> **實際結果**：印花從 X 枚增加至 Y 枚（增加了 Z 枚），預期應只增加一次（+W 枚）
>
> **預期結果**：無論點擊幾次，印花只增加一次
>
> **環境**：v3.5.2 / iOS 17.4

差別在於：「印花數量會狂加」是形容詞，「從 100 增加到 340，點了 4 次」是數字。RD 看數字，不看形容詞。

## Bug 管理：開完 ticket 之後

寫好 ticket 只是第一步，接下來的管理同樣影響 bug 最後有沒有被真正修好。

### 只開能重現的

「我覺得可能有問題」不是開 ticket 的理由。

這次的「多扣」就是例子：我觀察到一個現象，但沒有驗證，直接貼進 ticket，結果浪費了自己和 RD 的時間，最後那個問題根本無法確認。

不確定能不能重現 → 先記在測試筆記，繼續測試直到穩定重現，再開 ticket。

### 追蹤：確認 fix 的版本

Bug 分派給 RD 之後，不是放著等。你需要知道：

- RD 什麼時候修完了
- 哪個版本包含這個 fix

在哪個版本測、在哪個版本關票，要明確記錄。不然下個版本你不知道要不要重測，fix 有沒有帶到也不確定。

### 關票條件：說清楚「修好」是什麼

這次的主要 bug 最後是「android 正常扣印花兌換」——但 iOS 呢？其他印花數量的邊界條件呢？

關票前要確認：

1. **原始複現步驟執行一遍，結果符合預期**
2. **平台都有覆蓋**（如果當初有說 Android/iOS 需分開確認）
3. **相關邊界條件有驗過**（例如：剛好夠、差一枚不夠）

「Android 正常」≠「這個 bug 修好了」，除非你一開始就說這個 bug 只在 Android 出現。

### 不要讓 ticket 消失在 In Progress

每週花 5 分鐘看一次你開的 ticket 清單：哪些超過 3 天沒動靜？主動去問 RD 狀況，或者確認是不是 blocked 在什麼地方。

Bug 沒有人追，就會沉。沉了就等於沒開。

## 結尾

那次事件之後，我給自己訂了一個原則：

**在開 ticket 之前，先問自己：如果 RD 明天在家遠端工作，只有這張 ticket，他能重現嗎？**

如果答案是「不確定」，就繼續補充，直到你確定他能重現為止。

Bug report 不只是記錄，是溝通。RD 花在問清楚問題的時間，是從修 bug 的時間借來的。一張好的 ticket，省的不只是你的時間，是整個團隊的時間。
