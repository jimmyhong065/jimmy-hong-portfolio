# 測試設計：用最少資源換最大測試價值

> 定位：測試設計是一門手藝，不是把功能逐條翻成 case 的抄寫工。重點在「怎麼想」，不在「寫幾條」。
> 教學哲學：每篇 hook 開場、有方法、有範例、收尾「帶得走」。不寫成 tips 清單。
> 視角：QA IC 講師。從一個真實需求出發，示範怎麼把它拆成設計得起、抓得到 bug 的 case。
> 狀態：草稿（本地）。一課一檔 `article-Axx-*.md`。

## 課程定位

- 這是一門 craft 課：談的是「怎麼設計一條 case」，從需求到技巧到取捨，把測試設計拆成可以練的動作。
- 全課貫穿一個問題：「這樣設計，最少資源換到最大測試價值嗎？」每個技巧、每次取捨都拿這句話來檢驗。
- 和品質課互補：品質課 E04 站在管理視角談「測試設計與環境」要怎麼安排，本課站在執行者視角談「這條 case 該長什麼樣」。
- 和溝通課互補：溝通課教你怎麼把模糊需求問清楚，本課接手把問清楚的需求變成具體範例與 case。

## 課綱（13 課 / 5 模組）

### 模組 A · 概念地基

- A01 什麼是「好的」測試案例：可讀、可執行、抓得到 bug
- A02 Case / Scenario / Strategy / Plan 到底差在哪
- A03 最少資源最大價值：測試設計的核心取捨心態

### 模組 B · 從需求挖（範例規格）

- B04 用範例釐清需求：上下文／行動／預期的劇本三件套
- B05 範例的資料要準備到多完整：最小必要集
- B06 多少範例才夠：是釐清需求，不是測試覆蓋

### 模組 C · 設計技巧工具箱

- C07 等價劃分 + 邊界值
- C08 決策表
- C09 狀態轉換
- C10 組合爆炸與 pairwise

### 模組 D · 取捨與優先級

- D11 風險導向：把有限時間花在會痛的地方 ＋ 何時可以停

### 模組 E · AI 放大

- E12 用 AI 放大測試設計：澄清需求、邊界窮舉、決策表、髒資料、舊 case 健檢
- E13 AI 的陷阱：只給 happy path、幻覺、覆蓋假象，為何仍需要你
