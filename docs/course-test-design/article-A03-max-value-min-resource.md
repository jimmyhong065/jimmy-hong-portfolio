# 最少資源換最大價值——測試設計的那條主線問句

先承認一件事：你永遠測不完。

時間不夠、人力不夠，而輸入組合永遠多到測不完。一個看起來很單純的功能，背後的狀態、邊界、資料組合一展開，可能就是上百上千條路徑。你再認真、再加班，都不可能把每一條都覆蓋到。這不是能力問題，是數學問題。

很多人聽到「測不完」會以為這是在替偷懶找藉口。剛好相反。測試設計這件事之所以存在，正是因為測不完——如果資源無限、時間無限，那根本不需要設計，把所有組合跑一遍就好。正因為資源有限，你才需要決定：**哪些測、哪些不測、先測哪些**。承認測不完，是測試設計的起點，不是終點。

## 測試設計的本質是取捨

既然不能全測，那剩下的問題就只有一個：手上這些有限的資源，要花在哪裡？

我用一個很簡單的心智模型來想每一條 case：

一條 case 的**價值**，大致等於「它涵蓋的風險」乘上「這個風險真的會發生、會被使用者踩到的可能性」。一個會讓使用者付不了錢的風險，乘上一個很常被走到的流程，價值就很高；一個只有在某種幾乎不可能的組合下才會出現、就算出現也無傷大雅的問題，價值就很低。

一條 case 的**成本**，則包含三塊：設計成本、執行成本，還有最常被忽略的維護成本。一條寫起來很快、跑起來也快的 case，如果每次介面一改它就壞、每個版本都要花你半天修，它的真實成本其實高得嚇人。維護成本是自動化時代最容易被低估的隱形支出。

好的測試設計，追求的是**高的「價值 ÷ 成本」比**，而不是 case 的數量，也不是覆蓋率那個數字。

這裡有個很常見的誤區：把「case 寫得很多」「行覆蓋率衝到 90%」當成自己做得好。但數量和覆蓋率都只是把分母做大，它們不保證分子——也就是真正抓到的風險——有跟著變大。你很可能只是堆了一大疊低價值的 case，跑得很辛苦，看起來很努力，實際上把資源花在不會痛的地方。覆蓋率高，不等於測得好。

## 一個表單的算術

舉個具體的例子。假設你要測一個會員資料表單，上面有十個欄位，每個欄位粗估有幾種代表性的輸入狀態（空白、超長、特殊字元、合法值……）。理論上把所有欄位的狀態交叉組合起來，可能有兩百種以上的輸入組合。

如果你想「全測」：兩百條 case，光設計就要好幾天，執行一輪假設每條五分鐘也要十幾個小時，而且只要表單一改版，這兩百條有一大半要跟著重寫。這還只是一個表單。把這個成本乘上整個系統的功能數，你會發現全測在數學上就是不可能，而且就算做得到也完全不划算。

換個做法。你先問：這個表單真正會痛的風險在哪裡？大概是這幾類——必填欄位沒擋住造成髒資料、金額或數量欄位的邊界算錯、特殊字元沒處理好造成顯示或注入問題、欄位之間的相依規則（例如選了 A 就必須填 B）。你針對這些高風險點挑出大約二十條 case，集中火力。

這二十條的設計與執行成本，大概是全測的十分之一，但它們很可能涵蓋了那九成「真的會被使用者踩到、踩到會痛」的問題。剩下那一百八十條，多半是在反覆驗證一些不痛不癢、或彼此高度重複的組合。**用十分之一的成本，換九成的價值**——這就是測試設計在做的事。

## 這條問句會貫穿整門課

接下來這門課會講到一連串技巧：邊界值分析、等價類、決策表、pairwise（成對組合）、風險導向測試，一路到怎麼讓 AI 幫你做這件事。

它們看起來各自獨立，但其實都在回答同一個問題：**怎麼用更少的 case，換到更多的價值？** 邊界值是在說「最容易出錯的點集中在邊界，與其平均撒網不如守住邊界」；pairwise 是在說「多數缺陷由兩個參數的組合就能觸發，不必窮舉所有組合」；風險導向是在直接幫你排「哪個風險先測」。每一個技巧，都是這條主線問句的一種具體答案。

所以讀後面任何一篇時，都可以拿這把尺回來量一量：這個技巧，是怎麼幫我提高「價值 ÷ 成本」比的？

## 帶得走的三個問題

不用記公式。設計每一條 case 之前，先問自己三個問題就夠了：

1. 這條 case 在抓什麼風險？（如果答不出來，這條 case 很可能沒必要存在。）
2. 這個風險痛不痛、常不常發生？（決定它值不值得你花資源。）
3. 有沒有更省的方法，涵蓋同一個風險？（同樣的保障，能不能用更少、更好維護的 case 換到？）

把這三個問題變成設計前的反射動作，你寫出來的 case 自然會往「高價值、低成本」靠。這就是整門課要練的肌肉。
