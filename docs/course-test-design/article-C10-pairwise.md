# 組合爆炸與 pairwise：54 種組合測不完，先把「兩兩」蓋滿

一個功能要支援 3 種作業系統、3 種瀏覽器、3 種語系、2 種方案——把它們乘起來就是 54 種組合。聽起來還勉強，但只要產品再多兩個參數（例如登入方式、裝置尺寸），數字立刻破百、破千。沒有任何團隊測得完這麼多組合；可是反過來只挑「看順眼的幾組」來測，心裡又毛毛的，總覺得漏掉的那幾組偏偏就是會出事的那幾組。這一篇要講的 pairwise，就是專門化解這個兩難的設計方法。

## 組合爆炸：為什麼全測不可能

先看清楚問題長什麼樣。一個功能的全組合數，等於每個參數的取值數量相乘。參數是線性增加的——你只是多寫了一行「也要支援 Safari」——但組合數卻是相乘、是指數式地長大。

3 × 3 × 3 × 2 = 54 還能勉強看；可是同樣四個參數，每個多一兩個取值，或是再加一個參數，馬上就是好幾百種。每一種組合都當成一條測試案例去跑，無論人工或自動化都吃不消。這就是組合爆炸：不是你不夠認真，是這條路本身走不通。

## 關鍵洞察：多數 bug 只牽涉一兩個參數

那為什麼 pairwise 敢只測一小部分組合？因為它站在一個被大量實證觀察支撐的事實上——**絕大多數的缺陷，是由「單一參數」或「兩個參數之間的交互」觸發的**。

也就是說，bug 通常長這樣：「Safari 上就壞」（單一參數），或是「Safari 加上日文語系才會壞」（兩個參數的交互）。至於「非得 Linux＋Firefox＋日文＋付費四個條件同時成立才會出事」這種要三個以上參數同時交互才現形的 bug，當然存在，但相對罕見。既然出事的大頭集中在一兩個參數，測試資源就該優先押在那裡。

## pairwise：讓「任兩參數的每種組合」都出現一次

順著這個洞察，pairwise（成對測試）的設計目標就很清楚了：不去涵蓋所有完整組合，而是設計一組測試，讓**任意兩個參數的每一種取值組合，都至少出現一次**。

舉例來說，OS 和瀏覽器這一對，要保證「Windows＋Chrome」「macOS＋Safari」「Linux＋Firefox」⋯⋯每一種搭配都被某一列測到；OS 和語系這一對、瀏覽器和方案這一對，也都各自被蓋滿。只要每一對參數的組合都出現過，那些「兩個參數交互才現形」的 bug 就無所遁形——而這正是 bug 的大宗。

代價是：你放棄了「三個以上參數同時的某些特定完整組合」。但根據前面的洞察，被放棄的那一塊本來就是出事機率最低的。用很少的測試數，換到絕大多數會出事的交互覆蓋，這筆帳划算得很。

至於這個「兩兩都蓋到的最小集合」要怎麼算——手算非常痛苦，列到一半就會打結。實務上交給組合測試產生器處理，PICT 這類工具丟進參數和取值就能算出來（本篇重觀念，不展開安裝操作）。

## 範例：54 種壓到 10 組

回到開頭那個例子，四個參數：

- **OS**：Windows / macOS / Linux
- **瀏覽器**：Chrome / Firefox / Safari
- **語系**：中 / 英 / 日
- **方案**：免費 / 付費

全組合 = 3 × 3 × 3 × 2 = 54 種。經過 pairwise 計算，大約 10 組左右就能讓「任兩參數的每種組合」都至少出現一次。下面是一張示意的精簡組合表（重在讓你看懂它長怎樣，不必逐格驗算到數學最佳）：

| # | OS | 瀏覽器 | 語系 | 方案 |
|---|------|---------|------|------|
| 1 | Windows | Chrome | 中 | 免費 |
| 2 | Windows | Firefox | 英 | 付費 |
| 3 | Windows | Safari | 日 | 付費 |
| 4 | macOS | Chrome | 英 | 免費 |
| 5 | macOS | Firefox | 日 | 免費 |
| 6 | macOS | Safari | 中 | 付費 |
| 7 | Linux | Chrome | 日 | 付費 |
| 8 | Linux | Firefox | 中 | 免費 |
| 9 | Linux | Safari | 英 | 免費 |
| 10 | Windows | Chrome | 日 | 免費 |

從 54 砍到 10，省下八成的測試量。重點在於這 10 組蓋掉了什麼、又放掉了什麼：

- **覆蓋了**：所有成對交互。隨手挑幾對檢查——OS×瀏覽器的九種搭配、瀏覽器×方案的六種搭配、語系×OS 的九種搭配，都能在表裡找到至少一列。任何「兩個參數一起才會壞」的 bug 都跑不掉。
- **放棄了**：三個以上參數同時的特定完整組合。例如「macOS＋Safari＋日文＋免費」這一組剛好沒被列進去；如果真有 bug 非得這四個條件同時成立才現形，這張表就會漏掉它。

這個取捨是 pairwise 的本質，不是它的瑕疵——它清楚地知道自己用八成的省力，換來的是對「成對交互」的完整防護。

## 取捨提醒：pairwise 不是銀彈

正因為它明確放棄了高階組合，pairwise 不能無腦全收。如果你**已經知道**某三個參數的特定組合是高風險——比方說過去 Linux＋Safari＋付費方案這個搭配反覆出過事——那就不能指望 pairwise 自動幫你蓋到，要手動把那一組完整組合補回測試集裡。

換句話說，pairwise 是「預設省力」的起點，不是「禁止加測」的天花板。它把資源從注定測不完的全組合裡解放出來，讓你有餘裕去手動加強那幾組真正該擔心的高風險組合。這就是它呼應這門課主線最直接的地方——pairwise 幾乎是「最少資源換最大價值」這句話的數學化身。

## 帶得走：三步用 pairwise

什麼時候該掏出這招？當你面對**多個彼此獨立的參數、全組合明顯爆量、參數之間會交互但沒有強相依關係**時，pairwise 最對症。流程就三步：

1. **列出參數與取值**——把所有會影響行為的維度和它的選項寫成清單。
2. **丟給組合產生器算 pairwise 集**——交給 PICT 這類工具產出「兩兩都覆蓋」的最小組合表。
3. **補上已知高風險的特定完整組合**——把你心裡那幾組「就是它常出事」的搭配手動加回去。

參數一多就別硬碰全組合，先把兩兩蓋滿、再針對性地補強——這樣設計，就用最少的測試數，換到了最大的交互覆蓋。
