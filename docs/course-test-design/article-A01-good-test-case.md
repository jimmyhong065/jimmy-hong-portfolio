# 什麼是「好的」測試案例？三個性質與一份自我檢核清單

打開很多人寫的第一份測試案例，內容大概長這樣：標題寫「測登入」，步驟欄一句「輸入帳號密碼後登入」，預期結果填「成功」。看起來有在測，實際上什麼都沒測到。換個人接手，他不知道要用哪個帳號、密碼正不正確、按下去之後該看到什麼畫面才算「成功」。更糟的是，這條 case 根本沒有想抓什麼問題，它只是把「登入」這個動作抄了一遍。

測試案例不是操作說明書的縮寫，它是一個「驗證假設」的動作：你先假設系統在某個情境下應該有某個行為，然後設計一條路徑去證實或推翻它。寫不好的 case，問題通常不在文筆，而在於根本沒想清楚「我在驗什麼」。

## 好 case 的三個性質

要判斷一條測試案例好不好，不必看它寫得多長，看三件事就夠了。

**可讀** 別人接手，不來問你就看得懂。前置條件、操作步驟、預期結果三段分明，不需要靠默契或對系統的熟悉度去補腦。一條好 case 應該像食譜，照著做就能重現；而不是像便條，只有寫的人自己看得懂。

**可執行** 步驟與預期都具體到可以照表操課。輸入值是明確的（用哪個帳號、打什麼密碼），不是「輸入一組資料」；預期結果是可以打勾驗證的（看到什麼訊息、跳到哪一頁），不是「成功」或「正常」這種無法判定對錯的字眼。執行的人不需要自己決定「這樣算不算過」。

**抓得到 bug** 這條 case 有明確的設計目的，針對某一個風險而存在。它不是把畫面從上到下點一遍，而是問「這裡最可能壞在哪？」然後專門去戳那個點。一條抓不到 bug 的 case，就算每次都執行、每次都過，對品質也沒有貢獻。

三者缺一不可。可讀又可執行但沒設計目的，是勞動；有目的但寫得含糊，別人重現不出來也驗不了。

## 一個登入功能，兩種寫法

先看一版常見的爛 case：

| 欄位 | 內容 |
| --- | --- |
| 標題 | 測登入 |
| 前置條件 | （空白） |
| 步驟 | 輸入帳號密碼，按登入 |
| 預期結果 | 登入成功 |

逐點拆解它爛在哪。標題「測登入」沒有指出要測登入的哪一面，是正常流程、錯誤密碼、還是帳號鎖定？前置條件空白，執行的人不知道該用一個全新帳號還是既有帳號，狀態不同結果就不同。步驟沒有具體輸入值，「帳號密碼」是哪一組？預期結果「成功」無法驗證，畫面要出現什麼才叫成功，沒人說得準。最致命的是，這條 case 沒有設計目的，它只是把功能名稱複述一遍，抓不到任何特定風險。

現在改寫成一版好 case。假設我們真正想驗的風險是：連續輸入錯誤密碼時，系統有沒有確實在第五次失敗後鎖定帳號，避免被暴力破解。

| 欄位 | 內容 |
| --- | --- |
| 標題 | 連續 5 次密碼錯誤後，帳號應被鎖定 15 分鐘 |
| 前置條件 | 已存在啟用中帳號 `qa_user@example.com`，密碼為 `Correct#123`，且該帳號目前未被鎖定 |
| 步驟 | 1. 進入登入頁，輸入帳號 `qa_user@example.com`、密碼 `wrong000`，按登入<br>2. 重複步驟 1 共 4 次（累計 4 次錯誤）<br>3. 第 5 次同樣輸入錯誤密碼 `wrong000`，按登入<br>4. 改用正確密碼 `Correct#123` 嘗試登入 |
| 預期結果 | 第 1 至 4 次顯示「帳號或密碼錯誤」；第 5 次顯示「帳號已鎖定，請於 15 分鐘後再試」；步驟 4 使用正確密碼仍無法登入，並顯示鎖定訊息 |

對照三個性質：它可讀，三段分明，誰來執行都知道從哪個狀態開始；它可執行，輸入值與每一步的預期訊息都明確到可以逐項打勾；它抓得到 bug，因為它瞄準的是「鎖定機制的次數上限與生效時機」這個具體的安全風險，而不是把登入畫面點過一輪。如果開發把上限寫成 6 次、或鎖定後正確密碼竟能登入，這條 case 會立刻翻出來。

## 帶得走：好 case 自我檢核清單

寫完一條 case，送出前花十秒問自己這幾句：

- 別人不來問我，光看這條能不能執行？
- 預期結果是可以打勾的事實，還是「成功／正常」這種模糊形容？
- 前置條件交代清楚了嗎？換一個系統狀態結果會不會就不一樣？
- 輸入值是具體的，還是「一組資料」這種要執行者自己編？
- 這一條到底在抓什麼風險？我能用一句話說出它的目的嗎？
- 如果這個功能真的有 bug，這條 case 戳得到它嗎？

最後一句是試金石。一條測試案例存在的理由不是「為了測這個功能」，而是「為了抓某一個可能的失敗」。先想清楚你怕它壞在哪，再回頭寫步驟，case 自然就會具體、可驗、有殺傷力。把功能名稱抄一遍很容易，但那不是測試設計，那只是把待辦事項換句話說而已。
