# 測試策略、計畫、情境、案例——四個詞到底差在哪？

「這個測試計畫排好了嗎？」——同一句話，PM 想知道的是哪個版本要測哪些功能，工程師聽到的是某條情境的步驟，而你腦中浮現的卻是「我們到底要不要自動化」這種大方向。會議室裡每個人都點頭，但其實各自在講不同層級的事情。

測試策略（strategy）、測試計畫（plan）、測試情境（scenario）、測試案例（case）。這四個詞天天出現，卻最常被當成同義詞混用。有人嘴上說「策略」，講的其實是一條情境；有人說「測試計畫」，攤開來看是一堆 case。詞義一旦對不上，討論就會在空轉，明明在談同一個功能，卻好像各說各話。

把這四個詞分清楚，不是為了考名詞，是為了讓會議裡的人講的是同一件事。

## 四層，由大到小

最乾淨的理解方式，是把它們看成四個同心圓，包含關係是：策略 ⊃ 計畫 ⊃ 情境 ⊃ 案例。每一層回答的問題不同。

**測試策略** 回答的是「我們這個團隊、這個產品，打算怎麼測」。要不要做自動化、單元／整合／E2E 各佔多少比重、什麼樣的結果算 pass、什麼算 fail、上線的品質底線在哪。這是最高層、也最穩定的一層——它通常半年、一年才動一次，不會因為這次多了一個小功能就改寫。

**測試計畫** 往下一層，範圍縮到「這一次」。某個版本、某個功能要涵蓋哪些範圍、誰來測、什麼時候測完、有哪些風險與依賴。它是高層概述，重點在框出邊界與資源，不需要細到每個步驟。

**測試情境** 描述的是「要驗證哪些工作流」，講得比較廣。例如「驗證使用者可以成功建立帳號」——它指出要驗證什麼價值，但還沒講具體怎麼操作、用什麼資料。

**測試案例** 是最具體的一層：明確的步驟，加上明確的預期結果。一條情境通常會拆成好幾條 case，把正常路徑、例外、邊界各自釘死。

## 少即是多

知道有四層，不代表每一層都要寫成厚厚的文件。

敏捷環境下，過度文件化反而是負債——文件愈多，愈沒人維護，愈快過期，最後變成沒人信的擺設。真正該問的是：這份文件有沒有人會讀、會用？沒有的話就別寫。

探索性測試是個好例子。它可能完全不列出傳統的 case，靠的是測試者當下的判斷與發散。但即使如此，一份簡短的 charter（這次要探索什麼、時間多長、重點放哪）仍然有用——它給了方向，又不綁死手腳。

換句話說，分清楚這四個詞是為了讓溝通對齊，不是為了交差出更多 Word 檔。層級是思考工具，不是文件清單。

## 一個註冊功能，四層走一遍

抽象講完，用「使用者註冊」這個再普通不過的功能，把四層套上去對照看看。

**策略**（一句話就夠）：註冊屬於核心轉換流程，採 E2E 自動化覆蓋主路徑，邊界與錯誤提示用整合測試補，任一主路徑失敗即視為 release blocker。

**計畫**（這次版本的範圍）：本次涵蓋 email 註冊與驗證信流程，不含第三方登入（下個版本）；由我負責，預計兩天；風險是驗證信服務尚未進 staging，需要 mock。

**情境**（一條工作流）：驗證使用者能以有效資料完成註冊並啟用帳號。

**案例**（把情境釘死）：

- 用一個未被使用過的 email 註冊，密碼符合規則 → 建立成功，導向驗證信頁面。
- 用一個已註冊過的 email 註冊 → 被擋下，並提示「此信箱已註冊，請改用登入」。
- 密碼少於最小長度 → 被擋下，欄位顯示明確錯誤訊息。

同一個功能，從「我們怎麼測」一路收斂到「點哪個按鈕、看到什麼字」。四層不是四種文件，是同一件事在不同解析度下的樣子。

## 帶得走：聽到這個詞，它其實在哪一層

下次開會聽到有人丟出這幾個詞，先在心裡對一下它落在哪層，再決定要不要追問：

| 你聽到的 | 它在問什麼 | 判斷問句 |
| --- | --- | --- |
| 策略 | 整體怎麼測、品質底線 | 「這是會跨版本都成立的方針嗎？」 |
| 計畫 | 這次的範圍、資源、風險 | 「這是針對某個版本／功能的安排嗎？」 |
| 情境 | 要驗證哪條工作流 | 「這描述的是一段使用者流程嗎？」 |
| 案例 | 具體步驟＋明確預期 | 「這條有寫清楚操作和該看到什麼嗎？」 |

當對方說「測試計畫好了沒」，你可以反問一句：「你是想知道這版本要測哪些範圍（計畫），還是某條流程的具體步驟（案例）？」——一句話就把模糊的詞拉回它該在的層級，溝通成本立刻降下來。
