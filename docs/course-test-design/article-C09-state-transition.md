# 狀態轉換：訂單能不能從「已出貨」跳回「待付款」？bug 都藏在這

一張訂單從「待付款」變「已付款」、再變「已出貨」、最後「已完成」——這種有狀態的東西，最危險的 bug 往往不在任何一個狀態裡，而藏在狀態與狀態之間的那條線上。能不能從「已出貨」跳回「待付款」？一張已經取消的訂單，金流又非同步地把款項扣成功了會怎樣？這些跳轉聽起來都不該發生，但只要程式沒有明確擋下，系統就會默默放行，然後你的資料庫裡就會出現一張「已取消但已付款」的訂單，對帳的人欲哭無淚。狀態轉換測試，就是專門用來逼出這類問題的設計方法。

## 什麼東西適合用狀態轉換測

判斷標準很簡單：這個對象有沒有「生命週期」？凡是它會在幾個明確的階段之間移動，而且移動是由特定事件觸發的，就適合。訂單是最典型的例子，此外像帳號（啟用／停用／鎖定）、訂閱（試用／生效／逾期／取消）、審核流程（草稿／送審／通過／退件）、購物車、工單，全都有生命週期。只要你在需求裡看到「⋯⋯之後才能⋯⋯」「⋯⋯狀態下不能⋯⋯」這種句子，就是狀態轉換登場的訊號。

## 方法：畫出狀態與轉換，兩種 case 都要設計

第一步是把所有**狀態**列出來，第二步是把所有會觸發轉換的**事件／動作**列出來。有了這兩份清單，就能畫出一張「狀態 × 事件 → 下一個狀態」的表，這比畫狀態圖更不容易漏，因為表格的每一格都強迫你回答：在這個狀態下、發生這個事件，系統到底該往哪走？

接著是這個方法真正的價值所在——兩種 case 都要設計：

- **合法轉換**：該走得通的路徑，每一條都走一遍，確認系統正確地把對象帶到下一個狀態。
- **非法轉換**：不該發生的跳轉，系統必須擋下、給出正確的錯誤提示，而不是默默允許、或是直接崩潰。

多數人只測前者——把 happy path 走順了就收工。但嚴重的資料一致性 bug，幾乎都躲在後者。非法轉換最常被漏，偏偏又最容易藏大問題，這正是狀態轉換測試比「隨手測一測」高明的地方。

## 範例：訂單生命週期

假設一套訂單系統有五個狀態：**待付款**、**已付款**、**已出貨**、**已完成**、**已取消**。觸發轉換的事件有四個：付款、出貨、確認收貨、取消。規則是：付款後才能出貨、出貨後才能確認收貨；取消只允許在「待付款」或「已付款」時進行，一旦「已出貨」就不能再取消。

把它攤成一張狀態 × 事件表，每一格填「轉到哪個狀態」，不該發生的就標「擋下」：

| 目前狀態＼事件 | 付款 | 出貨 | 確認收貨 | 取消 |
|--------------|------|------|---------|------|
| 待付款 | → 已付款 | 擋下 | 擋下 | → 已取消 |
| 已付款 | 擋下 | → 已出貨 | 擋下 | → 已取消 |
| 已出貨 | 擋下 | 擋下 | → 已完成 | 擋下 |
| 已完成 | 擋下 | 擋下 | 擋下 | 擋下 |
| 已取消 | 擋下 | 擋下 | 擋下 | 擋下 |

表格一攤開，兩類 case 就清清楚楚。

**合法轉換清單**（表中有箭頭的格子）：

- 待付款 →（付款）→ 已付款 →（出貨）→ 已出貨 →（確認收貨）→ 已完成，這是主幹路徑。
- 待付款 →（取消）→ 已取消。
- 已付款 →（取消）→ 已取消（這條要驗付款後取消有沒有正確觸發退款）。

**非法轉換清單**（表中標「擋下」、且容易誤放行的重點格子）：

- 已出貨 →（取消）：系統應擋下並提示「已出貨訂單無法取消」，不能讓貨都寄出了還能取消。
- 已完成 →（取消）：應擋下，已完成的交易不該被回收。
- 已取消 →（付款）：應擋下，已取消的訂單不該還能被付款——這格漏掉，就會生出「已取消但已付款」的幽靈訂單。
- 已出貨 →（付款）／已付款 →（付款）：重複付款應擋下，避免重複扣款。

光是主幹四個狀態，非法的格子就比合法的多出好幾倍——這也說明為什麼只測 happy path 會漏掉一大片。

## 價值點：強迫你面對最常被漏的那一面

狀態轉換測試最大的好處，是它用一張表**強迫你正面回答每一種「狀態 × 事件」的組合**，尤其是那些你平常根本不會主動去想的非法跳轉。你不必窮舉所有可能的操作序列（那會爆量），只要把合法路徑走通、再把表中那些「不該發生卻可能被觸發」的格子逐一驗證有被擋下，就涵蓋了狀態類 bug 的大宗。這正呼應這門課一直在問的那句話——合法路徑保證功能會動，重點非法轉換保證資料不會亂，兩者加起來，就是用最少的案例換到最大的保護。

## 帶得走：四個步驟

下次遇到有生命週期的對象，照這個順序走：

1. **列出所有狀態**——把這個對象會處在的每個階段寫下來。
2. **列出觸發轉換的事件**——每個會讓它移動的動作或外部訊號。
3. **畫 state × event 表**，在每一格標出合法的目標狀態。
4. **回頭看表中的空白格與標「擋下」的格子**＝非法轉換，逐一設計「系統應該擋下並給正確提示」的 case。

合法路徑走一遍、重點非法轉換各擋一次——這樣設計，就用最省的力氣，守住了狀態類 bug 最容易潰堤的那道防線。
