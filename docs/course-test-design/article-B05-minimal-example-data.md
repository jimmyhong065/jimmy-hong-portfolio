# 範例的資料要準備到多完整？最小必要集才是重點

剛開始用範例釐清需求的人，幾乎都會有同一個焦慮：「資料是不是要列齊全，不然漏掉了，會不會就驗不出來？」於是一個簡單的情境，被塞進了商品明細、會員等級、付款方式、訂單時間、收件人電話……能想到的全寫上去，深怕少一個就不夠嚴謹。

結果範例又臭又長。要討論的人得先從一堆欄位裡撈出重點，到底這個範例想講什麼，反而被資料淹沒了。準備的人累，看的人也累，而真正要釐清的那個行為，常常還是沒講清楚。

問題不在於不夠認真，而在於用錯了判斷標準。

## 一個原則：只列會影響「這個範例要表達的行為」的資料

範例的資料不是越多越好。它的任務只有一個——把「這個範例要表達的那個行為或邏輯」說清楚。所以該列的，是有助於說明這個情境的資料，而不是「跟這張訂單沾得上邊」的所有資料。

判斷法很簡單：把某個欄位拿掉，這個範例想講的行為或結論，會不會改變？不會，那它就是雜訊，不必列。

注意這兩種說法的差別。「相關的資料」是一個很寬的標準——一張訂單上的所有欄位，幾乎都跟這張訂單相關。但「有助於說明這個行為的資料」窄得多，它逼你先想清楚：這個範例到底要講什麼？

回顧前一篇談過的，一個需求範例有三件套——上下文（Context）、觸發的行動（Action）、預期結果（Expected Outcome）。最小必要集處理的，就是上下文裡到底要放哪些資料：只放會讓那個行動與結果成立或改變的部分。

## 範例一：修改送貨地址

一次需求討論裡，有人提了個「修改送貨地址」的範例。為了完整，他在上下文塞了：商品明細、會員身分證字號、信用卡資訊、原地址、新地址、訂單狀態、下單時間……洋洋灑灑一整段。

其他人開始焦慮：「每個範例都要列這麼多東西嗎？那一場討論列得完嗎？」

引導討論的人把問題拉回來：「這個範例要表達的行為是什麼？是『改地址』，不是驗證身分，也不是處理付款。」

順著這句話一個一個檢查：身分證字號拿掉，改地址的行為會變嗎？不會。信用卡資訊拿掉呢？也不會——這個範例又不碰收款。商品明細、下單時間，同樣不影響。最後留下來的，只有 **訂單編號** （指出要改哪一筆）跟 **新的送貨地址** （改成什麼）。

範例瞬間清爽，行為一眼就看得懂，後面要轉成驗證步驟也快。更重要的是，大家不再花時間去準備一堆對這個情境毫無作用的冗餘資料。

## 範例二：下載發票 vs 取消訂單——行動不同，資料需求也不同

最小必要集不是一張固定的欄位清單，它是跟著行動走的。同樣的背景，換了行動，要的資料就不一樣。

假設上下文完全相同：使用者已登入、停在訂單查詢頁，每一筆訂單旁邊有「下載發票」和「取消訂單」兩個按鈕。

**下載發票** 這個行為，需要的是：訂單編號、發票號碼（或由後端依訂單查出）。不需要的是：商品內容、收件地址、付款方式、訂單狀態——因為發票能不能下載，跟這張訂單寄到哪、買了什麼、用什麼付，都沒關係。

**取消訂單** 這個行為，需要的卻是另一組：訂單編號、訂單狀態（要確認還能不能取消，例如已出貨就不行）、取消原因（可選）。這時候反而不需要發票號碼、下載紀錄、商品詳細。

看出來了嗎？背景一模一樣，只因為行動不同，該列的資料就整組換掉。訂單狀態在「取消」裡是關鍵——它直接決定結果能不能成立；但在「下載發票」裡卻是雜訊。資料是跟著「要驗的行為」走的，不是套一份萬用欄位模板，到哪都照抄。

## 帶得走的判斷

下次準備範例資料前，先問自己三句話：

- 這個範例要講的行為，到底是什麼？
- 把這個欄位拿掉，結論會變嗎？不會就刪。
- 這個欄位是在描述行為本身，還是只是「剛好相關」？

把這三句問完，範例自然會收斂到最小必要集——聚焦、好討論、也省下準備冗餘資料的力氣。

最後提醒一點：最小必要集是給「釐清需求的範例」用的。它的目的是把某個行為說清楚，不是把一個功能驗到滴水不漏。等到真正要設計測試覆蓋、追求把各種資料組合都掃過時，資料的考量會完全不同——那是下一篇 B06 的主題。
