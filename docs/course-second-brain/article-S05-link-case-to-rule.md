# 把 Case 連到業務規則，讓影響範圍自己浮現

PM 在 Slack 丟了一句話：「結帳流程的折扣計算邏輯改了一下，這週要上線。」

你打開 case 清單，開始憑記憶圈範圍。折扣相關的 case——有，大概在這幾條。運費有沒有折扣？記得有，找一下。會員等級折扣呢？等一下，這個折扣是在結帳算的還是在購物車就算了？稅金的計算順序會不會受影響？

圈了二十分鐘，有了一份清單。心裡還是不踏實——萬一漏了一個？上線後最怕的就是「啊那個地方原來也會受影響，當時沒想到」。這不是能力問題，是這件事本來就不應該靠記憶算。

## 算不出回歸範圍，根源在 case 各自為政

**每條 case 都在驗某件事，但「在驗哪件事」這個資訊沒有被記錄下來。**

case 清單通常長這樣：按功能分資料夾，或按模組加 tag，每一條描述的是「怎麼驗」。但很少有地方記著：「這條 case 對應的是哪一條業務規則」「這條業務規則牽動了哪些功能流程」。

這層關係只活在資深成員的腦袋裡。改動一發生，沒有任何結構幫你把受影響的 case 找出來——只能靠人腦回想。人腦回想有兩個問題：它慢，而且它會漏。越是「看起來不相關但其實依賴同一條規則」的邊緣情況，越容易在回想裡消失。

需求分析階段如果有把業務規則拆解出來，這個問題其實在更早就埋下了解法的種子——但多數時候，那些規則拆出來之後沒有被帶進 case 的設計流程，更沒有被帶進 case 的儲存方式。兩邊一直是平行的，從未連起來。

## 解法：建立連結，而不是分更細的類別

**不是把 case 分類分得更細，是把它們跟業務規則連起來。**

想像有三種節點：

**業務規則**——那些不常改、但改了就牽動很廣的邏輯核心。折扣怎麼算、免運門檻是多少、VIP 享什麼額外權益、退款能退幾天內的訂單。這些是系統的地基，不是功能，是規則。

**功能 / 流程**——使用者實際操作的路徑。結帳流程、購物車頁、會員中心、訂單查詢。一條業務規則通常在多個功能裡都有體現：「折扣計算」不只在結帳出現，購物車也會預覽、訂單詳頁也會顯示。

**case**——具體怎麼驗。這一層大家都熟悉，通常也是數量最多的。

連結的方向很直覺：**case 指向它在驗的業務規則；業務規則關聯到它影響的功能流程**。

於是當「折扣計算邏輯」這條規則改了，你順著連結就能找到：這條規則影響了購物車、結帳、訂單詳頁三個功能流程，這三個流程對應了以下這些 case。影響範圍不再靠回想，而是**順著連結被觸發出來**。

這就是連結網的核心：它不讓你記，它讓你查。

## 不需要一開始就完整

**連結網不需要第一天就全部建好。從最痛的地方開始連，價值最快浮現。**

連結網有一個特性：稀疏的時候幾乎沒用，但在某個規則周圍累積了夠多連結之後，它就從一條孤立的筆記變成一個會把相關知識自動聚攏起來的核心節點。所以起點的選擇很重要。

不是隨機把所有 case 都補上「對應規則」欄位，而是先問：「哪些業務規則改動最頻繁？哪些規則一改、影響範圍最廣、最容易踩到沒預期到的地方？」從那裡開始。電商系統先從折扣、庫存、付款狀態轉換連起來；金融系統從計息邏輯、手續費計算先連。高風險、高改動頻率的核心規則先連上去，這幾個節點一旦有了密度，每次改動你就會立刻感受到差異——不再是「我猜大概這幾條要測」，而是「系統告訴我這幾條要測，加上我個人判斷這幾條也要補」。

需求分析時把業務規則拆開、用 user story mapping 把功能流程攤開看，都是在幫你找到這些核心節點——連結網需要的節點，在那些工作裡其實已經出現了，只是沒有被帶進 case 設計的後續流程。

## 怎麼落地

**任何能做交叉引用的方式都行——工具不是重點，能不能查到才是。**

最輕量的起點：幫每條核心 case 加一個「對應規則」欄位或標籤。TestRail 可以用 custom field，Notion 可以用 relation，Excel 加一欄，Jira 用 label 加前綴——形式不限。

重點只有一個：**每條核心 case 都答得出它在驗哪條業務規則。**

不一定需要建一張專門的「業務規則表」；一開始可以只是在 case 的備註欄寫一句「驗：折扣計算規則 v3」，或在規則的筆記裡列一個「對應 case 清單」。單向連結先做起來——有了第一條連結，查詢和維護就可以開始。雙向、完整，是後來慢慢補的。

## 連結網帶來的附帶好處

建連結的過程，會讓兩種問題自然浮出來：

**覆蓋漏洞。** 一條業務規則，沒有任何 case 指向它——那這條規則從來沒被測過，或是之前的 case 已經被刪了、改了，但連結沒人清。這個缺口，不建連結的情況下，可能永遠不會被發現。

**過期或目的不明的 case。** 一條 case，指不出它在驗哪條業務規則——它可能是過期的（規則已經不存在了，case 忘了跟著刪）；也可能是當初設計時就沒想清楚為什麼要測（只是「感覺要測」）。這兩種 case 都是雜訊，連結讓你有清楚的標準去判斷一條 case 是否還該存在。

case 清單的信任度，從「不知道裡面有多少過期的，不敢刪」，慢慢變成「每條我都知道它為什麼在這裡、對應哪條規則」。

## 帶得走

- 回歸範圍算不出來的根源：case 各自為政，「在驗哪條業務規則」這層關係沒被記錄，只活在人腦裡——人腦會漏。
- 解法不是分更細的類別，是**建連結**——case 指向業務規則，業務規則關聯功能流程；改動一觸，影響範圍順著連結浮現，不靠回想。
- 不需要第一天就完整：從**高風險、高改動頻率**的核心規則先連——那裡周圍連結夠密了，查詢才有感，先連最痛的地方。
- 落地方式工具無關：custom field、relation、標籤、備註欄都行，重點是「每條核心 case 答得出它在驗哪條規則」。
- 連結網的副產品：沒有 case 指向的規則 = 覆蓋漏洞；指不出規則的 case = 過期或目的不明——兩種問題都會自然浮出。
- **今天的最小行動**：挑一條最近改動過（或即將改動）的業務規則，把你認為對應這條規則的 case 列出來，看看：連結有沒有斷掉的？有沒有覆蓋漏洞？有沒有列了半天卻指不出任何規則的 case？
