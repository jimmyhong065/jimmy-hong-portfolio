# 維護習慣：讓第二大腦長在工作流裡，不再爛回去

三個月前，你下定決心建了一套系統。

收件位置設好了，分層定好了，零件庫也有了幾個可重用的 checklist。前兩週很勤——需求討論完順手記，踩到雷的當下馬上補。有種終於把東西整理清楚的感覺。

然後一個趕版本的衝刺來了。三週內兩個功能、設計稿改了四次，bug 一批接一批。沒空整理，就先欠著。衝刺結束之後，說要補，但下個版本又來了。

三個月後，你打開那個系統——收件位置積了七十幾條沒處理的筆記，分層亂掉了，零件庫幾個檔案不知道是不是還準。

你心想：果然這種東西撐不久。

## 爛掉的原因不是你不夠勤

**系統爛回去，幾乎都不是因為方法錯，也不是因為你不夠努力——而是因為你把建系統當成一個一次性的專案。**

專案有起點、有終點：「我要建一套知識系統」，建好，打勾，完成。但知識管理不是這樣運作的。它比較像是養一座花園——你不能蓋完就走。不澆水，花園就荒；不維護，系統就腐。

很多人的解法是「更勤奮地維護」——安排固定的清理時間，計畫每週整理一小時。這通常撐不過一個月。固定排出來的維護時間，是第一個在忙碌衝刺裡被砍掉的行程。

真正的解法不是更勤，是更輕：**讓維護輕量到「就算在最忙的衝刺也不會被放棄」，系統才活得下來。** 做法是把維護綁進你本來就會做的事，不另外開時間。

## 三個維護錨點

不是三個新習慣，是三個把維護嵌進去的現有節奏。

**第一個錨點：蒐集綁進日常。**

需求討論的當下、站會裡聽到一條資訊、測試踩到一個雷——順手丟一條進收件位置。一句話就夠，不需要整理好再放。「這個欄位驗證有個邊界條件：空字串和 null 行為不同」，就這樣，丟進去，繼續手邊的工作。

這個動作發生在工作的自然中斷點，不需要切換狀態、不需要另外找時間。它是工作的一部分，不是在工作之外再加一件事。蒐集的門檻越低，進來的知識就越多，系統就越有後勁。

**第二個錨點：整理綁進版本收尾。**

每個版本收尾本來就有一些固定動作——retro、handoff、上線後觀察。在這個節點，順手清一次收件位置：把該分類的分類、該下沉的下沉、這版踩到的坑補進零件庫。

版本節奏就是整理節奏。一個版本一次，不積，不另外安排。

這之所以有效，是因為版本收尾本來就是一個回顧期——你的注意力本來就在「這版發生了什麼、學到了什麼」，整理系統跟這個節奏完全對齊，不是額外的切換。

**第三個錨點：回顧綁進固定時段。**

每週或每兩週，花十分鐘掃一遍整個系統：有沒有死掉的連結、有沒有零件已經長大到可以獨立一個分類、功能地圖還準不準。十分鐘，不求完美，只要有沒有跑偏的感覺。有，調一下；沒有，關掉，繼續下一件事。

這個回顧不需要很頻繁。兩週一次比一週一次更容易撐下去。目標不是讓系統完美，是讓它不要悄悄腐化而你沒察覺。

## 收尾時留一條筆記給未來的自己

有一個小但效益大的習慣，可以嵌進蒐集那個錨點裡：**每次測試告一段落，花一分鐘留一條筆記——現在卡在哪、下一步要做什麼、有什麼快忘掉的細節。**

下班前、衝刺結束前、中途被打斷去開會之前——就一分鐘，寫下「目前狀況 + 下一步」。

下次回來，不用花二十分鐘重新進入狀況。順著那條筆記就接上了。

這條筆記也是蒐集的一種：它讓中斷點的 context 不流失。你腦袋裡那個「下次要記得……」的念頭，不再依靠你真的記得，而是落在系統裡，等你回來。

## 不要追求完美結構

系統爛掉的另一個常見原因是：**花太多時間糾結結構本身，而不是使用它。**

「這條筆記應該放在哪個分類？」「這個零件的命名不夠精準，要先想清楚再放。」「等我把結構設計好再開始。」

這種完美主義的心態，讓整理變成一件消耗能量的事，累了就放棄。

一個有點亂但持續活著、持續被用的系統，遠勝一個結構漂亮但三個月沒人碰的系統。分類不精準、命名有點將就——都沒關係，因為你在用它。用的過程裡，結構會自己慢慢校準。

**追求持續，不追求完美。**

## 九篇，一個完整的迴圈

這是課程的最後一篇。

從 **S02 蒐集**開始——讓知識進來有地方落，而不是散在 Slack、Figma、嘴巴，下次需要的時候根本找不到。**S03 分層**讓進來的東西不腐爛——活的案例浮上來，死的安靜下沉，整個案例庫才有辦法被信任。**S04 可重用知識塊**讓你不再重造輪子——做過一次、整理一次，下次直接套用。

**S05** 把 case 連到業務規則，讓影響範圍自己浮現——「這次改動要回歸哪些？」不用靠記憶，連結網自己說話。**S06** 把 bug 蒸餾成測試啟發，讓痛只痛一次——同一類 bug 踩過之後，下次在測試設計就攔住它。**S07** 讓第二大腦變成 onboarding 資產——索引筆記當地圖，新人走得進來，不再依賴口耳相傳。**S08** 讓知識兌現價值——從零件組起測試計畫和報告，不從白紙開始。

這篇 **S09**，讓整套長在工作流裡——不靠意志力撐，而是靠輕量錨點讓它自己活下去。

回到 S01 那個場景：上線前三天，負責回歸的資深 QA 臨時請假，整個版本卡住。

如果知識在系統裡，如果有三個錨點讓它持續被維護著，那位 QA 請假，版本測試不會停擺——因為知識不再只住在她腦袋裡，它也住在系統裡、對其他人可見、可以被繼續使用。這就是第二大腦真正要解決的事。

## 帶得走

- 系統爛掉，幾乎都不是因為方法錯——是因為**把建系統當成一次性專案**，而不是長在工作流裡的持續習慣。
- 維護要輕量到「最忙的衝刺也不會被放棄」才能活下來。另外安排的維護時間，是第一個被砍的。
- 三個維護錨點：**蒐集綁日常、整理綁版本收尾、回顧綁固定時段**——把維護嵌進本來就會做的事，不另外開時間。
- 每次告一段落，花一分鐘留一條筆記給未來的自己：卡在哪、下一步做什麼。中斷點的 context 不流失，下次回來直接接上。
- 追求持續，不追求完美。一個有點亂但持續被用的系統，遠勝一個結構漂亮但三個月沒人碰的系統。
- **今天的最小行動**：在行事曆上排一個固定時段——每週十分鐘，或下個版本收尾時的整理時間。就先排進去，其他的等你坐下來再說。
