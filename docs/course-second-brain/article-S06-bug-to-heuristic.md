# 把 Bug 蒸餾成測試啟發，讓痛只痛一次

又是時區問題。

這季的跨日結算又因為 UTC／本地時間搞錯出包，跟去年那次幾乎一樣，只是換了個功能。工程師修掉了，開了 retro，列了根因，大家點頭稱是。三個月後，同一類 bug 換了個地方、換了個人，又出現了一次。

團隊修掉的是「這一個 bug」，不是「這一類 bug」。

## 為什麼 Retro 學不到教訓

**團隊不是不檢討，是檢討的產物沒有進入下次測試的流程。**

Retro 筆記記錄的是「上次發生了什麼」，但設計 case 的時候沒有人回頭翻 retro——不是懶，是因為它放在另一個地方，格式也不是「下次測 X 類功能要記得什麼」。它是一份事後報告，不是一份設計提示。

兩邊永遠平行：bug 在那邊累積，case 設計在這邊照舊，中間沒有橋接。所以同樣的根因每次都從零撞上——不是因為沒記錄，而是記錄的格式跟使用情境不搭。這是一個格式問題，不是一個紀律問題。

## 蒸餾的三層：從原始 Bug 到可重用啟發

**從一個具體 bug 到一條可重用的測試啟發，需要逐層往上抽，每層只留更精煉的本質。**

**第一層——原始 bug 紀錄。** 完整的 context：什麼操作、什麼環境、什麼預期、什麼實際結果。這是原料，也是最長的一層。好的 bug 報告在這裡就要清楚——紀錄模糊，後面每一層都抽不準。這也是 bug 報告值得寫清楚的另一個理由：它不只是溝通工具，更是後續蒸餾的原料；原料爛了，就抽不出根因。

**第二層——根因模式。** 抽掉這個 bug 的特定細節，問一個更大的問題：「這一類錯誤的共通結構是什麼？」不是「跨日結算的時區邏輯寫錯了」，而是「凡是牽涉時間的計算，UTC 與本地時間的轉換邊界都可能出錯」。注意這裡丟掉的東西：功能名稱、特定情境、這次出錯的細節——留下的是可以遷移到其他功能的骨幹。

**第三層——可重用的測試啟發。** 再往上壓縮一次，目標是一句設計 case 時能直接觸發的提示。例：「測到任何時間相關功能，檢查跨時區、跨日界、夏令時間三個邊界。」這句話不依附在「跨日結算」這個功能上——它在任何有時間計算的地方都成立。

每往上一層，細節更少，可遷移性更高。最後那一句，才是會在未來救你的東西。

## 蒸餾出來的啟發要進得了流程

**蒸餾出來的那條啟發，放在 retro 筆記裡等於沒蒸餾。**

Retro 文件有一個致命傷：只有在「回顧的時候」才會被翻到。設計 case 的時候，沒有人會打開 retro 找提示。

蒸餾的終點不是「被記錄」，是「會被重用」。這條啟發需要放進你的測試零件庫——和 S04 講的啟發式清單放在一起——讓它在下次設計任何時間相關功能的 case 時自然被掃到。啟發進了零件庫，它才算真的活著：不是等你去找它，而是在對的時候出現在你面前。

記錄 ≠ 沉澱。能被觸發才算沉澱。

## 不是每個 Bug 都值得蒸餾

**只有「換個地方會再來」的那類 bug，才值得花力氣抽成啟發。**

判準很簡單：這個 bug 的根因，換個功能會不會再踩到？

會的，值得蒸餾。典型的例子——時區與跨日邊界、並發與競爭條件、權限邊界、狀態殘留、空值與預設值處理——這些根因不依附在特定功能，換個地方、換個功能、換個工程師照樣踩。

不值得蒸餾的：低級筆誤、一次性的設定搞錯、特定測試環境汙染。根因解決了就消失，換個地方不會出現，不需要抽成啟發。

每次修完 bug 可以問自己一個問題：「如果這個功能是另一個人設計的，他會不會踩到一樣的坑？」會，就抽。不會，就讓它停在 bug 記錄就好。

有時候同一類 bug 反覆出現，背後還有更深一層的原因——不只是測試設計沒跟上，而是某些系統性因素讓同類問題一直被製造出來。如果你發現某條啟發需要一再被提醒，值得停下來多問一層：為什麼這類錯誤在這個團隊裡特別容易發生？測試知識的建立，有時候只是讓問題被看見，結構性的問題還需要往根源挖。

## 帶得走

- 團隊學不到教訓，不是不檢討，是**檢討產物的格式與使用情境不搭**——retro 是事後報告，不是設計提示，兩邊永遠平行，知識進不了流程。
- 蒸餾三層：原始 bug 紀錄（原料）→ 根因模式（抽掉特定細節，找共通結構）→ 可重用的測試啟發（一句設計時能觸發的話）。每層丟掉細節，留下可遷移的本質。
- 好的 bug 紀錄是蒸餾的原料——寫清楚 bug 報告，不只是溝通用，更是讓後續每一層都能抽得準。
- 蒸餾的終點是**會被重用**，不是被記錄——抽出來的啟發要放進測試零件庫，在設計 case 時自然被掃到；放在 retro 文件裡等被翻，等於沒蒸餾。
- 只對「換個地方會再來」的根因下功夫——時區邊界、並發、權限、狀態殘留、空值，跨功能都成立；一次性的低級筆誤不需要抽。
- **今天的最小行動**：翻最近一個「感覺以前踩過」的 bug，問自己：根因換個功能會不會再出現？會的話，把它抽成一句「以後測 X 類功能要檢查 Y」，放進你的啟發清單。
