# 組織：讓活的浮上來、死的下沉

那份 case 庫上次認真整理是兩年前的事了。

有人離職之前把自己的 case 全丟進去，說「以後可能用到」。之後每次 sprint 有人加幾條，有人把舊的 copy 過來改，有人在標題後面加「v2」或「（新）」。慢慢地，幾百條 case 堆在同一個地方，按功能模組分好幾個資料夾，看起來很整齊。

但沒有人知道哪些還準。

「這條 case 說要檢查 A 欄位是否必填，但我記得上個版本就拿掉了？」「這個 checklist 是哪個版本的？」沒人能回答。大家開始各做各的副本，因為原始的那份信不過。案例庫仍然存在，只是沒有人真的在用它。打開只是為了複製幾條貼到自己的試算表，然後手動比對哪些還有效。

想清理，又不敢動——萬一刪掉的那條剛好有人要用怎麼辦？於是就這樣繼續積著。

## 問題不是東西太多，是分類軸用錯了

**案例庫腐爛的根因不是數量，是分類的方式讓你看不出「這條現在還算不算數」。**

多數人按功能模組或測試類型來分——登入模組、購物車、付款流程、回歸測試、冒煙測試。這種分法直覺，搜尋起來也順。但它有個盲點：功能模組是永遠存在的分類，不管那個模組現在是不是你的工作重點，case 都靜靜躺在那裡，看不出新舊，看不出活躍度。

按主題分類，不會告訴你「這條我現在還要不要管」。

正確的軸是**行動性**——這條知識跟我現在的工作有多近。離工作越近，越該放在手邊；越遠的，自然下沉。不同的距離，需要的操作也不一樣。

## 四層，讓東西按距離自己站好

把知識的位置想成同心圓，從最近到最遠分四層。

**本版任務層。** 這個 sprint 或這個版本要測的東西。測試計畫、本次新增的 case、這版特別要注意的驗收條件。最近、最常開、每天都會碰到。數量刻意保持小——只放這版真正要動的東西。

**常駐領域層。** 長期存在、需要持續維護的測試知識。核心模組的 case、常規回歸範圍、業務規則筆記。不是每天都碰，但每個版本都會回來看一次。這裡放的是「這個產品我一直在負責」的知識。

**通用參考層。** 不綁特定任務或模組的知識。測試啟發清單、邊界值技巧、歷史 bug 摘要、跨專案都能套的共用 checklist。不知道什麼時候會需要，但需要的時候能找到就好。

**封存層。** 過期了、不再適用了，但還不到刪掉的東西。功能被砍的 case、三個版本前的測試計畫、已關閉的邊界條件。下沉到這裡，不干擾現在的工作，但保留可查的歷史。

這四層最重要的洞察是：**同一條知識會隨時間移動**。本版任務做完，測試計畫移進常駐領域；業務規則大改，舊版的 case 下沉進封存。分層不是一次性的分類決定，是給東西一個隨時間流動的軌道。

## 不要在蒐集的當下分類

看到這裡可能會想：「那我每次寫一條 case，就要想清楚它屬於哪一層？」這樣會讓蒐集的門檻又變高，S02 好不容易降下來的摩擦又會回來。

**別在收資訊的當下做分類決定。** 先進收件位置，定期批次處理。

每週或每個版本收尾，專門抽一段時間坐下來整理。對每條問一串問題：這條服務於正在進行的任務嗎？是，放本版任務層。不是的話，它屬於某個長期要顧的領域嗎？是，進常駐領域層。都不是的話，是不特定任務都可能用到的參考知識嗎？是，進通用參考層。三個問題都答否，進封存層。

批次做有個好處：不用每次都重新啟動認知切換。集中在一個時段，用同一套標準掃過去，效率高、判斷也更一致。邊蒐集邊分類是最慢的做法——把兩件不同性質的事強行綁在一起，兩件都做不好。

## 封存，而不是刪除

最反直覺的部分：過期的 case，不刪，下沉。

**心理門檻。** 「刪掉萬一以後要用」的焦慮是真實的，它讓人根本不敢動手清理。封存解決了這個問題——東西還在，只是退場了。知道可以找回來，就敢把它移走。活的東西才能浮上來；怕刪、不敢動，就永遠都是一鍋粥。

**歷史可追溯。** 那條過期的 case 裡可能藏著「當時為什麼這樣測」的邏輯。業務規則的邊界為什麼定在這裡、某個不常見的 flow 為什麼要特別測。這些判斷的原因往往不在 case 本身，而是當時的 context。封存保留了這段歷史；刪除是知識的淨損失。

過期不等於無價值。只是它的位置換了。

## 帶得走

- 案例庫腐爛的根因是**分類軸用錯**——按主題分類看不出行動性，現役和過期的東西混在一起，無從判斷。
- 正確的軸是「這條知識跟我現在的工作有多近」——分四層：**本版任務**（這版要動的）、**常駐領域**（長期要顧的）、**通用參考**（需要時翻的）、**封存**（退場但可查）。
- 同一條知識會隨時間在四層之間移動，分層不是一次性決定，是動態軌道。
- 蒐集和分類分開做：先進收件位置，定期批次問三個問題歸位。不要邊蒐集邊分類。
- 不刪、下沉——心理門檻低了，才敢清理，活的才浮得上來。
- **今天的最小行動**：打開你的案例庫，找一眼就看出過期的東西（三個版本前的測試計畫、已砍功能的 case），把它移到「封存」層。別刪，移走就好。
