# 測試的成功始於設計：怎麼制定一份能執行的測試計畫

「測試計畫」這四個字常讓人聯想到一份沒人看的 Word 文件——寫給流程交差用的。但真正有用的測試計畫不是文件，是決策：**這場仗在哪裡打、打哪些、不打哪些、什麼時候算贏。**

如果這些問題沒先想清楚就開測，你就是在沒有地圖的情況下亂走，測得再勤也只是瞎忙。

## 一份計畫要回答的五個問題

把格式忘掉，一份測試計畫的本質是回答五件事：

1. **範圍**：這次交付要測什麼、不測什麼。明確寫下「不測」的部分，跟寫「要測」一樣重要——它是你跟團隊對齊風險的地方。
2. **風險**：哪裡最可能出事、出事的代價多大。這決定了力氣往哪集中（接續上一篇的風險排序）。
3. **方法**：每一塊用什麼方式測——手動探索、自動化、效能、安全。不是所有東西都值得自動化，也不是所有東西手動測得完。
4. **資源與時程**：誰測、用什麼環境、多少時間。時間永遠不夠，所以計畫的重點是「怎麼分配不夠的時間」。
5. **進出場標準**：什麼條件才開始測（進場）、什麼條件才算測完可以上線（出場）。沒有出場標準，「測完了嗎」就永遠沒有客觀答案。

## 用四象限當覆蓋地圖

第一個問題「範圍」最容易出的錯，不是測得不夠細，而是**整類測試被遺漏**——功能全測過了，卻沒人想到效能與安全。有一張好用的地圖能擋這個漏：把測試攤在兩條軸上，橫軸是「支援團隊（幫開發把東西做出來）↔ 批判產品（站外部視角挑毛病）」，縱軸是「面向業務 ↔ 面向技術」，切成四格：

| | 面向業務 | 面向技術 |
|---|---|---|
| **支援團隊** | Q2 功能測試、需求情境驗證——確認「做對的東西」 | Q1 單元／整合測試、自動化、快速回饋——確認「東西做對」 |
| **批判產品** | Q3 探索式測試、可用性、使用者視角——挑使用者會踩的雷 | Q4 效能、安全、穩定性、相容性——用工具壓系統極限 |

規劃範圍時，逐格問一句「這格我這次有沒有測？要不要測？」。四格不是每次都要全測，但每一格都該是「想過之後的決定」，而不是「根本沒想到」。實務上最常被整格漏掉的是 Q4——功能都對，一上線被流量打垮或被掃出漏洞。

## 進出場標準：把「測完了嗎」變成可回答

「測完了嗎」之所以難回答，是因為大多數團隊沒定義過「完」。出場標準就是把這個「完」寫清楚：

- 高優先級案例全數通過。
- 沒有未解決的 P1／P2 缺陷。
- 高風險區域的覆蓋達到約定程度。
- 已知風險都有人簽字接受。

注意最後一條：出場不代表零風險，而是「剩下的風險都被知情地接受了」。資深 QA 的工作不是消滅所有風險，是讓決策者在知情的情況下做選擇。

## 計畫是活的，不是刻在石頭上

測試計畫最大的陷阱是把它當成一次性文件。實際上需求會變、時程會被砍、線上會冒出新風險。好的計畫是輕量、可調整的：

- 砍時間時，計畫告訴你「先放掉哪一塊風險最小」，而不是無差別少測。
- 冒出新風險時，計畫讓你能快速重排優先級。
- 與其寫一份 20 頁沒人讀的完美計畫，不如寫一頁能讓全隊對齊「打哪、不打哪、何時算贏」的活計畫。

## 帶得走

- 測試計畫的本質是決策，不是文件：在哪打、打哪些、不打哪些、何時算贏。
- 五個必答問題：範圍、風險、方法、資源時程、進出場標準。
- 用四象限當覆蓋地圖逐格自問，別讓整類測試（最常是 Q4 效能／安全）被遺漏。
- 明確寫下「不測什麼」，跟「要測什麼」一樣重要。
- 出場標準把「測完了嗎」變成客觀可答——剩下的風險都被知情接受了。
