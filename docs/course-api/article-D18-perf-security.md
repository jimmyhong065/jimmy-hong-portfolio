# 功能對了還不夠

到這裡，你的 API 功能測試已經很紮實了。但「功能對」只是能上線的必要條件之一。一支 API 在大促被流量打爆、或被人改個 id 就撈走整張用戶表，功能測試一條都不會紅——因為那不是它管的事。

這篇談 API 測試往兩邊延伸的交界：效能與安全。

## 往效能延伸：API 是壓測的天然對象

好消息是，你做功能測試累積的東西，幾乎能直接拿去壓測：

- 你整理過的**契約**（URL、method、body）就是壓測腳本的素材。
- 你設計過的**用戶旅程**（登入→下單→付款）就是壓測要還原的流量模型。

差別只在判準變了：功能測試問「結果對不對」，效能測試問「在 N 倍流量下，p95 還守得住嗎、錯誤率有沒有爆」。把關鍵的 API 旅程接上負載工具（例如 k6），就從功能測試跨進了效能測試。觀念與工具，可以接著看本站的效能測試課與 k6 實戰課。

這裡只記一個交界提醒：**功能 case 跑得過 ≠ 撐得住量**，這兩件事要分開驗。

## 往安全延伸：QA 該做的基本面

安全測試的專業滲透有它的深度，但有一批**基本面**，是 QA 在功能測試時就該順手帶到的。對著 OWASP API Security Top 10 的精神，至少檢查：

- **認證與授權**（B08 已專講）：越權是 API 安全的頭號風險，務必驗水平／垂直越權。
- **注入**：對輸入欄位試 SQL／NoSQL／指令注入字串，看會不會被當成指令執行，或回 `500`。
- **敏感資料外洩**：response 不該回密碼雜湊、完整信用卡號、內部堆疊錯誤訊息。故意打錯時，錯誤訊息有沒有洩漏內部實作（資料庫結構、檔案路徑）。
- **過度暴露資料**：API 是不是回了前端根本用不到的整包欄位，讓攻擊者撿現成。
- **缺乏限流**：同一支 API 能不能被無限快速地打（暴力破解、刷驗證碼），有沒有 rate limiting。
- **傳輸層**：是不是強制 HTTPS，敏感操作有沒有走加密通道。

## 劃清界線：QA vs 專業滲透

別把自己當滲透測試團隊，但也別什麼都不做。合理的分工是：

- **QA 該做**：上面這些「設計與配置層級」的基本檢查，當成功能測試的一部分，特別是越權與敏感資料外洩——這些用功能測試的手法就能抓。
- **交給專業**：深度漏洞挖掘、模糊測試、繞過防護的攻擊鏈，交給資安團隊或外部滲透。

把基本面守住，就能擋掉很大一部分最常見、最低級也最致命的漏洞。

## 帶得走

- 功能對 ≠ 能上線：還要扛得住量（效能）、守得住資料（安全）。
- 功能測試的契約與旅程可直接變壓測素材，接上 k6 就跨進效能測試（詳見效能／k6 課）。
- QA 該順手做的安全基本面：越權、注入、敏感資料外洩、過度暴露、限流、HTTPS。
- 基本面自己守、深度滲透交專業，別越界也別擺爛。
