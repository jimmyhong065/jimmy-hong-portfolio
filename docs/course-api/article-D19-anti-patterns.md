# 技術上正確、卻沒人信任的測試

有一種測試最尷尬：它會跑、會綠、覆蓋率報表也漂亮，但團隊就是不信它——紅了第一反應是「大概又是測試自己壞了」，綠了也不敢拍胸脯說沒問題。

信任是測試的全部價值。以下七個反模式，每一個都在悄悄侵蝕它。

## 一、脆弱測試：把易變值寫死

把 response 裡每個欄位都精確斷言，包括時間戳、自動產生的 id、排序——後端動一點點就紅一片。**解法**：結構交給 schema 驗、關鍵業務值精確驗、易變值只驗存在或型別（呼應 B07）。

## 二、硬編碼：URL、帳密、資料寫死在碼裡

`requests.post("https://staging-3.internal...")` 散落各處、密碼明文躺在 repo 裡。換環境要全域取代，還是資安風險。**解法**：URL／環境走設定或環境變數、機密走 secrets、共用資料集中管理。

## 三、過度驗證 vs 驗不足

兩個極端都不好：把每個欄位寫死是過度驗證（脆弱）；只 `assert status == 200` 是驗不足（漏 bug）。**解法**：驗「契約保證的」和「業務關鍵的」，放過「會合理變動的」。

## 四、測試之間有順序依賴

`test_create` 建資料、`test_update` 假設它在、`test_delete` 收尾——靠執行順序串起來。單獨跑會壞、平行跑會打架。**解法**：每支測試自己用 fixture 造資料、用完即清，彼此獨立（呼應 C14）。

## 五、只測 happy path

只驗「對的輸入回對的結果」，從不驗錯誤輸入、邊界、越權。於是上線後被各種異常輸入打爆。**解法**：異常與邊界才是 API 測試的主戰場——故意打錯該回 4xx、越權該被擋，這些都要有。

## 六、測試金字塔失衡

把所有測試都堆在最慢最脆的 e2e 層，API 層空空。跑得慢、紅得多、維護到崩。**解法**：把驗業務邏輯的重擔下放到 API 層，e2e 只留關鍵旅程（呼應 A01）。

## 七、慢測試：每支都真打第三方

每個 case 都真的去呼叫不穩的第三方、不隔離、不 mock，於是測試又慢又時好時壞。**解法**：不可控依賴用 mock 隔離、快測當 PR 門檻、真實整合測試另外排（呼應 D15／D17）。

## 一句總結

這七個反模式，最終都指向同一個後果——**測試失去信任**。而信任一旦沒了，再多的測試也只是擺設。反過來說，一套被信任的 API 測試有共同特徵：穩定、獨立、驗在刀口上、快、好維護。

## 帶得走

- 測試的價值是信任；脆弱、硬編碼、順序依賴都在侵蝕它。
- 驗在刀口上：結構用 schema、關鍵值精確驗、易變值放過。
- 主戰場是異常／邊界／越權，不是只測 happy path。
- 金字塔別失衡、慢測別卡 PR——穩定、獨立、快，測試才被信任。
