# 從契約到用例：把黑箱技術套到 API

上一個模組你已經會蒐集 API 契約了。現在問題來了：手上這份「`name` 必填、最長 50 字、`sex` 只能填 male／female」的清單，要怎麼變成一組有系統、不漏洞的測試用例？

好消息是——你不需要學新東西。API 測試的用例設計，骨子裡就是你熟悉的**黑箱測試技術**，只是換個對象套上去。

## 三個發想方向

先用三個方向把思路打開：

- **正常使用情境**：給對的輸入，預期成功。例如登入 API 給正確帳密，預期回 `200` 並拿到 token。這是 happy path。
- **錯誤使用情境**：針對每個欄位的限制，給錯的輸入，預期被擋。例如不填帳號、密碼錯誤、格式不對。
- **特殊使用情境**：業務邏輯上的特例。例如註冊 API 帳號不能重複——但如果有「註銷」功能，註銷後應該又能用同一個帳號註冊。

## 一個關鍵紀律：一次只放一個錯誤

設計錯誤情境時，**每個 case 只能有一個錯誤資料**。

為什麼？因為你要能歸因。如果一個請求同時帳號空白又密碼錯誤，回了 `400`，你不知道是哪個欄位的驗證生效了——搞不好其中一個驗證根本是壞的，只是被另一個遮住。一次錯一個，才驗得出每一條驗證規則。

## 把等價分割與邊界值套上欄位限制

每個欄位的限制，直接長成用例。以「`name`：必填、字串、最長 50 字」為例：

- **邊界值分析**：50 字（剛好通過）、51 字（剛好被擋）、1 字、空字串。長度限制的兩端，就是 bug 最愛躲的地方。
- **等價分割**：對「`sex` 只能填 male／female」，有效類是這兩個值、無效類是其他任何值（`unknown`、空、數字、超長字串），每類各抽代表測一個。
- **必填驗證**：欄位整個不給，預期回 `400`。
- **型別違反**：該給字串卻給數字、該給數字卻給字串。

## 從清單到用例表

把契約裡每個欄位，套上面的技術跑一遍，就會得到一張結構化的用例表：正常一組、每個欄位的邊界與等價各幾條、必填各一條。這比「憑感覺打幾發」完整太多，而且**可以講清楚為什麼這樣測**——這正是黑箱技術的價值。

## 帶得走

- API 用例設計不是新技術，是把黑箱測試套到契約上。
- 三方向打開思路：正常、錯誤、特殊情境。
- 鐵律：**錯誤情境一次只放一個錯誤**，才能歸因到單一驗證規則。
- 欄位限制直接長成用例：邊界值顧長度兩端、等價分割顧可填範圍、必填與型別各補。
