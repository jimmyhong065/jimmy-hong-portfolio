# 後端偷改了欄位，你上線才爆

微服務團隊都聽過這個故事：使用者服務把回應裡的 `name` 拆成了 `firstName` 和 `lastName`，沒特別通知誰。三個下游服務照舊讀 `name`，全部在上線當天才發現拿到 `undefined`。

你可能會說：那就做整合測試啊。問題是——服務一多，整合測試就**又慢又脆又組合爆炸**：要把所有相關服務一起拉起來、資料準備地獄、一個掛全部紅。這就是**契約測試（Contract Testing）**要解的問題。

## 核心想法：驗共同的契約，不用同時拉起所有服務

契約測試的精神是：消費者（呼叫方）和提供者（被呼叫方）之間有一份**契約**，雙方各自獨立地驗自己有沒有守約，不必把整個系統兜在一起跑。

- **消費者**驗：「我預期你的回應長這樣，我照這個契約寫程式。」
- **提供者**驗：「我的真實回應，有沒有違反任何消費者期望的契約。」

只要提供者那邊的契約驗證過了，它就知道「我這次改動會不會弄壞某個下游」——**在合併前就知道**，而不是上線才知道。

## 消費者驅動（Consumer-Driven）

最流行的模式是消費者驅動，代表工具是 **Pact**。流程大致是：

1. **消費者**在自己的測試裡，對一個本地的 mock 寫下期望：「我打 `GET /users/1`，預期回 `id`、`firstName`、`lastName`。」
2. 這些期望被產生成一份 **contract 檔**（pact 檔）。
3. 這份契約交給**提供者**，提供者跑一個驗證：用真實的服務去回放這些互動，確認每一條都還滿足。
4. 提供者哪天想拿掉 `firstName`，它的契約驗證會立刻紅——因為有消費者還靠著它。

關鍵在於：契約是**從消費者真實需要的東西**長出來的，所以提供者不會為了沒人用的欄位綁手綁腳，但只要動到有人用的，馬上被擋。

## 跟 schema 驗證差在哪

C13 的 schema 驗證和契約測試容易搞混，但層次不同：

- **Schema 驗證**：在你自己的測試裡，驗「這一支回應的結構對不對」。單向、單點。
- **契約測試**：驗「兩個服務之間的互動約定」，而且**提供者那端也會驗**。它管的是跨服務的相容性，不只是一支回應的形狀。

schema 驗證是契約測試的一塊基石，但契約測試多了「提供者承諾不破壞消費者」這層雙向保證。

## 什麼時候值得導入

契約測試不是免費的，它有自己的學習與維運成本。值不值得，看你的處境：

- **很值得**：多團隊、多服務互相呼叫，整合測試已經痛到不行。
- **未必需要**：單體應用、或只有少數穩定的內部 API——這時 schema 驗證加少量整合測試可能就夠了。

## 帶得走

- 微服務下「提供者偷改、消費者上線才爆」靠整合測試擋，又慢又脆。
- 契約測試讓消費者與提供者**各自獨立**驗一份共同契約，不必同時拉起所有服務。
- 消費者驅動（Pact）：消費者定義期望→產生契約→提供者驗自己沒破壞它。
- 跟 schema 驗證的差別：契約測試是**跨服務、雙向**的相容性保證。
- 多服務、整合測試已失控時最值得導入；單體或少量穩定 API 未必需要。
