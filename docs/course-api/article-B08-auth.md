# 我能不能拿 A 的帳號看 B 的資料？

功能測試大家都記得做，但有一類測試最常被整個漏掉，出事卻是上頭條等級的——認證與授權。資料外洩的新聞，很多根因就是一句話：**「改個 id 就看得到別人的資料。」**

## 先分清兩個詞

- **認證（Authentication）**：你是誰？驗的是身分。沒帶 token、token 過期或偽造，系統該認不出你。
- **授權（Authorization）**：你能做什麼？驗的是權限。就算認得出你，你也只能碰你有權限的東西。

兩者的失敗 status code 不一樣：認證失敗回 `401`（你根本沒登入或身分無效），授權失敗回 `403`（認得你，但不准你做這件事）。測試時別把這兩個混為一談。

## 認證測試：身分這關怎麼破

對需要登入的 API，逐一試這些情境，預期都該回 `401`：

- 完全不帶 token。
- 帶過期的 token。
- 帶格式錯誤或亂改過的 token。
- 帶已登出 / 已撤銷的 token。

還要驗 token 的**生命週期**：過期後是否真的失效、刷新（refresh）機制是否正確、登出後舊 token 是否立刻作廢。

## 授權測試：用權限矩陣設計

授權的用例別用猜的，畫一張**權限矩陣**：角色（admin／一般用戶／訪客）× 資源 × 操作（讀／寫／刪）。矩陣每一格都是一個用例——有權限的該成功，沒權限的該回 `403`。

這張表能逼出那些「想當然耳」的漏洞，例如「一般用戶其實也能呼叫刪除 API」。

## 最該抓的：越權漏洞

越權（Broken Access Control）是 OWASP 榜上常年的頭號風險，分兩種：

- **水平越權（IDOR）**：同層級的用戶互看資料。把 URL 裡的 `/orders/1001` 改成 `/orders/1002`，如果不是你的訂單卻看得到，就中了。**測法**：用 A 的身分，去存取 B 的資源 id，預期該被擋（`403` 或 `404`）。
- **垂直越權**：低權限碰高權限功能。一般用戶直接呼叫只有 admin 該能用的 API，如果成功，就中了。

以 `restful-booker` 為例：修改和刪除 booking 都需要帶 token，沒帶就該擋——這是認證。再進一步，「能不能帶自己的 token 去刪別人的 booking」，就是水平越權該驗的點。

## 帶得走

- 認證（你是誰，失敗 `401`）和授權（你能做什麼，失敗 `403`）要分清楚。
- 認證測試：無／過期／偽造／已登出的 token，都該被擋；並驗 token 生命週期。
- 授權測試：畫**權限矩陣**（角色 × 資源 × 操作），每格一個用例。
- 最該抓**越權**：水平越權用 A 的身分存取 B 的 id；垂直越權用低權限呼叫高權限 API。
