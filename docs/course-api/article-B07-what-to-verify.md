# 只 assert 200 的 API 測試，只驗了五分之一

打開很多人的 API 測試，斷言只有一行：`assert status_code == 200`。

這不是錯，是不夠。一支 API 回了 `200`，只證明「它沒爆」，不證明「它做對了事」。要把 API 真的驗紮實，得分層去看。

## 五個該驗的層次

**一、status code 對不對**

成功情境回對的 2xx（建立該 `201` 不是 `200`），失敗情境回對的 4xx。這是最表層的一關。

**二、response body 的內容對不對**

回傳的值是不是預期的？建立後回的 id 有沒有給、回填的欄位跟你送的一致嗎、計算欄位（總價、折扣）算對了嗎。這一層才開始驗業務邏輯。

**三、response 的結構對不對（schema）**

欄位該有的都在嗎、型別對嗎、不該多的有沒有多。手寫一個個 assert 很累又容易漏，這該交給 schema 驗證工具自動把關（C13 會專講）。

**四、副作用：系統狀態真的變了嗎**

這是**最常被漏、卻最重要**的一層。一支下單 API 回了 `201`，不代表訂單真的進了資料庫、庫存真的扣了。要驗副作用，得跳出這支 API：

- 用另一支 API 反查（建立後 `GET` 回來，確認真的存在、值正確）。
- 有權限的話直接查資料庫，確認狀態真的變了。

「回應說成功」和「系統真的做了」是兩件事。只信回應，會漏掉「回 201 但其實沒寫進去」這種最陰險的 bug。

**五、冪等與重複行為**

`POST` 重送兩次會不會建出兩筆？`DELETE` 刪兩次第二次回什麼？這層在 A04 講冪等性時鋪過，這裡要把它變成實際斷言。

## 一個完整的例子

測「下單」這支 API，紮實的驗法是：

1. `POST /orders` → 驗回 `201`。
2. 驗 response body：訂單 id 有給、金額正確、狀態是「待付款」。
3. 驗 schema：回傳結構符合契約。
4. **副作用**：`GET /orders/{id}` 反查，確認訂單真的建了；再查商品庫存，確認真的少了一件。
5. 重送同一個冪等鍵，確認不會建出第二筆。

## 別矯枉過正：避免過度驗證

驗得多不代表驗得好。如果你把 response 裡每個欄位的值都寫死成斷言，包括那些會變動的（時間戳、自動產生的 id、排序），測試會變得**極度脆弱**——後端動一點點就紅一片，最後跟 UI 測試一樣沒人信。

原則是：**驗「契約保證的」和「業務關鍵的」，放過「會合理變動的」。** 結構用 schema 驗、關鍵值精確驗、易變值只驗存在或型別。

## 帶得走

- 只 assert 200 遠遠不夠，API 要分層驗：status、body 值、schema、副作用、冪等。
- **副作用最常被漏**：用反查 API 或查 DB，確認系統狀態真的變了，別只信回應。
- 避免過度驗證：驗契約與業務關鍵值，放過會合理變動的欄位，否則測試脆弱。
