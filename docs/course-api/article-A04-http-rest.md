# status code 回 200 就算成功嗎？

很多人驗 API 只看一件事：「有沒有回 200？」這太粗糙了。要做好 API 測試，得先把 HTTP 這層底層語義搞懂——尤其是 status code，它本身就是 API 在跟你說話。

## status code 的語義家族

status code 分五大族，先記住它們各自的意思：

- **2xx 成功**：`200` OK、`201` Created（新增成功）、`204` No Content（成功但沒內容，常見於刪除）。
- **3xx 重導向**：資源搬家了，跟著跳。
- **4xx 用戶端錯誤**：是**你這邊**送錯了。`400` Bad Request、`401` 未認證、`403` 沒權限、`404` 找不到、`409` 衝突（例如重複建立）、`422` 格式對但語意不合法。
- **5xx 伺服器錯誤**：是**伺服器自己爆了**。`500` Internal Server Error、`503` 服務不可用。

## 最該記住的一條紅旗：500

這裡有個 API 測試的鐵則：**你故意輸入的錯誤資料，系統應該回 4xx（多半是 400 Bad Request），絕不該回 500。**

為什麼？因為 4xx 代表「系統正確地辨識出你輸入有問題、擋下來了、回你清楚的錯誤訊息」——這是健康的。而 500 代表「系統沒擋住、自己在內部爆掉了」——這是一個 bug。所以在異常測試裡，**只要看到 500，幾乎就是一張該開的缺陷單**，別輕易放過。

## 冪等性：重送一次會不會出事

冪等（idempotent）的意思是：同一個請求送一次和送很多次，結果一樣。這在測試裡很關鍵，因為網路會重試。

- **冪等**：`GET`（純讀）、`PUT`（整筆覆蓋）、`DELETE`（刪了再刪還是不存在）。
- **不冪等**：`POST`（送兩次通常會建出兩筆資料）。

所以測 `POST` 類的建立操作時，要特別驗「重複送出會不會產生重複資料」；而冪等的操作則該驗「重送結果一致」。

## Content-Type 與認證機制

- **Content-Type**：告訴對方 body 是什麼格式，最常見 `application/json`。送錯格式或漏帶這個 header，常會收到 4xx，這本身也是個測試點。
- **認證機制**：API 怎麼認得你是誰，常見幾種——API Key（放 header）、Basic Auth（帳密編碼）、Bearer Token（`Authorization: Bearer <token>`）、Session Cookie、OAuth。測試前要先搞清楚目標 API 用哪一種，才知道請求該怎麼帶身分。

## 帶得走

- 別只看「有沒有 200」，status code 的語義要分清 2xx／4xx／5xx。
- **故意打錯該回 4xx，回 500 就是 bug**——異常測試最該抓的紅旗。
- 冪等性決定「重送會不會出事」：`POST` 不冪等，要驗重複建立。
- 先確認 API 用哪種認證機制，請求才知道怎麼帶身分。
