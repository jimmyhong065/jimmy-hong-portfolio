# 分型別設計：四類 API 各該怎麼測

上一篇講了通用的用例設計方法。但實務上，不同型別的 API 各有各的「常出事的地方」。先把 API 歸類，再對著該類的常見坑去想，就不容易漏。

以下四類提供的是發想方向，不是完美 SOP，也不保證窮盡——目的是讓你至少寫得出最該有的那些用例。

## 第一類：查詢類 API（Search）

查詢的麻煩在於「輸入組合多、輸出狀態雜」。想這些：

- 查詢條件的**組合**：單條件、多條件交集、互斥條件。
- **分頁與排序**：第一頁、最後一頁、超出範圍的頁碼、每頁筆數邊界、排序欄位正確性。
- **空結果**：查不到時回的是空陣列還是 `404`？兩種設計都常見，要對齊契約。
- **模糊比對與特殊字元**：大小寫、前後空白、`%` `_` 萬用字元、SQL／NoSQL 注入字串。
- **資料量**：大量資料下的回應時間與正確性。

## 第二類：CRUD API（增刪改查）

CRUD 要分開測，也要**串起來測**：

- **Create**：必填驗證、重複建立（該回 `409` 還是允許？）、格式錯誤、建立後 id 正確回傳。
- **Read**：存在的、不存在的（`404`）、沒權限的（`403`）。
- **Update**：整筆更新 vs 部分更新（`PUT` vs `PATCH`）、更新不存在的資源、併發更新同一筆。
- **Delete**：刪除後再讀該 `404`、重複刪除（冪等嗎）、連帶資料怎麼處理。
- **跨操作一致性**：建立後讀得回來嗎？更新後讀到的是新值嗎？刪除後真的查不到嗎？——這是 CRUD 最該驗、卻最常被漏的一塊。

## 第三類：匯入類 API（Import）

匯入處理的是「一批資料」，坑特別多：

- **檔案格式**：正確格式、錯誤格式、副檔名對但內容錯。
- **大檔**：超大檔案的處理與逾時。
- **部分失敗**：一批一百筆，第五十筆壞了——是整批退回（全有全無），還是成功九十九筆只報第五十筆？這個策略一定要先問清楚再測。
- **重複匯入**：同一批匯入兩次會不會重複。
- **編碼與欄位對應**：UTF-8／BOM、欄位順序錯、缺欄、多欄。

## 第四類：匯出類 API（Export）

匯出是「把資料倒出來」，重點在正確與權限：

- **內容正確**：匯出的資料跟系統內一致、沒缺漏沒重複。
- **格式**：CSV／Excel／JSON 的格式正確、特殊字元跳脫（逗號、換行）。
- **大量資料**：大筆數匯出的效能與完整性。
- **權限過濾**：不同角色匯出，該只拿到自己有權限看的那部分。
- **空資料**：沒資料時匯出的是空檔還是報錯。

## 帶得走

- 先把 API 歸類，再對著該類常見坑想用例，最不容易漏。
- 查詢看條件組合、分頁排序、空結果、特殊字元。
- CRUD 除了各自測，**跨操作一致性**最該補。
- 匯入的核心問題是「部分失敗怎麼處理」；匯出的核心是「內容正確＋權限過濾」。
