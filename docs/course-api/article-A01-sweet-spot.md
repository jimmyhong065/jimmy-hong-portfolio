# 為什麼 API 測試是自動化的甜蜜點？

「我們有五百個 UI 自動化測試。」聽起來很厲害，直到後半句：「但每天紅兩百個，沒人去看了。」

這是無數團隊踩過的坑：把自動化全押在 UI 層，結果改個版面、換個按鈕 id，測試就紅一片，維護成本高到最後整套被放生。問題不在他們不努力，在於押錯了層。

## 測試金字塔：每一層的性價比不同

經典的測試金字塔由下往上是三層：

- **單元測試**：量大、極快，但只驗單一函式，離真實業務行為很遠。
- **API／整合測試**：中量、快，驗的是「服務真的做對事了嗎」。
- **UI／端對端測試**：量該最少，最慢、最脆，但最貼近用戶真實操作。

一個常被引用的建議比例是 **70／20／10**——七成單元、兩成 API／整合、一成 UI。很多團隊的問題剛好是把這個金字塔倒過來：UI 測試一大堆、API 測試很少。倒三角一定撐不住。

## 為什麼中間這層最甜

API 測試剛好卡在「夠真實」和「夠穩定」的交會點：

- **比 UI 穩**：它不綁畫面。按鈕換顏色、版面重排、前端改框架，API 測試一條都不會紅。它只在乎「契約有沒有變」。
- **比單元測試貼業務**：它打的是真實的服務、跨真實的鏈路，驗的是「下單真的會扣庫存嗎」這種業務層級的行為，不是某個函式的回傳值。
- **快**：沒有瀏覽器、沒有渲染，一支請求毫秒級回來，幾百個 case 幾秒鐘跑完，適合每次 commit 都跑。
- **定位精準**：UI 測試紅了，你不知道是前端、後端還是網路；API 測試紅了，你馬上知道是哪一支介面、哪條鏈出事。

## 一個對照

同樣測「登入」這件事：

- 走 UI：開瀏覽器、找帳號欄、填、找密碼欄、填、點按鈕、等頁面跳轉、驗畫面文字——慢、而且任何一個元素改了就壞。
- 走 API：對 `POST /auth` 丟帳密，驗回 `200` 且帶 token——一行請求、一個斷言，前端怎麼改都不影響。

兩者都該有，但你會希望「驗登入邏輯對不對」這種事，大量落在便宜又穩的 API 層；UI 測試只留幾條關鍵旅程的煙霧測試。

## 帶得走

- 自動化別把雞蛋全放在 UI 層，倒三角撐不住。
- API 測試是甜蜜點：**比 UI 穩、比單元測試貼業務、快、定位精準**。
- 把「驗業務邏輯」的重擔交給 API 層，UI 測試只留關鍵旅程。
