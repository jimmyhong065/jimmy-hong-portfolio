# App 層瓶頸：別動不動就說「機器不夠」

應用層慢，最偷懶的結論是「機器不夠，加就對了」。但很多時候加了機器也沒用——因為瓶頸根本不在機器算力，在程式碼或配置。來看四類常見的 App 層瓶頸。

## 一、執行緒池

池太小，請求排隊；池太大，上下文切換成本與記憶體爆掉。要看**活躍執行緒數、佇列長度、拒絕數**。

想知道執行緒卡在哪，用執行緒 dump（jstack / pstack）看它們在等什麼——等鎖？等 IO？答案就在那裡。

## 二、GC

- 頻繁的 Full GC 或長停頓，會讓 RT 出現週期性的尖刺。
- 記憶體洩漏的特徵是：老年代只升不降，最終 OOM——這種要靠浸泡測試跑久了才現形。

看 GC 頻率、單次停頓、堆使用曲線，決定要調堆大小、換 GC 演算法，還是去修洩漏。

## 三、下游連接池

對資料庫、Redis、第三方的 client 連接池一旦打滿，請求就卡在「等連接」——這跟前一篇講的連鎖偽裝是同一回事。每個下游都有自己的池，都要監控。

## 四、N+1 查詢

一個列表請求，觸發了 N 次子查詢（常見於 ORM 的懶載入）。單看沒事，併發一放大，這 N 倍的查詢直接壓垮資料庫。根因在應用程式碼，不在資料庫。解法是改成批次查詢、JOIN 或快取。

## 破除迷思：TPS 鋸齒不一定是 GC

很多人一看到 TPS 或 P99 呈規律鋸齒，就直接歸咎 GC 停頓。**GC 只是其中一種可能。**

有個經典的鋸齒案例，根因其實是某個資源**週期性累積到臨界點**才觸發——可能是定時清理、佇列、連接、網路參數。關鍵特徵是：降低線程數，鋸齒仍規律出現（只是週期變長）。所以遇到規律性鋸齒，先往「定時觸發 / 累積到閾值」的方向查，逐一比對，才能確定到底是不是 GC。

## 一個好用的分類：資源用不用得上

可以把性能分析的現象分成兩大類，非常實用：

1. **資源用不上 + TPS 上不去 + RT 隨壓力升**：這是「路不夠寬」——連接池、執行緒池、佇列、配置在卡。要先把路擴寬。
2. **資源用得上 + TPS 上不去 + RT 升**：這才是真的碰到硬體或演算法上限，要擴容或改程式碼。

先判斷自己是哪一類，再決定怎麼動手，能少走很多冤枉路。
