# L10 監控體系：APM、資源監控、應用層指標

**模組**：M4 執行與監控　|　**狀態**：草稿

## 學習目標

- 知道壓測時「客戶端數據」與「服務端監控」缺一不可。
- 建立三層監控：資源層、應用層、業務層。
- 會用 USE / RED 方法論決定要看哪些指標。

## 大綱

1. 開場：只看壓測工具的 RT/TPS = 半盲；要同時看服務端。
2. 三層監控：資源 / 應用 / 業務。
3. USE 方法（資源層）與 RED 方法（服務層）。
4. 全鏈路追蹤（trace）定位慢在哪一跳。
5. 收尾：壓測監控儀表板必備面板清單。

## 核心觀念

- **三層監控**：
  - 資源層：CPU、記憶體、磁碟 IO/IOPS、網路、檔案描述符。
  - 應用層：JVM GC、執行緒池、連接池使用率、佇列長度、快取命中率。
  - 業務層：各 API 的 TPS/RT/錯誤率、下單成功率。
- USE 與 RED 是兩個業界常用的「該看哪些指標」檢查口訣：
  - **USE 方法**（資源視角，Brendan Gregg）：每項資源看 **U**tilization 利用率 / **S**aturation 飽和度（佇列）/ **E**rrors 錯誤。
  - **RED 方法**（服務視角）：**R**ate 流量 / **E**rrors 錯誤 / **D**uration 延遲。
  - 例：用 RED 看下單 API，發現 Rate（TPS）上不去但 Duration（RT）飆高 → 確定有瓶頸；再用 USE 看 DB 連接池，Saturation（等待數）飆高 → 定位到連接不夠。
- **APM + 分散式追蹤**：一條請求跨多服務時，用 trace（traceId 串接）找出耗時集中在哪一跳——這是定位的起點（見 L13）。
- 客戶端 vs 服務端對照：壓測工具報 RT 含網路；服務端報處理時間。兩者差距大 → 網路/排隊問題。
- **監控設計（高樓）**：先畫架構圖 → 列組件 → 對每個組件定監控手段 → **先全局、後定向定量分層**（別一上來就幹到代碼層 SQL 執行時間）→ 由數據決定下一步收集什麼 → 串成證據鏈。性能團隊要**自己部署、看懂計數器**，不是只點別人搭好的鏈接。

## 實戰／案例

- 壓測工具顯示 RT 飆高，但服務端處理時間正常 → 瓶頸在連接排隊或施壓端網路，不是應用本身。
- 連接池打滿是高併發頭號隱形殺手：應用 CPU 沒滿、DB 也沒滿，但請求都在等連接——要監控**連接池使用率與等待數**。
- 儀表板必備：機器資源（USE）、API RED、JVM GC/堆、DB 連接池與慢查詢數、快取命中率、MQ 堆積。

## 書本參考
