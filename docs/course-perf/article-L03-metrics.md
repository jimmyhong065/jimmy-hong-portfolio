# 效能指標到底怎麼看才不會被騙？

「我們系統平均回應時間 80ms。」

這句話聽起來很健康，但它可能正在騙你。效能測試的第一門功課，就是看懂指標——而很多人連最基本的幾個都用錯、講混。

## 兩條主軸：快不快 vs 撐多少

所有效能指標，先歸到兩條軸上：

- **響應時間（RT）**：一個請求多快回來——衡量「快不快」。
- **吞吐**：單位時間能處理多少——衡量「撐多少」。

這兩件事不一樣，也不能互相取代。一個系統可以「每個請求都很快但同時撐不了幾個」，也可以「撐很多但每個都慢」。

## 響應時間：永遠看百分位，不要看平均

回到開頭那句「平均 80ms」。平均值的問題是，它被大量很快的請求稀釋掉了——少數幾個 3 秒的請求，藏在平均裡完全看不出來。

正確的做法是看**百分位**。白話解釋：把 1000 個請求的耗時排隊，

- **P95** = 第 950 快的那個（95% 的請求比它快）
- **P99** = 第 990 個（看「最慢的 1% 有多慢」）
- **P999** = 最慢的千分之一

大促下，「最慢的 1%」的絕對人數一點都不少。**只報平均，等於隱瞞問題。**

## 吞吐：TPS 才是關鍵，別亂用 QPS

吞吐這條軸，業界資深做法是**統一用 TPS（每秒事務數）**當關鍵指標。

這裡有個常見的混淆：QPS 本來是 MySQL 的「每秒查詢數（Query Per Second）」，後來被很多文章拿去描述系統吞吐，造成一堆誤會。建議就用 TPS，其他層級要講就加限定詞（例如「這個介面 1000 Calls/s」）。

TPS 裡的「事務（T）」要定義多大，取決於你的測試目標——可以是一個介面、一個完整業務流、或一次用戶點擊。**重點是團隊對「T 是什麼」有共識**，不然各說各話。

## 「並發 1000 人」這句話為什麼常常是錯的

這是最多人栽的地方。先分清兩個數字：

- **在線用戶數**：進到系統裡的人（包含只是掛著、沒動作的）。
- **並發用戶數**：同一時間真的有動作的人。
- 兩者的比例叫**並發度**，多數業務低於 5%，甚至低於 1%。

所以「我們有 1 萬用戶在線，要測 1 萬並發」這種推論是錯的。正確的換算是：

```
TPS        = 在線用戶數 × 並發度
並發線程數 = TPS ÷ (1000ms ÷ RT)
```

舉例：10000 在線 × 5% 並發度 = 500 TPS；如果 RT 是 100ms，那壓力工具只需要掛 `500 ÷ (1000/100) = 50` 條線程。

這背後就是 **Little's Law**——名字嚇人、其實很白話：**同時在處理的請求數 = 每秒進來幾個 × 每個待多久**。

也因此，**壓力工具的線程數 ≠ 真實用戶數**。對外溝通直接說「並發 500 TPS」就好，別糾結線程數。而且 RT 會隨壓力變動，這個比例不是固定的——所以要看**趨勢**，不是抓一個瞬間的數字。

## 還有兩個容易踩的坑

- **錯誤率要先看。** 系統開始回 HTTP 500 錯誤後，RT 反而會「變快」——因為失敗的請求秒回。不看錯誤率，你會誤判成「效能變好了」。**先確認成功率，再看延遲。**
- **別追「拐點」。** 很多人想在 TPS 曲線上找一個明確的轉折點，但多數系統的衰減是漸進的、沒有清晰拐點。要看的是「TPS 增幅何時開始變緩」，那就是容量逼近的訊號。

## 一份效能結果至少要報什麼

- 各關鍵交易的 TPS、錯誤率
- 響應時間的 P95 / P99（不要只給平均）
- 對應的資源利用率
- 測試的假設（流量模型、思考時間、暖機是否排除）
