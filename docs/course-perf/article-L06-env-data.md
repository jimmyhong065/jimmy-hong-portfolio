# 環境與造數：數字漂亮，不代表結論能用

壓測環境裡一切完美，上線就崩。其中一個最常見、又最隱形的原因是：**你的測試環境，跟生產根本不是同一個系統。**

## 環境等比的四個維度

要讓壓測結論能外推到生產，至少要在四個維度上對齊：

1. **規格**（CPU、記憶體、IOPS）：縮小的環境只能看趨勢，不能看絕對值。
2. **拓撲**（節點數、負載平衡、分片數）：單機壓測暴露不了分散式才會有的問題。
3. **資料量**：這是最常被忽略、殺傷力最大的一個（下面細講）。
4. **配置**（連接池、JVM 參數、快取大小）：必須跟生產一致，不然測的是另一套系統。

## 資料量是最大的隱形變數

我看過太多這種坑：測試庫只有幾千筆資料，查詢飛快，少了索引也沒事。一上線，億級資料表加上缺索引，同一支 SQL 從 2ms 變成 2s，連接池被慢查詢占滿，全站逾時。

**索引、查詢計畫的行為，在小資料量下根本不會現形。** 所以壓測前的造數，資料量級一定要對齊生產，而且要包含歷史資料。

造數還有幾個重點：

- 分佈要真實（含熱點與長尾），不要全部唯一或全部相同。
- 外鍵與關聯要完整。
- 別只造重複資料充數——那會造成假的快取命中或假瓶頸。

## 一次只改一個變數

這是效能測試的鐵律。一輪壓測裡，配置、程式、資料、流量——**一次只改一處**，否則效能變化你根本無法歸因。

一個常見的翻車：同時改了 JVM 參數又加了索引，效能變好了，但你不知道是哪個生效的，也就無法複製到生產。

## 別讓壓測機自己變成瓶頸

還有一個容易忽略的干擾源：**施壓端**。JMeter 單機開太多執行緒，瓶頸可能卡在壓測機自己（CPU 100%、執行緒爆），數字假低，你卻以為是被測系統的問題。壓測前先確認壓測機的 CPU、網路有餘量；規模大就用分散式壓測，或換協程/async 引擎。

同時，被測環境裡不能有其他旁路流量、排程任務在干擾。

## 壓測前環境檢查清單

- [ ] 規格、拓撲、配置盡量等比生產
- [ ] 資料量級對齊生產（含歷史資料、真實分佈）
- [ ] 施壓端本身有餘量，不會先成為瓶頸
- [ ] 環境乾淨，無旁路流量干擾
- [ ] 記錄環境快照（版本、配置、資料量），讓結果可重現

最後這點很關鍵：每輪壓測都記下環境快照，結果才**可重現、可比較**。這也是後面「執行紀律」會深入的主題。
