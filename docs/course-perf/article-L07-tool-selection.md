# JMeter 還是 k6？你可能問錯問題了

「我們壓測該用 JMeter 還是 k6？」

這是新手最常問的問題，但它本身就有點問錯方向。工具只是手段——你真正要的，是「能還原流量模型、能採到指標、能看懂計數器」。先回到目的，再來談工具。

## 選型的六個維度

挑工具，沿著這幾條軸想就好：

- 團隊的語言技能（GUI？JavaScript？Scala？Python？）
- 目標併發規模（幾百？幾萬？）
- 協定（HTTP、gRPC、WebSocket、訊息佇列？）
- CI 整合需求
- 報表需求
- 預算

## 四大工具對照

| 工具 | 腳本 | 引擎 | 強項 | 弱項 |
|------|------|------|------|------|
| JMeter | GUI / XML | 執行緒（每 VU 一條線程） | 生態大、協定多、GUI 好上手 | 高併發吃資源、需分散式、版控差 |
| k6 | JavaScript | Go 協程，高效 | 輕量高併發、CLI/CI 友善、腳本即程式碼 | 重 UI 流程弱、分散式雲要商業版 |
| Gatling | Scala DSL | async，高效 | 高併發省資源、報表漂亮、可版控 | Scala 學習曲線 |
| Locust | Python | 協程 | Python 寫腳本、易擴充、分散式簡單 | 單機吞吐略遜、需自備監控 |

## 執行緒 vs 協程：高併發下差很多

這是選型時最該懂的一個技術點。JMeter 是「每個虛擬用戶一條作業系統線程」，在超高併發下記憶體與切換成本會吃緊；k6、Gatling、Locust 用協程或非同步模型，同樣的硬體可以掛多很多的虛擬用戶。

## 一個經典翻車：瓶頸壓在壓測機自己

真實常見的坑：有人用 JMeter 單機壓 5000 併發，結果 TPS 上不去——一查，瓶頸根本卡在**壓測機自己**：CPU 100%、執行緒爆掉。數字假低，他還以為是被測系統的問題。

這提醒我們：施壓端本身的效率，決定了你能模擬多大的負載。高併發要嘛上分散式壓測，要嘛換協程/async 引擎。

## 怎麼選

- 團隊熟 JS、重 CI → **k6** 幾乎是預設首選（腳本即程式碼、跟 Grafana 整合好）。
- 要 GUI 給非工程師用、協定又雜 → **JMeter**。
- 要漂亮報表 + 高併發，又能接受 Scala → **Gatling**。

實務上一個好組合是：**k6 跑 CI 的常態效能回歸 + 全鏈路時用雲壓測平台補規模**。
