# L19 出報告：寫給 RD/PM 看懂的效能報告

**模組**：M6 調優與複盤　|　**狀態**：草稿

## 學習目標

- 把壓測數據寫成能驅動決策的報告，而非數字堆。
- 知道不同讀者（RD / PM / 主管）要看什麼。
- 會下「達標 / 不達標 + 風險 + 建議」的明確結論。

## 大綱

1. 開場：報告的目的是「讓人做決定」，不是炫數據。
2. 報告骨架：結論先行（BLUF）。
3. 分讀者：PM 看業務影響、RD 看瓶頸定位。
4. 必備欄位與圖表。
5. 結論寫法：達標判定 + 風險 + 行動建議。
6. 收尾：一頁式報告模板。

## 核心觀念

- **結論先行 BLUF**：第一段就講「能不能扛、達不達標、最大風險、要不要上」，細節後置。
- **分讀者**：
  - PM/業務：對照 SLO、換算成用戶體驗與 GMV 風險（多少人會等超過 X 秒、預估掉單）。
  - RD：瓶頸定位、trace、資源曲線、可重現參數。
  - 主管：go/no-go + 風險等級 + 需要的資源（擴容/工時）。
- **必備欄位**：測試目的與假設、環境快照、流量模型與加壓曲線、目標 SLO、實測結果（含百分位）、瓶頸定位、結論與建議。
- **必備圖表**：TPS-時間、RT 百分位-時間、錯誤率-時間、資源利用率-時間、TPS vs 併發（容量上限）。
- **高樓**：報告不能只給壓力工具的 RT 就下通過/不通過——要做響應時間拆分、把瓶頸定位到**具體計數器的值**。只測不調 = 性能驗證，非完整性能項目。
- **指標要誠實**：報 P95/P99 不只平均；報錯誤率；標註暖機排除與假設（think time、模型）。
- **結論三件**：是否達標（對 SLO 逐條）→ 殘餘風險 → 行動建議（擴容量、優化項、限流值）。

## 實戰／案例

- 反例：報告塞滿截圖沒結論，主管看不懂能不能上線——加一句「結論：當前配置可扛預估尖峰 1.2 倍，付款鏈 P99 280ms 達標；風險：DB 連接池為瓶頸，建議擴 X」。
- 把技術風險翻成業務語言：「DB 連接池在 1.5 倍尖峰耗盡 → 開賣 5 分鐘約 3% 用戶下單失敗 → 估掉單 NTD X」。
- 報告同時是**回歸基線**的素材（見 L12、L20）。

## 書本參考

《性能測試實戰30講》ch03、ch06：高樓主張報告要做時間拆分、瓶頸定位到計數器值，「只測不調 = 性能驗證」；數據定義（TPS/RT/百分位）回到 ch03。報告的「結論先行 / 分讀者」屬溝通實務，書外補充。
