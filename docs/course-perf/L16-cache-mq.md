# L16 快取／MQ：Redis 熱 key、穿透／擊穿／雪崩、MQ 堆積

**模組**：M5 瓶頸定位（核心）　|　**狀態**：草稿

## 學習目標

- 理解快取是高併發的主力卸載手段，也是新的故障點。
- 分清快取穿透、擊穿、雪崩三種失效模式與對策。
- 知道 MQ 削峰填谷的價值與堆積/消費問題的定位。

## 大綱

1. 開場：快取救你也害你——多一層就多一種失效。
2. 熱 key 與大 key 問題。
3. 三種失效：穿透 / 擊穿 / 雪崩。
4. 一致性：快取與 DB 的更新策略。
5. MQ：削峰、堆積、重複消費。
6. 收尾：快取/MQ 壓測關注點。

## 核心觀念

- **熱 key**：單一爆品 key 流量集中到 Redis 單分片/單節點 → 該節點 CPU/網路飽和。對策：本地快取 + 多副本打散 + 熱點探測。
- **大 key**：單 key 過大（大集合/大 value）→ 操作阻塞、網路放大。對策：拆分、避免大範圍操作。
- **三種失效**：
  | 模式 | 成因 | 對策 |
  |------|------|------|
  | 穿透 | 查不存在的 key，每次打到 DB | 空值快取、布隆過濾器 |
  | 擊穿 | 單一熱 key 過期瞬間大量打 DB | 互斥重建鎖、邏輯過期、永不過期+異步刷新 |
  | 雪崩 | 大量 key 同時過期/Redis 宕 | 過期時間加隨機抖動、多級快取、熔斷降級 |
- **快取一致性**：Cache-Aside（旁路）最常見；更新採「先改 DB 再刪快取」並容忍短暫不一致；強一致場景慎用快取。
- **MQ 削峰填谷**：把瞬時尖峰寫入轉異步，平滑後端壓力（秒殺下單常用，見 L18）。
  - 壓測關注：**堆積數**（生產 > 消費）、消費延遲、重複消費（冪等）、消息丟失。
  - 堆積暴增 = 消費端成瓶頸，要看消費者並發/處理 RT。

## 實戰／案例

- 雪崩翻車：活動商品快取統一設 30 分過期，半點同時失效 → 流量瞬間打穿 DB → DB 崩。加隨機過期 + 多級快取後解。
- 擊穿：秒殺爆品 key 過期那一刻萬請求同時重建快取打 DB，用互斥鎖只放一個重建。
- MQ 堆積：下單異步化後消費端 N+1 拖慢，堆積百萬，用戶遲遲收不到結果——擴消費者並發 + 修消費端慢點。
