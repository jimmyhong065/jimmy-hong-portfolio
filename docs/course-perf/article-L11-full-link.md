# 單服務全綠，全鏈路還是崩

每個服務單獨壓測都過關，為什麼串起來還是崩？

因為系統是個木桶，能裝多少水由**最短的那塊板**決定。訂單服務自己壓沒問題，但在真實的依賴鏈裡，下游的庫存服務、第三方風控可能先飽和，整條鏈一起逾時。這就是為什麼大促要做**全鏈路壓測**。

## 為什麼要在線上壓

理想上你想在等比的測試環境壓全鏈路。但現實是：要把規格、拓撲、資料量、第三方依賴全都等比複製，成本極高，而且仍有偏差。**只有生產環境，才有真實的一切。**

所以業界發展出「線上全鏈路壓測」——直接在生產環境壓，但要有一整套機制確保不污染真實資料、不影響真實用戶。

## 三個核心機制

**流量染色**：幫每個壓測請求蓋一個記號（放在 header 或上下文裡），並讓這個記號全鏈路透傳，讓沿途每個系統都認得出「這是壓測流量」。

**影子庫 / 影子表**：被染色的流量，寫入要導到「影子儲存」——結構跟生產一樣、但資料隔離。這樣壓測的寫入就不會污染生產資料。快取、訊息佇列、檔案也都要對應的影子隔離。

**流量隔離**：確保染色流量和真實流量在資料層徹底分開。

## 一個真實的木桶效應

訂單服務單壓 OK，但全鏈路一壓，下游庫存服務加上第三方風控先飽和，整條鏈逾時。如果只壓單服務，你永遠看不到這塊最短的板。

還有一種坑：染色沒透傳乾淨——某個中間件沒識別到記號，壓測的寫入污染了生產報表。所以上線前一定要**逐跳驗證透傳**。

## 風險控管：線上壓測不能莽

在生產壓測，安全機制要先到位：

- 壓測開關可以秒級關閉
- 限流閥值、熔斷保護都就位
- 排在**業務低峰時段**、有人**值守**、能**一鍵熔斷**
- 壓完要清理影子資料並核對

## 最難的其實不是技術

有句話講得很到位：在生產做壓測，**最重要的工作不是技術，而是組織協調能力**。要協調這麼多團隊、這麼多依賴方一起配合、一起值守，參與過的人都懂這句話的重量。
