# L14 DB 瓶頸：慢查詢、鎖、連接池、索引、分庫分表

**模組**：M5 瓶頸定位（核心）　|　**狀態**：草稿

## 學習目標

- 知道 DB 為何常是高併發電商的第一瓶頸。
- 會定位慢查詢、鎖競爭、連接池耗盡三類問題。
- 理解索引、讀寫分離、分庫分表的適用與代價。
- 懂隔離級別、行鎖/間隙鎖/Next-Key Lock 與 MVCC 如何左右高併發下的鎖競爭。

## 大綱

1. 開場：DB 是最難水平擴展的一層，常先倒。
2. 慢查詢：定位與索引。
3. 鎖與隔離：行鎖/間隙鎖/Next-Key Lock、隔離級別、MVCC 快照讀 vs 當前讀。
4. 連接池：池大小、等待、洩漏。
5. 擴展手段：讀寫分離、分庫分表、快取卸載。
6. 收尾：DB 瓶頸排查清單。

## 核心觀念

- **慢查詢**：開慢查詢日誌，用 `EXPLAIN` 看執行計畫；全表掃描、未用索引、回表多、排序/暫存表是常見元兇。資料量小時看不出，億級才爆（連 L06）。
- **索引**：缺索引→全表掃；索引過多→寫入變慢、占空間。注意最左前綴、選擇性、覆蓋索引；高併發寫的表索引要克制。
- **鎖（InnoDB）**：
  - 鎖類型：X（排他，讀寫都擋）、S（共享，S-S 兼容）；表級**意向鎖 IX/IS** 讓表級鎖能快速判斷可否加。
  - **行鎖競爭**：高併發改同一行（如熱門商品庫存）→ 序列化排隊（秒殺核心痛點，見 L18）；現象是 **TPS 卡死但 CPU 不高**。
  - **Record Lock 鎖的是索引**，不是記錄本身——**無適用索引時退化成鎖更大範圍甚至類表鎖**，是高併發隱形殺手（連索引設計）。
  - **間隙鎖 Gap Lock / Next-Key Lock**：InnoDB 在可重複讀（RR，MySQL 預設）下用「Record + Gap = 前開後閉 (a,b]」解幻讀，但**範圍 `FOR UPDATE`/範圍更新會鎖住間隙、阻塞插入**——壓測下表現為寫入排隊。
  - 死鎖：交易未按同一順序存取對象、交錯加鎖所致；觀察鎖等待與死鎖日誌，**事務內按固定順序存取、事務拆小**可減少。
- **隔離級別與 MVCC**：
  - 隔離級別越高鎖範圍越大；報表/讀多場景可降到提交讀（RC），別一律用預設 RR 或 Serializable。
  - **MVCC**：普通 SELECT 是**快照讀**（讀 Undo 版本、**不加鎖**），高併發讀用它卸載；INSERT/UPDATE/DELETE 與 `FOR UPDATE`、`LOCK IN SHARE MODE` 是**當前讀**（加鎖讀最新版）。壓測要分清讀寫路徑——**碰鎖的是當前讀，不是快照讀**。
- **連接池**：池太小→請求排隊等連接（偽裝成 App 慢，見 L13）；池太大→DB 連接數爆、上下文切換成本高。要監控**使用率 + 等待數 + 洩漏**（借不還）。
- **MySQL 全局監控（高樓 ch22）**：沒工具時，`SHOW GLOBAL STATUS;`（狀態值）+ `SHOW GLOBAL VARIABLES;`（配置值）絕對夠用；`information_schema`（表/列/索引/配置/狀態，`show processlist` 來源）、`performance_schema`（低代價收運行時資源消耗）。SQL 經 Connection Pool → SQL Interface → Parser → Optimizer → Caches → 磁碟，**先全局再定向**。
- **擴展手段與代價**：
  - 讀寫分離：卸載讀流量，但有主從延遲（讀到舊資料）。
  - 分庫分表：突破單機上限，但跨片查詢/交易/全域唯一鍵變複雜。
  - 快取卸載：讀熱點移到 Redis（見 L16），最常見且見效快。

## 實戰／案例

- 億級訂單表缺複合索引，活動查詢全表掃，單 SQL 從 2ms → 2s，連接池被慢查詢占滿，全站逾時。加索引後秒解。
- 庫存行鎖：秒殺單品所有請求改同一行，TPS 卡在個位數百，CPU 卻不高——典型鎖等待，要用減庫存優化（Redis 預扣/排隊，見 L18）。
- 連接洩漏：某路徑異常未歸還連接，壓測跑久連接數只增不減，浸泡測試才抓到（見 L01 浸泡）。
