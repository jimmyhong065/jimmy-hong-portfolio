# L09 加壓模型：階梯、尖峰、固定併發、容量探測

**模組**：M3 工具與腳本　|　**狀態**：草稿

## 學習目標

- 會依測試目的設計對應的加壓曲線（load profile）。
- 掌握階梯、尖峰、固定併發、容量探測四種曲線與用途。
- 會用 Little's Law 把目標 TPS 換算成 VU / arrival rate。

## 大綱

1. 開場：加壓曲線 = 測試目的的具體化。
2. 四種曲線逐一：形狀、用途、判讀。
3. 開放模型 vs 封閉模型（VU vs arrival rate）。
4. 暖機與 ramp-up 的必要性。
5. 收尾：四種測試型態對應的曲線速查。

## 核心觀念

- **階梯加壓 ramping/stepped**：每階維持一段時間取穩態指標 → 找容量上限（看 TPS 增幅衰減，別追單一拐點）。容量測試主力。**遞增必須連續、有梯度**，不可 100→200→300 斷開執行；遞增要與 TPS 遞增成比例，不能突然到上限（高樓 ch06）。
- **尖峰 spike**：流量瞬間階躍（如 0 → 滿載）→ 驗暴衝與恢復能力。對應雙11開賣。
- **固定併發 steady/constant**：維持定值跑長時間 → 負載/浸泡測試。
- **容量探測 breakpoint/stress**：持續加壓到劣化 → 找極限與失效樣態。
- **開放 vs 封閉模型**（用餐廳比喻最好懂）：
  - 封閉（固定 VU/併發數）：像店裡固定人數的客人，吃完一道才點下一道——系統慢時壓力會自動降低，**反而會掩蓋問題**。
  - 開放（固定 arrival rate 到達率）：像不管你出餐多慢、客人都按固定速率一直進門——**更貼近真實大促**（用戶不會因為你慢就不來）。
  - 大促秒殺建議用開放模型。
- **換算（高樓公式）**：並發線程數 = TPS ÷ (1000ms÷RT)。目標 2000 TPS、RT 100ms → 200 線程（封閉）；開放模型直接設 arrival rate = 2000/s。
- **性能衰減判讀（高樓）**：算「每線程每秒請求數 = TPS ÷ 線程數」，此值開始下降即瓶頸已出現（即使總 TPS 仍在升）。看趨勢，不必等拐點。
- **暖機 ramp-up**：JIT 編譯、連接池預熱、快取填充前的數據不可信，要排除暖機段；階梯中出現抖動 = 配置不合理（資源動態分配 or 數據未預熱）。

## 實戰／案例

- 用封閉模型測秒殺，系統一慢 VU 就「禮貌等待」，壓力自動縮，測不出真實雪崩——換開放模型才壓得出。
- 容量測試判讀：每加一階看 TPS 增幅是否衰減 + 每線程每秒請求數是否下降；**不必找單一「拐點」**（多數系統沒有清晰拐點），TPS 增幅明顯變緩處即接近容量上限。具體例（高樓 ch06）：24 線程時每線程每秒 75 個請求、72 線程降到 60、137 線程剩 36——這個「每線程產出」一路掉，就是性能在衰減的鐵證，TPS 大約就在此時觸頂。
- 必加 ramp-up + 排除暖機，否則前段假慢污染整體 P99。

## 書本參考

《性能測試實戰30講》ch06（線程遞增策略、性能衰減過程）：遞增必須連續有梯度、與 TPS 成比例；每線程每秒請求數下降即瓶頸已現；秒殺場景需**預熱後**才可線程突增。開放/封閉模型非書中主軸，屬本課補充。
