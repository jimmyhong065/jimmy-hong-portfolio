# L20 複盤常態化：CI 整合、效能回歸、看板

**模組**：M6 調優與複盤　|　**狀態**：草稿

## 學習目標

- 把一次性的大促壓測變成常態化的效能保障機制。
- 會把效能測試接進 CI/CD 做回歸防退化。
- 建立效能看板與大促複盤的閉環。

## 大綱

1. 開場：大促壓測不該是一年一次的消防演習。
2. 效能左移：把效能納入日常開發。
3. CI 整合：自動化效能回歸 + 閾值門檻。
4. 效能看板：基線 + 趨勢 + 告警。
5. 大促複盤：對照預測 vs 實況，回饋下一輪。
6. 收尾：常態化效能保障閉環。

## 核心觀念

- **效能左移 shift-left**：關鍵 API 在開發/PR 階段就跑輕量效能檢查，及早發現退化（比大促前才壓便宜千倍）。
- **CI 整合**：
  - 用輕量工具（如 k6，見 L07）在 pipeline 跑基準場景。
  - 設**效能門檻（quality gate）**：P95/錯誤率/吞吐超過閾值就 fail，擋住退化合併。
  - 與基線比較（見 L12），報「相對上次 ±X%」。
- **效能看板**：長期追蹤關鍵交易的 RT 百分位、吞吐、錯誤率趨勢；異常告警；留歷史基線可回溯何時開始退化。
- **大促複盤閉環**：
  1. 對照「壓測預測 vs 線上實況」——模型準不準、尖峰倍率對不對。
  2. 線上真實流量數據回饋，修正下一輪流量模型（見 L05）。
  3. 把這次的瓶頸與修復沉澱成 checklist 與回歸用例。
- 目標：效能成為**可持續的工程能力**，而非靠英雄式救火。

## 實戰／案例

- CI 攔退化：某次重構讓核心 API P95 從 120ms→400ms，pipeline 效能門檻直接擋下，沒進到大促才爆。
- 看板抓緩慢退化：每週 P99 漲一點，三個月後翻倍——趨勢圖才看得出「溫水煮青蛙」型退化。
- 複盤校準：上次預估尖峰 10×，實際 18×——下一輪模型倍率上修，壓測上限跟著調（連 L04/L05）。

## 書本參考

《性能測試實戰30講》ch29 與結束語：高樓強調「要有自己**可複用的分析邏輯**，而不是經歷多少個性能問題」——常態化的本質就是把每次複盤沉澱成可複用的 checklist 與回歸用例。CI/看板屬 DevOps 實務，書外延伸，理念一脈相承：效能靠持續測量與比較維持。
