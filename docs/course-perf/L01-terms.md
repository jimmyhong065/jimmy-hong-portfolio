# L01 釐清術語：效能 / 負載 / 壓力 / 浸泡測試差別

**模組**：M1 基礎心法　|　**狀態**：草稿

## 學習目標

- 講清楚 performance / load / stress / soak（spike、capacity）各自要回答什麼問題。
- 知道每種測試的「停止條件」與「成功判準」不同，不要混為一談。
- 面對「幫我壓一下系統」這種模糊需求，能反問出真正要測的型態。

## 大綱

1. 開場：一個常見誤會——「壓測」被當成所有效能測試的代名詞。
2. 用一張二維圖區分：橫軸=施壓強度，縱軸=持續時間。
3. 逐一定義五種型態 + 各自要回答的問題。
4. 對照表：目的 / 加壓方式 / 觀察重點 / 停止條件。
5. 收尾：電商雙11情境下，哪幾種要做、做的順序。

## 核心觀念

| 型態 | 要回答的問題 | 加壓方式 | 停止／判準 |
|------|-------------|---------|-----------|
| 負載測試 Load | 預期流量下表現如何？達標嗎？ | 加到「預期尖峰」並維持 | RT/錯誤率在 SLA 內 |
| 壓力測試 Stress | 極限在哪？崩潰怎麼崩？ | 持續加壓直到劣化 | 容量上限 / 失效樣態 |
| 尖峰測試 Spike | 瞬間暴衝扛不扛得住？ | 流量階躍式瞬增 | 暴衝後能否恢復 |
| 容量測試 Capacity | 還能再撐多少？何時要擴容？ | 階梯加壓找飽和點 | 吞吐不再上升 |
| 浸泡測試 Soak/耐久 | 長時間跑會不會壞？ | 中等負載跑數小時~數天 | 記憶體洩漏、連接洩漏、效能衰減 |

- 關鍵分辨：**負載**問「達標否」，**壓力**問「極限與失效樣態」，**浸泡**問「時間維度的劣化」。
- 容量上限：TPS 不再隨壓力上升、RT 開始陡升的區間。⚠️ 別執著找單一「拐點」——高樓指出多數系統的性能衰減是漸進的、沒有清晰拐點；瓶頸在最大 TPS 之前早已出現，看 TPS 增幅趨勢即可（見 L03/L09）。
- 浸泡測試最常抓到的不是 RT，而是**洩漏類問題**（記憶體、連接池、檔案描述符 FD），短時壓測看不到。
- ⚠️ 高樓的提醒：light/heavy/buckle 區或「性能/負載/壓力測試」這類分類，操作層面差別其實只有壓力工具的線程數，其餘都在資源分析層。他更愛用**場景**分類：基準 / 容量 / 穩定性 / 異常。本課按「要回答的問題」分型即呼應此意——別為分類而分類。

## 實戰／案例

- 踩坑：只做負載測試（壓到預期尖峰就收工），雙11當天遇到行銷推播造成的 spike 直接雪崩——少了尖峰測試。
- 反問需求的話術：對方說「壓一下」，要問：(1) 目標流量多少？(2) 是要驗達標、找極限、還是測長時間穩定？(3) 要不要模擬瞬間暴衝？
- 電商雙11建議順序：負載（驗 SLA）→ 容量（知道天花板與擴容點）→ 尖峰（驗行銷暴衝）→ 浸泡（驗活動期數小時穩定）。

## 書本參考

《性能測試實戰30講》ch01–02：高樓強調**性能必有「場景」**（含壓力策略、業務模型、監控模型、數據、軟硬體環境、分析模型），並反對把 light/heavy/buckle 硬對應到性能/負載/壓力測試。本課改以「要回答的問題」分型，呼應其「先想清楚場景目的」的立場。ch02 也釐清：關注服務端能處理的 TPS，不是壓力工具的線程數。
