# 只看壓測工具的數字，你等於半盲

壓力工具會告訴你「下單 API 變慢了」。但它告訴不了你「為什麼慢」——是 CPU 滿了？連接池滿了？還是在等資料庫？

只盯著壓測工具那幾條曲線，等於半盲。要定位瓶頸，**客戶端數據和服務端監控缺一不可**。

## 三層監控

服務端要分三層看：

- **資源層**：CPU、記憶體、磁碟 IO、網路、檔案描述符。
- **應用層**：JVM 的 GC、執行緒池、連接池使用率、佇列長度、快取命中率。
- **業務層**：各 API 的 TPS / RT / 錯誤率、下單成功率。

## 兩個好用的檢查口訣：USE 與 RED

監控指標太多，記不住該看什麼？業界有兩個「該看哪些指標」的口訣：

- **USE 方法**（資源視角）：每項資源看 **U**tilization 利用率 / **S**aturation 飽和度（排隊）/ **E**rrors 錯誤。
- **RED 方法**（服務視角）：**R**ate 流量 / **E**rrors 錯誤 / **D**uration 延遲。

串起來用就很有威力。舉個例：先用 RED 看下單 API，發現 Rate（TPS）上不去但 Duration（RT）飆高 → 確定有瓶頸；再用 USE 看資料庫連接池，Saturation（等待數）飆高 → 定位到連接不夠。

## 監控原則：先全局，後定向分層

設計監控有個容易犯的錯：一上來就把方法執行時間、SQL 執行時間全列出來，直接幹到代碼層，看起來很厲害。

資深做法剛好相反——**先全局、後定向定量分層**。先畫架構圖、列出所有組件，對每個組件找成本最低的監控手段，拿到全局數據後，再由數據決定下一步要深入收集什麼，一路串成證據鏈。

他還有一句話我很認同：**性能團隊要自己部署監控、看懂計數器**，不是點別人搭好的儀表板、看個數字就算。不然瓶頸真的出現時，你連那個數字代表什麼都說不清楚。

## 客戶端 vs 服務端：差距會說話

壓測工具報的 RT 含網路往返，服務端報的是純處理時間。兩者一比就有資訊：如果壓測工具顯示 RT 飆高、但服務端處理時間正常，那瓶頸就在連接排隊或施壓端網路，不在應用本身。

## 隱形殺手：連接池

最後講一個讓無數系統陣亡的隱形殺手。現象是：應用 CPU 沒滿、資料庫 CPU 也沒滿，但請求就是慢——因為它們全卡在「等一條資料庫連接」。

所以一定要監控**連接池的使用率與等待數**。這個指標不看，你會在「大家都很閒卻很慢」的迷霧裡繞很久。

## 壓測儀表板必備面板

機器資源（USE）、各 API 的 RED、JVM 的 GC 與堆、資料庫連接池與慢查詢數、快取命中率、訊息佇列堆積。
