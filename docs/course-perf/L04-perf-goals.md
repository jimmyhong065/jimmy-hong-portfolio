# L04 從業務目標推效能目標（SLA/SLO、預估 QPS 怎麼算）

**模組**：M2 需求與規劃　|　**狀態**：草稿

## 學習目標

- 把「系統要很快」翻譯成可驗收的數字目標（SLI/SLO/SLA）。
- 會用業務數據（DAU、轉化率、活動規模）反推目標 QPS / TPS。
- 知道效能目標要綁在「關鍵業務交易」上，而非整站平均。

## 大綱

1. 開場：「夠快」不是需求，可量測的閾值才是。
2. SLI / SLO / SLA 三者關係與差別。
3. 從業務推流量：DAU → 尖峰 QPS 的估算公式。
4. 大促修正係數：尖峰倍率、活動瞬時集中。
5. 把目標綁到關鍵交易（下單、付款、查詢）。
6. 收尾：一張「效能目標表」範例。

## 核心觀念

- **SLI**（指標）/ **SLO**（目標，內部）/ **SLA**（協議，對外含罰則）。壓測驗的是 SLO。
- SLO 寫法：`下單 API P99 < 300ms 且錯誤率 < 0.1% @ 尖峰 QPS`。要含**百分位 + 閾值 + 條件流量**三要素。
- **目標 TPS 估算**（高樓的換算才是主幹，見 L03）：
  - `目標 TPS = 在線用戶數 × 並發度`；並發度從**生產統計**得，多數業務 < 5%。
  - 拿不到生產數據時可粗估 `平均 QPS = 日請求數 / 86400` 再乘峰谷倍率，但**僅供占位**。
  - ⚠️ **別用「28 原則」直接算並發**（如「1000 萬用戶在 2 小時內完成」）。高樓明言：並發度由**業務**決定、不靠定律支配；沒做樣本數據分析就套 28 比例＝耍流氓。要嘛有真實統計，要嘛標明是假設。
- **大促修正**：日常尖峰 × 活動倍率（10~50×）；秒殺單品再疊瞬時集中係數。例：平日尖峰 200 TPS × 活動 15 倍 = 3000 TPS，秒殺單品開賣瞬間再 ×5，抓 15000 TPS 當壓測上限。
- 目標要**分交易**：讀（瀏覽/搜尋）與寫（下單/扣庫存）SLO 不同，混在一起會被平均掩蓋。
- ⚠️ RT 目標**別套「258 / 2510 原則」**（2 秒滿意、5 秒可接受…）。高樓考證其出處是 1980 年代英國音樂緩衝服務調查，過時 40 年。RT 標準應靠**同行對比 + 樣本用戶統計**訂。

## 實戰／案例

- 估算示例（標明假設）：預估活動日下單 100 萬筆，**假設**集中在約 1 小時高峰 → 平均 ≈ 278 TPS，再乘瞬時開賣係數定壓測上限。這只是缺數據時的占位估算；有生產統計就用真實並發度反推，別當定律。
- 踩坑：只定「整站 P95 < 1s」，結果付款這條關鍵鏈被大量靜態請求拉平均，真正的慢藏在裡面。
- 跟 PM 對齊話術：把 SLO 換算成「多少人會等超過 X 秒」「掉單損失多少 GMV」，業務才有感。

## 書本參考
