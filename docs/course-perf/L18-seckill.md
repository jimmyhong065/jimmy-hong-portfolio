# L18 秒殺專題：庫存扣減、超賣、排隊、預熱

**模組**：M6 調優與複盤　|　**狀態**：草稿

## 學習目標

- 把前面所有觀念綜合應用到最極端的秒殺場景。
- 理解超賣的根因與庫存扣減的高併發解法。
- 掌握秒殺的削峰、排隊、預熱、防刷整體架構。

## 大綱

1. 開場：秒殺 = 高併發的極限考題，集大成。
2. 超賣根因：併發讀改寫的競態。
3. 庫存扣減方案演進：DB 行鎖 → Redis 預扣 → 異步落庫。
4. 削峰排隊：MQ + 排隊頁。
5. 預熱與防刷。
6. 怎麼壓測秒殺（呼應 L09 開放模型 + L05 熱點）。
7. 收尾：秒殺壓測 checklist。

## 核心觀念

- **超賣根因**：`讀庫存→判斷→扣減` 非原子，高併發下多請求同時讀到同值 → 超賣。
- **庫存扣減方案**：
  | 方案 | 機制 | 取捨 |
  |------|------|------|
  | DB 行鎖 / 樂觀鎖 | `UPDATE ... WHERE stock>0` 原子扣 | 簡單但單行鎖序列化，TPS 受限（L14） |
  | Redis 預扣 | 原子操作（Lua/DECR）先扣快取 | 高 TPS，需與 DB 對帳 |
  | Redis 預扣 + MQ 異步落庫 | 快取扣減 + 異步寫 DB | 高吞吐、最終一致，主流方案 |
- **削峰**：排隊頁 + MQ 把瞬時百萬請求平滑成後端可處理速率（見 L16）。
- **排隊 / 限流**：入口限流（令牌桶）、單用戶單 SKU 限購、答題/驗證碼拉長時間分散尖峰。
- **預熱**：活動前把商品/庫存/頁面載入快取與 CDN，避免開賣瞬間冷快取擊穿（見 L16 擊穿）。
- **防刷**：限頻、風控、驗證碼，擋機器人放大真實尖峰。
- **壓測秒殺的前提（高樓）**：秒殺是少數允許**線程突增**的場景，但前提是**先做好系統預熱**；若模擬不出開賣瞬間的陡增，就是不合理場景。描述容量永遠用 TPS，不是壓力工具的線程數——線程再多，對已達 TPS 上限的系統只會增加 RT。

## 實戰／案例

- 超賣事故：用「先查再扣」兩步，壓測高併發下超賣 N 件；改原子 `UPDATE stock=stock-1 WHERE stock>0` 或 Redis Lua 解。
- 壓測秒殺要點：用**開放模型**（arrival rate，L09）灌瞬時尖峰、**熱點集中單 SKU**（L05）、驗證不超賣 + 排隊不雪崩 + 預扣與 DB 最終一致對帳。
- 預熱沒做：開賣瞬間冷快取，萬請求擊穿打 DB，DB 秒崩。

## 書本參考

《性能測試實戰30講》ch05–06：高樓的「秒殺需預熱後才可線程突增」「容量用 TPS 不用線程數」是本課壓測秒殺的方法依據。庫存扣減/超賣/MQ 削峰屬業界架構實務，本課把它們拼成綜合實戰場景。
