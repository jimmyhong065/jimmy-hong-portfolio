# 腳本不像真人，壓的就不是真系統

壓測腳本的目標，是模擬「真實用戶」。一旦腳本的行為跟真人差太遠，你壓的就是一個假系統，數字再漂亮也沒意義。有五個關鍵細節，做錯任何一個都會讓結果失真。

## 一、參數化：別讓所有人都是同一個人

帳號、SKU、地址這些要用資料池輪替。需要唯一的欄位（手機、訂單冪等鍵）要保證不重複，否則直接觸發唯一約束報錯。

參數化資料一般分兩類：

- 用戶輸入的資料**在資料庫裡已存在**（例如查訂單）→ 要符合真實的資料分佈。
- 用戶輸入的資料**不存在**（例如註冊新帳號）→ 要符合真實用戶的輸入。

還有一個常被忽略的：**造數要按真實生產分佈，不是按「壓測場景需要的量級」造。** 參數化做錯，後果可以很嚴重——業界有個經典案例：就因為參數化資料失真，TPS 在壓測中途直接陡減到底，降線程也只是延後出現。

## 二、關聯：動態值要抓出來接上

登入回傳的 token、下單回傳的訂單號，必須**動態擷取**後帶進後續請求。寫死的話，要嘛大量失敗，要嘛全部打到同一筆資料上。

## 三、思考時間：真人會停頓

真人不會無間斷地連發請求，中間會看、會想、會停。沒有思考時間（think time）的腳本，會高估併發壓力、低估系統能承載的虛擬用戶數。

實際影響有多大？我看過一個案例：加上 think time 後，「可承載 VU」從 5000 掉到 500。但這不代表系統變差了——是壓力模型不同。**所以報告一定要寫清楚假設。**

## 四、登入態：每個 VU 要有自己的身分

每個虛擬用戶要有獨立的 session / token，還要注意 token 過期要刷新。共用單一帳號會撞鎖、撞限流，壓出來的全是假象。

## 五、防快取：但別矯枉過正

如果所有請求都帶相同的查詢參數，會全部命中 CDN 或快取——QPS 衝得超高，但後端其實零壓力。要加隨機或真實分佈的查詢值來打散。

但也別反過來把快取命中率壓成 0，那同樣不真實。**目標是還原真實的命中率。**

## 兩個會騙過你的假象

- **假高吞吐**：所有請求帶同一個 query，快取全命中，QPS 漂亮但後端沒被壓到。
- **假低延遲**：token 寫死過期，請求全回 401 秒回，RT 看起來超快——其實錯誤率才是真相。這也是為什麼「先看錯誤率，再看延遲」這麼重要。

## 腳本上線前的檢查清單

- [ ] 參數化資料池夠大、分佈真實、唯一欄位不重複
- [ ] 動態值（token、訂單號）都有正確關聯
- [ ] 有合理的思考時間，且假設寫進報告
- [ ] 每個 VU 獨立登入態
- [ ] 快取命中率接近真實，不是全命中也不是全不命中
