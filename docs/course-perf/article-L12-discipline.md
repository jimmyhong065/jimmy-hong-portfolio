# 不可重現的壓測，就是不可信的結論

「這版怎麼變慢了？」「沒有啊，上次測比較快。」「上次配置是什麼？」「……忘了。」

這種對話無解。問題不在誰對誰錯，在於**缺了基線與記錄**——壓測結果不可重現，就什麼都證明不了。要讓結論站得住腳，有四個執行紀律。

## 一、基線：一切比較的錨

在一個已知良好的配置下跑一輪、記錄完整指標，這就是**基線**。之後每一次優化、每一次回歸，都跟它比。

沒有基線，你連「變快了」還是「變慢了」都說不出口——因為沒有對照點。

## 二、單變數：歸因的前提

一輪壓測裡，配置、程式、資料、流量——**一次只改一處**。

否則效能變化你無法歸因。最常見的翻車：同時改了 JVM 參數又加了索引，效能變好了，但你不知道是哪個生效，也就無法複製到生產。改了等於白改。

## 三、暖機與穩態：只取乾淨的數據

JIT 編譯、連接池預熱、快取填充之前的數據都不可信。要排除暖機段，只取系統進入穩態之後那段窗口的指標。

## 四、可重現三件套：每輪都要記

每一輪壓測，記下這三樣，結果才可重現、可比較：

1. **環境快照**：版本、機器規格、配置、資料量。
2. **測試參數**：流量模型、加壓曲線、腳本版本、思考時間。
3. **結果**：完整指標、監控截圖、原始數據存檔。

另外，數據多取幾輪的平均或中位，別用單次抖動下結論；也要留意環境裡有沒有旁路流量、排程任務在干擾。

## 落地成 SOP

把上面三件套做成一份固定模板：**環境 / 參數 / 結果 / 結論 / 下一步假設**，每輪壓測都產一份，存進版控或 wiki。

這份東西不只是紀律，它還會變成你之後寫報告、做效能回歸的現成素材——一魚多吃。
