# L15 App 瓶頸：執行緒池、GC、連接池、N+1

**模組**：M5 瓶頸定位（核心）　|　**狀態**：草稿

## 學習目標

- 定位應用層常見瓶頸：執行緒池、GC、下游連接池、N+1 查詢。
- 看懂 GC 日誌與執行緒 dump 的關鍵訊號。
- 知道哪些是程式碼問題、哪些是配置問題。

## 大綱

1. 開場：App 層問題常被誤判成「機器不夠」。
2. 執行緒池：大小、佇列、拒絕策略。
3. GC：停頓、頻率、記憶體洩漏。
4. 下游連接池：DB/Redis/HTTP client 池。
5. N+1 與序列化等程式碼級熱點。
6. 收尾：App 層排查清單。

## 核心觀念

- **執行緒池**：池太小→請求排隊；池太大→上下文切換 + 記憶體爆。看**活躍執行緒數、佇列長度、拒絕數**。用執行緒 dump（pstack/jstack）看執行緒卡在哪（等鎖？等 IO？）。
- **GC**：
  - 頻繁 Full GC / 長停頓 → RT 週期性尖刺（P99 鋸齒）。
  - 記憶體洩漏：老年代只升不降，最終 OOM——浸泡測試才顯形（見 L01）。
  - 看 GC 頻率、單次停頓、堆使用曲線；調堆大小/GC 演算法或修洩漏。
- **下游連接池**：對 DB/Redis/第三方的 client 連接池打滿 → 請求等連接（同 L13 連鎖偽裝）。每個下游都有自己的池要監控。
- **N+1 查詢**：一個列表請求觸發 N 次子查詢，併發放大後壓垮 DB；ORM 懶載入常見元兇。改批次查詢/JOIN/快取。
- **序列化/日誌**：高頻 JSON 序列化、同步寫日誌、鎖粒度過大也會成熱點——用 profiler 抓 CPU 熱點函式。
- **兩大類瓶頸現象（高樓 ch31）**：(1) **資源用不上 + TPS 上不去 + RT 隨壓力升** = 「路不夠寬」（連接池/執行緒池/隊列/配置限制）；(2) **資源用得上 + TPS 上不去 + RT 升** = 硬件/算法上限。先判哪一類再動手。

## 實戰／案例

- P99 週期性鋸齒：**GC 只是其中一種可能**。高樓 ch29 的鋸齒案例根因是週期性資源累積到臨界（定時清理/連接/隊列/網絡參數），降線程仍規律出現。規律性鋸齒先往「定時觸發/累積閾值」查，逐一比對才能定到底是不是 GC。
- 列表頁 N+1：單頁觸發 200 次 DB 查詢，QPS 一上來 DB 連接池爆，根因在 App 程式碼不在 DB（連 L13/L14）。
- 執行緒池拒絕：突發流量下池滿觸發拒絕策略大量失敗，需配合限流與佇列調整（見 L17）。
