# 「系統要很快」不是需求

PM 說：「這個功能要做到很快、不能卡。」

這不是需求，這是願望。效能測試開工前的第一件事，是把這種模糊的願望，翻譯成一個可以驗收的數字——否則你壓完了，也沒人能說「過」還是「不過」。

## SLI / SLO / SLA：先分清楚

- **SLI**（指標）：你拿來衡量的東西，例如「下單 API 的 P99 回應時間」。
- **SLO**（目標，對內）：你訂的目標值，例如「P99 < 300ms」。
- **SLA**（協議，對外）：跟客戶簽的承諾，沒達到要賠。

壓測驗的是 **SLO**。一條好的 SLO 要包含三件事：**百分位 + 閾值 + 條件流量**。例如：

> 下單 API 在尖峰 QPS 下，P99 < 300ms 且錯誤率 < 0.1%。

少了任何一塊，這個目標都不可驗收。

## 怎麼從業務反推目標 TPS

正確的主幹是用真實數據反推（詳見指標篇的換算）：

```
目標 TPS = 在線用戶數 × 並發度
```

並發度要從**生產統計**拿，多數業務低於 5%。拿不到生產數據時，可以粗估「平均 QPS = 日請求數 ÷ 86400」再乘一個峰谷倍率——但這只是占位，要標明是假設。

大促還要再修正。舉個具體的串法：平日尖峰 200 TPS × 活動倍率 15 倍 = 3000 TPS；秒殺單品開賣瞬間再 ×5，抓 **15000 TPS** 當壓測上限。

## ⚠️ 別信這些迷信

這部分我想特別講，因為太多人把沒根據的東西當定律在用。

**28 原則算並發**——「1000 萬用戶在 2 小時內完成業務，所以用 2 小時來算並發」。有經驗的效能顧問講得很直白：並發度是由**業務**決定的，不是靠這種定律支配。你沒做樣本分析就直接套 28 比例，跟耍流氓沒兩樣。要嘛有真實統計，要嘛老實標明這是假設。

**258 / 2510 響應時間原則**——「2 秒滿意、5 秒可接受、8 秒就跑掉」。這個說法的出處，其實是 1980 年代英國一家媒體對音樂緩衝服務做的調查，距今超過 40 年。拿它當現代系統的 RT 標準，完全不合時宜。**RT 標準應該靠同行對比 + 你自己的樣本用戶統計來訂。**

## 目標要綁在關鍵交易上

最後一個常見錯誤：只訂一個「整站 P95 < 1s」。

問題是，付款這條關鍵鏈的慢，會被大量很快的靜態資源請求拉平均，真正的問題被藏起來。正確做法是**分交易訂 SLO**：讀（瀏覽、搜尋）跟寫（下單、扣庫存）的目標本來就該不一樣。

## 怎麼讓 PM 有感

把技術目標翻成業務語言。與其說「P99 要小於 300ms」，不如說「這條鏈一慢，開賣 5 分鐘大概有 3% 的人下不了單，估計掉多少 GMV」。這樣 PM 才會真的把效能目標當回事。
