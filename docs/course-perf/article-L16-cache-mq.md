# 快取救你也害你

快取是扛高併發的主力——把讀流量從資料庫卸載掉。但要記得：**多一層快取，就多一種失效方式**。用好了它救你，用不好它害你。

## 熱 key 與大 key

- **熱 key**：單一爆品的 key 流量全集中到 Redis 的某個分片 / 節點，把那個節點的 CPU、網路打爆。對策：本地快取 + 多副本打散 + 熱點探測。
- **大 key**：單一 key 太大（大集合、大 value），操作會阻塞、網路被放大。對策：拆分、避免大範圍操作。

## 三種失效：穿透、擊穿、雪崩

這三個名字很像，但成因和對策完全不同：

| 模式 | 成因 | 對策 |
|------|------|------|
| 穿透 | 一直查一個不存在的 key，每次都打到資料庫 | 空值也快取、布隆過濾器 |
| 擊穿 | 單一熱 key 過期的瞬間，大量請求同時打資料庫 | 互斥重建鎖、邏輯過期、永不過期 + 異步刷新 |
| 雪崩 | 大量 key 同時過期，或 Redis 整個宕掉 | 過期時間加隨機抖動、多級快取、熔斷降級 |

一句話記：穿透是「查不存在的」，擊穿是「單一熱 key 過期」，雪崩是「大量 key 同時失效」。

## 一個經典雪崩

活動商品的快取統一設「30 分鐘過期」，結果半點一到全部同時失效——流量瞬間穿透到資料庫，資料庫直接崩。

修法很簡單：過期時間加上隨機抖動，讓它們不要同時死，再配多級快取。這也說明為什麼「統一過期時間」是個危險的設計。

## 快取一致性

最常見的是 Cache-Aside（旁路快取）：更新時「先改資料庫，再刪快取」，並容忍短暫的不一致。強一致的場景要慎用快取。

## MQ 削峰填谷：壓測該盯什麼

訊息佇列的價值，是把瞬時尖峰的寫入轉成異步，平滑掉後端壓力（秒殺下單常用）。壓測時要盯：

- **堆積數**（生產速率 > 消費速率）
- 消費延遲
- 重複消費（消費端要冪等）
- 訊息遺失
