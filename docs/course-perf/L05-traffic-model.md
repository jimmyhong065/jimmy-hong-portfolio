# L05 建流量模型：用戶旅程、讀寫比、熱點數據

**模組**：M2 需求與規劃　|　**狀態**：草稿

## 學習目標

- 把總 QPS 拆解成貼近真實的「請求組合」。
- 會用漏斗與用戶旅程推各 API 的相對比例。
- 還原熱點分佈與讀寫比，避免「均勻流量」的假測試。

## 大綱

1. 開場：總量對了、組合錯了，照樣測不出真問題。
2. 用戶旅程 → 漏斗 → 各步驟流量比。
3. 讀寫比：瀏覽/搜尋 vs 下單/支付的數量級差距。
4. 熱點數據：爆品 SKU、熱門活動頁的集中度建模。
5. 參數化資料的真實性（帳號池、SKU 分佈）。
6. 收尾：一張流量模型表（API : 比例 : 熱點策略）。

## 核心觀念

- **用戶旅程漏斗**：首頁 → 列表 → 商品頁 → 加購 → 下單 → 付款，每層轉化率遞減；流量比 ≈ 各層轉化率連乘。例：100 瀏覽 → 30 加購 → 10 下單 → 9 付款。
- **讀寫比**：電商常見 95:5 ~ 99:1（讀遠多於寫）。但**寫的代價高**（鎖、一致性），少量寫可能才是瓶頸。
- **熱點建模**：用權重分佈（80-20：少數品項佔掉多數流量）讓少數 SKU 拿走多數請求；秒殺場景甚至單一 SKU 集中。
- **參數化真實性**：帳號池要夠大避免單帳號鎖；SKU 池要照熱點權重抽樣；避免全測試打同一筆資料造成假瓶頸或假快取命中。
- 流量模型 = **比例 + 分佈 + 資料**三件事都要對。

## 實戰／案例

- 反例：所有 VU 都搜同一關鍵字 → 全打到快取，QPS 漂亮但根本沒壓到 DB。
- 反例：所有下單打同一 SKU → 行鎖爆掉，但這不是真實分佈造成的「假崩潰」（除非你就是要測秒殺單品，見 L18）。
- 做法：從生產日誌/APM 取真實 API 佔比與 SKU 訪問分佈，按比例還原；拿不到就用漏斗轉化率估，並**標明是假設**（別把 28 比例當定律，見 L04）。

## 書本參考

《性能測試實戰30講》ch14（如何理解業務模型）：高樓主張業務模型要從**生產環境統計**或**全鏈路壓測（流量複製）**得到，而非套定律。本課的「比例/分佈/資料」三要素即對應其「貼近真實負載」要求。
