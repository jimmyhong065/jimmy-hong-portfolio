# L06 環境與造數：環境等比、測試資料、變數隔離

**模組**：M2 需求與規劃　|　**狀態**：草稿

## 學習目標

- 知道測試環境與生產的差距如何讓壓測結論失效。
- 會做「等比」評估與資料量造數。
- 建立「一次只改一個變數」的隔離紀律。

## 大綱

1. 開場：環境不對等，數字再漂亮也不能外推。
2. 環境等比：規格、拓撲、資料量、配置四個維度。
3. 造數：資料量級、分佈、關聯完整性。
4. 變數隔離：壓測機、網路、旁路流量的干擾。
5. 收尾：壓測前環境檢查清單。

## 核心觀念

- **環境等比四維度**：
  1. 規格（CPU/記憶體/IOPS）——縮小環境只能看趨勢不能看絕對值。
  2. 拓撲（節點數、LB、分片數）——單機壓測無法暴露分散式問題。
  3. **資料量**——空表 vs 億級表的索引/查詢行為天差地別（最常被忽略）。
  4. 配置（連接池、JVM、快取大小）——必須與生產一致。
- **造數重點**：量級對齊生產（含歷史資料）、分佈真實（熱點/長尾）、外鍵與關聯完整、避免全唯一或全相同造成假象。
- **變數隔離**：
  - 壓測機（施壓端）本身不能是瓶頸——先驗壓測機 CPU/網路有餘量。
  - 同環境不能有其他旁路流量/排程任務干擾。
  - **一次只改一個變數**，否則無法歸因。
- 縮比環境換算要謹慎：效能非線性，不能簡單「機器數 ×N 推結論」。

## 實戰／案例

- 經典坑：測試庫只有幾千筆，查詢飛快；上線億級資料 + 缺索引，同一支 SQL 慢千倍。**資料量是最大隱形變數。**
- 施壓端瓶頸：JMeter 單機開太多執行緒，瓶頸在壓測機不在被測系統，數字假低。改用分散式壓測或 k6/Gatling 高效引擎。
- 紀律：每輪壓測記錄環境快照（版本、配置、資料量），結果才可重現可比較（見 L12）。

## 書本參考

《性能測試實戰30講》ch06（場景比對，只改單一因素才能歸因）、ch30（參數化/造數）：高樓強調造數要按**真實生產分佈**而非「場景需要的量級」，否則結果失真。本課把「資料量是最大隱形變數」與「一次只改一變數」整理成檢查清單。
