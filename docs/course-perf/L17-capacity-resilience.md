# L17 容量規劃：擴容、限流、降級、熔斷

**模組**：M6 調優與複盤　|　**狀態**：草稿

## 學習目標

- 從壓測結果推導容量規劃與擴容決策。
- 掌握限流、降級、熔斷三大穩定性保護手段。
- 知道「保護」比「全扛」更務實——大促要保核心。

## 大綱

1. 開場：沒有系統能扛無限流量，目標是優雅劣化而非雪崩。
2. 容量規劃：用容量上限 + 安全水位推擴容量。
3. 限流：閥值、演算法、層級。
4. 降級：非核心功能讓路。
5. 熔斷：快速失敗、防連鎖。
6. 收尾：大促穩定性保護組合拳。

## 核心觀念

- **容量規劃**：用容量測試找到單節點容量上限 TPS（見 L09，看 TPS 增幅衰減而非追拐點），設**安全水位**（常取 60~70% 利用率），反推所需節點數；預留大促 buffer。
- **限流 rate limiting**：保護系統不被打垮。
  - 演算法：固定/滑動窗口、漏桶、令牌桶（令牌桶可容忍突發）。
  - 層級：閘道層、服務層、熱點參數級（單用戶/單 SKU）。
- **降級 degradation**：高壓時關閉非核心功能（推薦、評論、非即時統計），把資源讓給核心鏈（下單付款）。可手動開關 + 自動觸發。
- **熔斷 circuit breaker**：下游故障/逾時達閾值即快速失敗，避免請求堆積拖垮自己（防連鎖雪崩）；半開試探恢復。
- 三者關係：**限流**擋過量入口、**熔斷**斷故障下游、**降級**捨非核心——大促前都要壓測驗證真的會觸發且有效。

## 實戰／案例

- 限流閥值要壓測校準：設太高沒保護、太低誤殺正常流量；用容量上限的安全水位定值。
- 降級沒演練 → 真出事時開關失靈或誤關核心。大促前做**降級演練**（混沌測試），驗證開關與影響面。
- 熔斷未設 → 第三方支付逾時，請求全堆在等待，執行緒池耗盡連鎖崩潰（連 L15）。

## 書本參考

《性能測試實戰30講》ch31（CPU / 磁碟 I/O 為最終常見瓶頸）支撐容量規劃的資源視角；限流/降級/熔斷屬高可用架構實務，書外延伸。動機「飽和即劣化」呼應高樓「TPS 達上限後再加壓只增 RT」的衰減觀。
