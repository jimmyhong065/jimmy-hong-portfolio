# L13 分層排查法：LB → App → DB → 中間件

**模組**：M5 瓶頸定位（核心）　|　**狀態**：草稿

## 學習目標

- 建立一套系統化的瓶頸定位方法論，而非亂槍打鳥。
- 會自上而下分層排查，用 trace 與資源指標縮小範圍。
- 知道「先定位再優化」，避免猜測式調優。

## 大綱

1. 開場：瓶頸定位是偵探工作，要方法不要直覺。
2. 總原則：先看現象（RED）→ 縮小層級（trace）→ 深入單層（USE）。
3. 分層地圖：客戶端 → LB/閘道 → App → 快取/中間件 → DB → 第三方。
4. 每層的典型瓶頸與第一個該看的指標。
5. 木桶與連鎖：一層飽和如何偽裝成另一層的問題。
6. 收尾：分層排查決策樹。

## 核心觀念

- **排查三步**：
  1. 現象層（RED）：哪個 API 慢/錯？錯誤率還是延遲？
  2. 定位層（trace）：用分散式追蹤看耗時集中在哪一跳。
  3. 深入層（USE）：對那一跳的資源看利用率/飽和/錯誤。
- **分層地圖與首查指標**：
  | 層 | 典型瓶頸 | 先看 |
  |----|---------|------|
  | LB/閘道 | 連接數上限、限流 | 連接數、5xx、佇列 |
  | App | 執行緒池/GC/連接池 | 執行緒池佔用、GC、池等待 |
  | 快取 | 熱 key、命中率掉 | 命中率、熱 key、RT |
  | MQ | 堆積、消費慢 | 堆積數、消費速率 |
  | DB | 慢查詢、鎖、連接 | 慢查詢、鎖等待、連接數 |
  | 第三方 | 限流、逾時 | 外呼 RT、錯誤率 |
- **連鎖偽裝**：DB 慢 → App 連接池被占滿 → 表面像「App 連接池不足」，根因在 DB。要順 trace 找**最上游的飽和點**。
- 原則：**先定位再優化**；沒有數據支撐的調優是賭博（見 L12 單變數）。
- **高樓的分析決策樹（招牌方法）**：從壓力工具的 **TPS / 響應時間 / 錯誤率三條曲線**判斷有無瓶頸 → **分段分層**（響應時間拆成 t1…t8，用 Nginx `$request_time`/`$upstream_response_time`、Tomcat 日誌或 SkyWalking 鏈路）→ 結合監控/日誌平台定位環節 → 對每個組件建專屬決策樹逐一拆解 → 找根因。能力階梯：工具操作 → 數值理解 → **趨勢/相關性/證據鏈分析** → 調優。
- **瓶頸判斷要細化到「具體計數器的值是多少」**：高樓的真實場景——瓶頸出現時 DBA 說 CPU 90% 沒問題、開發說是 DB 索引問題，各執一詞；唯有拿出執行計畫/計數器值才具體。**只測不調 = 性能驗證，不是完整性能項目。**

## 實戰／案例

- 表象 App CPU 不高卻很慢 → trace 顯示時間都耗在等 DB 連接 → DB 慢查詢拖垮連接池釋放 → 根因是缺索引（轉 L14）。
- 快取雪崩偽裝成 DB 崩潰：大量 key 同時過期，流量瞬間打穿到 DB（見 L16）。
- 養成順 trace 由現象往上游推，找到「第一個飽和的資源」才動手。
