# L08 腳本設計：參數化、關聯、思考時間、登入態、防快取

**模組**：M3 工具與腳本　|　**狀態**：草稿

## 學習目標

- 寫出「像真人」的壓測腳本，避免技術性假象。
- 掌握參數化、關聯（correlation）、思考時間、登入態維持、防快取五個關鍵。
- 知道哪些細節做錯會讓整份結果失真。

## 大綱

1. 開場：腳本不像真人，壓的就不是真系統。
2. 參數化：帳號池、SKU 池、唯一性需求。
3. 關聯：抓動態值（token、訂單號）串接請求。
4. 思考時間：還原真人停頓，影響併發換算。
5. 登入態：token/session 維持與刷新。
6. 防快取：避免全打快取的假高吞吐。
7. 收尾：腳本 review 檢查清單。

## 核心觀念

- **參數化**：帳號、SKU、地址等用資料池輪替；需唯一的欄位（手機、訂單冪等鍵）要保證不重複，否則觸發冪等/唯一約束報錯。高樓分兩類：(a) 用戶輸入的數據**在 DB 中已存在** → 要符合真實數據分佈；(b) **不存在** → 要符合真實用戶輸入。造數要按**真實生產分佈**，不是按「場景需要的量級」造（ch30 案例：參數化做錯，TPS 會陡減到底，降線程也只是延後出現）。
- **關聯 correlation**：登入回傳的 token、下單回傳的 orderId 必須動態擷取後帶入後續請求；寫死會大量失敗或打到同一筆資料。
- **思考時間 think time**：真人會停頓。沒有 think time 會高估併發壓力、低估可承載 VU 數；用 Little's Law 校準（見 L03）。
- **登入態**：每 VU 獨立 session/token，注意 token 過期要刷新；共用單帳號會撞鎖、撞限流。
- **防快取**：加隨機參數 / 真實分佈的查詢值，避免 100% 快取命中；但也別矯枉過正讓快取命中率為 0（不真實）——目標是**還原真實命中率**。

## 實戰／案例

- 假高吞吐：所有請求帶相同 query → CDN/快取全命中，QPS 衝很高但後端零壓力。
- 大量 4xx 卻沒人發現：token 寫死過期，請求全 401 秒回，RT 假低、錯誤率才是真相（見 L03 錯誤率優先）。
- think time 影響：去掉 think time 後「可承載 VU」從 5000 掉到 500，但這不代表系統變差，是壓力模型不同——報告要寫清楚假設。

## 書本參考

《性能測試實戰30講》ch09（關聯與斷言）、ch10（參數化）、ch30（參數化致 TPS 陡降）：高樓視關聯/參數化為「取數據」的核心，並用真實案例示範參數化失真如何讓整個場景失去意義。本課把這些具體化成五個可檢查項。
