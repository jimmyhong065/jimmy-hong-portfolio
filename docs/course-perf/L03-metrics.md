# L03 指標體系：RT、P95/P99、TPS/QPS、吞吐、併發數、錯誤率

**模組**：M1 基礎心法　|　**狀態**：草稿

## 學習目標

- 分清楚「響應時間」與「吞吐」兩條主軸，以及它們的關係。
- 會用百分位（P95/P99/P999）描述延遲，而不是只看平均。
- 懂在線用戶數、並發度、TPS、RT、線程數之間的換算（高樓公式，底層即 Little's Law）。
- 知道一份效能結果至少要報哪些指標。

## 大綱

1. 開場：兩條主軸——快不快（響應）vs 撐多少（吞吐）。
2. 響應時間：avg 為何不可信 → 百分位 → P99 的意義。
3. 吞吐：TPS / QPS 定義與差別。
4. 併發數：在跑的請求數，連結 Little's Law。
5. 錯誤率與飽和度（USE 概念帶過）。
6. 收尾：一張「效能結果最小報告欄位」清單。

## 核心觀念

- **響應時間 RT**：單請求往返耗時。**永遠看百分位不看平均**——平均被少數極值與大量快請求稀釋。
  - 白話：把 1000 個請求的耗時排隊，P95 = 第 950 快的那個（95% 請求比它快）、P99 = 第 990 個、P999 = 最慢的千分之一。看 P99 就是看「最慢的 1% 有多慢」——大促下這 1% 的人數也很可觀。
- **吞吐**：單位時間處理量。**高樓主張統一用 TPS 為關鍵指標**，其他層級加限定（如「接口調用 1000 Calls/s」）。
  - TPS（每秒事務數）：T 的粒度依場景目標定——接口級 / 業務級 / 用戶級，團隊須對 T 的定義一致。
  - QPS 本是 MySQL 的 Query Per Second，常被誤移用為吞吐；**不建議用 QPS 描述系統整體性能**。RPS / HPS / CPS 同理，要指明層級。
- **在線 vs 並發用戶數**：在線 = 進入系統者（含無動作者）；並發 = 同時有動作者；**並發度** = 並發 / 在線，多數業務 < 5%、甚至 < 1%。
- **核心換算公式（高樓，務必記）**：
  ```
  TPS        = 在線用戶數 × 並發度
  並發線程數 = TPS ÷ (1000ms ÷ RT)
  ```
  例：10000 在線 × 5% = 500 TPS；RT=100ms → 線程數 = 500 ÷ (1000/100) = 50 線程。這背後是 Little's Law——名字嚇人、其實很白話：**同時在處理的請求數 = 每秒進來幾個 × 每個待多久**。
- ⚠️ **並發線程數 ≠ 真實用戶數**。對外直接說「並發 500 TPS」，別糾結線程數。RT 隨壓力變、比例非固定——要看**趨勢**。
- **TPS 判容量、RT 判業務快慢**；別追單一「拐點」，看 TPS 增幅何時開始衰減（見 L09）。
- **錯誤率**：成功率 < 100% 時，RT/吞吐都失去意義（失敗請求往往很快返回，反而拉低 RT 製造假象）。**先看錯誤率再看延遲。**
- **飽和度 / 利用率**：CPU、連接池、執行緒池的使用率，逼近 100% 時延遲非線性暴增（佇列理論）。

## 實戰／案例

- 假象：壓測「平均 RT 80ms」很漂亮，但 P99 是 3s——大促下 1% 就是大量真人卡住。**只報平均 = 隱瞞問題。**
- 錯誤率陷阱：系統開始回 HTTP 500 錯誤後 RT「變快」，因為失敗請求秒回；不看錯誤率會誤判「效能變好」。
- 用換算設計壓測：要驗 2000 TPS、RT 目標 100ms → 線程數 = 2000 ÷ (1000/100) = 200 線程；VU 要足夠並加 think time 還原真實。
