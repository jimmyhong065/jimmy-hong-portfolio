# L02 為何雙11／秒殺難測：尖峰、長尾、流量模型

**模組**：M1 基礎心法　|　**狀態**：草稿

## 學習目標

- 說清楚電商大促的流量「形狀」跟平日完全不同，平均值會騙人。
- 理解三個難點：瞬間尖峰、熱點集中、長尾延遲被放大。
- 建立「測的是流量模型，不是流量總量」的觀念。

## 大綱

1. 開場：用「平日 vs 雙11」的流量曲線對比圖開場。
2. 難點一：尖峰——00:00 開賣那一秒，QPS 可能是平日均值的數十倍。
3. 難點二：熱點——流量不均勻，集中在少數爆品 SKU / 單一活動頁。
4. 難點三：長尾——高併發下 P99 被放大，平均值看起來還好其實已壞。
5. 難點四：強一致場景（庫存扣減）在高併發下的鎖競爭。
6. 收尾：所以測試要「複製形狀」而非「灌總量」。

## 核心觀念

- **平均值陷阱**：QPS 平均 5k 不代表平穩，可能是 00:00 衝 50k、其餘時段 1k。要看**峰值**與**峰值持續時間**。
- **尖峰倍率**：日常 → 大促尖峰常見 10~50 倍；秒殺單品開賣瞬間更可達百倍級的瞬時請求。
- **熱點集中**：80/20 甚至 95/5——少數爆品承受絕大多數流量，單點（單列、單 key、單分片）成為瓶頸。
- **長尾放大**：併發上升時排隊效應讓 P99/P999 先壞；只看平均/中位數會誤判系統健康（見 L03 飽和度）。
- **強一致熱點**：庫存扣減是「同一行資料的高併發寫」，鎖競爭讓 DB 成為最先倒下的點（深入見 L14、L18）。

## 實戰／案例

- 經典翻車：壓測用平均 QPS 灌均勻流量，全綠；上線開賣瞬間 spike + 熱點 SKU，DB 行鎖排隊，下單逾時雪崩。
- 熱點要刻意設計：壓測腳本不能對所有 SKU 平均分佈，要還原真實的爆品集中度（少數 SKU 拿走大部分請求）。
- 觀念金句：**「壓測要複製流量的形狀，不是它的總量。」**

## 書本參考

《性能測試實戰30講》ch02–03：高樓強調**平均值會騙人**（平均 QPS 5k 可能是 00:00 衝 50k、其餘 1k），要看峰值與峰值持續時間，並以「趨勢」看待指標變化。本課把這點落到「雙11流量形狀」——測的是形狀不是總量。（利用率逼近飽和時等待時間非線性暴增的佇列理論，可參考《圖解性能優化》第2章，未實際讀取。）
