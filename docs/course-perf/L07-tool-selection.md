# L07 工具選型：JMeter / k6 / Gatling / Locust

**模組**：M3 工具與腳本　|　**狀態**：草稿

## 學習目標

- 知道四大主流壓測工具的取捨維度與適用場景。
- 會依團隊技能、協定、規模、CI 整合做選型。
- 避免「為工具選工具」，回到測試目的。

## 大綱

1. 開場：沒有最好的工具，只有最合的。
2. 選型維度：語言/腳本、引擎效率、協定、報表、CI、學習曲線。
3. 四工具逐一比較。
4. 一張對照表 + 選型決策樹。
5. 收尾：電商高併發場景的建議組合。

## 核心觀念

| 工具 | 腳本語言 | 引擎模型 | 強項 | 弱項 |
|------|---------|---------|------|------|
| JMeter | GUI / XML | 執行緒（thread-per-VU） | 生態大、協定多、GUI 上手 | 高併發吃資源、需分散式、版控差 |
| k6 | JavaScript | Go goroutine，高效 | 輕量高併發、CLI/CI 友善、code-as-test | 重 UI 流程弱、商業版才有分散式雲 |
| Gatling | Scala DSL | Akka/async，高效 | 高併發省資源、報表漂亮、可版控 | Scala 學習曲線 |
| Locust | Python | 協程（gevent） | Python 寫腳本、易擴充、分散式簡單 | 單機吞吐略遜、需自備監控 |

- **選型維度**：團隊語言技能 / 目標併發規模 / 協定（HTTP、gRPC、WS、MQ）/ CI 整合 / 報表需求 / 預算。
- **執行緒模型 vs 協程/async**：thread-per-VU（JMeter）在超高併發下記憶體吃緊；協程/async（k6/Gatling/Locust）同硬體可掛更多 VU。
- 回到目的：要的是「能還原流量模型 + 採到指標」，工具只是手段。

## 實戰／案例

- 團隊已用 JS / 重 CI → k6 幾乎是預設首選（code-as-test、Grafana 整合好）。
- 需要 GUI 給非工程師、協定雜 → JMeter。
- 要漂亮報表 + 高併發又能接受 Scala → Gatling。
- 建議組合：k6 跑 CI 常態效能回歸 + 全鏈路用雲壓測平台補規模（見 L11）。
- 選錯的代價（真實常見）：有人用 JMeter 單機壓 5000 併發，結果瓶頸卡在壓測機自己——CPU 100%、執行緒爆掉，數字假低還誤以為是被測系統的問題。高併發要嘛分散式壓測、要嘛換協程/async 引擎（k6/Gatling）。

## 書本參考

《性能測試實戰30講》ch04（JMeter 和 LoadRunner：工具僅僅只是工具）：高樓強調工具只是手段，重點在能否還原場景、採到指標、看懂計數器——直接呼應本課「回到測試目的」與「引擎模型」維度。
