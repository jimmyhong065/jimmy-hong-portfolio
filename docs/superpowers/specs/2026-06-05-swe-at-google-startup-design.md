## 目標與受眾

**深度：** 窄而深，挑 3 個最有影響力的概念，每個都展開「新創為何難做到、怎麼小規模導入」

## 文章結構

### 標題（草案）
> 《Software Engineering at Google》讀完，我才知道新創少了什麼

### 開場鉤子
個人切入，符合部落格既有文風：一段對比場景——在新創工作時遇到的某個混亂時刻（code 沒人 review 直接上線、出 bug 沒有後續、測試全靠手動）。然後說「後來讀了這本書，才有語言描述那個混亂叫什麼」。

### 主體：三節，格式一致

每節固定結構：
1. **Google 怎麼做**（書中具體概念）
2. **新創通常怎麼做**（現實對比，帶同理不刻薄）
3. **為什麼很難改**（阻力，讓讀者有共鳴）
4. **最小化導入方式**（今天就能做的第一步）

---

## 三個核心概念

### ① Code Review 文化

**Google 怎麼做：**
每一行 code 上 production 前必須至少一個 reviewer approve。Reviewer 看的不只是 bug，還有可讀性、可維護性、設計一致性。書中核心觀念："code is a liability, not an asset"——code 越多，未來維護成本越高，每次 review 都是在控制負債。

**新創通常怎麼做：**
「先上再說，有空再 review」→ review 變橡皮圖章 → 最後沒人 review。

**為什麼很難改：**
速度壓力。老闆要快，工程師說 review 擋進度。

**最小化導入：**
定義一條 review checklist（3 條就夠），PR 沒填不能 merge。不需要工具，Notion 或 PR template 就行。

---

### ② Testing Grouplet（測試文化的組織推動）

**Google 怎麼做：**
不是靠規定要求寫測試，而是有工程師自願組成 Testing Grouplet，做內部教育、建工具、降低寫測試的摩擦。衍生制度：**Test Certified** 程式，讓團隊自我評估測試成熟度分級。

**新創通常怎麼做：**
沒人負責推測試文化 → 測試全靠個人自覺 → flaky test 沒人修 → 最後測試被關掉。

**為什麼很難改：**
沒有 slack（資源餘裕），每個人都在趕功能，沒人有時間「推文化」。

**最小化導入：**
一個人扮演 Testing Grouplet——不是寫所有測試，而是每週找一個測試痛點修掉，並分享給團隊。文化從行動開始，不從規定開始。

---

### ③ Blameless Postmortem

**Google 怎麼做：**
出事後 24-48 小時內寫 postmortem，格式固定：timeline、root cause、action items。關鍵原則：不寫人名在「誰造成的」欄位。問題是系統問題，不是人的問題。

**新創通常怎麼做：**
出事 → 口頭討論 → 找到兇手 → 沒有 action items → 同樣的事再發生。

**為什麼很難改：**
心理安全感不夠，寫下來怕被秋後算帳；主管沒有這個習慣，沒有人推動。

**最小化導入：**
第一次 postmortem 就算只有你一個人寫也值得——把 timeline 和 action items 寫下來，貼到 Slack，不點名。讓「寫下來」這個動作先存在。

---

## 結尾方向

不是「你要學 Google」，而是「這三件事不需要 Google 的規模，只需要一個人先開始」。呼應開場：當時那個混亂，其實有解，解法不複雜，只是要有人願意先做。
