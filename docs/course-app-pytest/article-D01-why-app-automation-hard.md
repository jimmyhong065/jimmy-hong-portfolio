# 為什麼 App 自動化比 Web 難

接過 Web 自動化的人第一次碰 App 自動化，通常在第三天就會有一個感受：「這怎麼這麼麻煩？」

不是工具爛，是環境本身的複雜度高出一個數量級。Web 的測試環境是瀏覽器，統一、可控、啟動快。App 的測試環境是裝置，多樣、易碎、啟動慢——每一條都是坑。

## 裝置與 OS 碎片化

Web 的主戰場是瀏覽器，Chrome 佔七成市場，主流版本差距不大，頂多加一條 Safari 就覆蓋絕大多數用戶。

App 的主戰場是裝置。Android 這邊光是要支援的 OS 版本就可能橫跨三到四個大版本，不同廠商又各自改了 AOSP——相同的 `click()` 動作，在三星 One UI 和原生 Android 上的行為細節可能不一樣。iOS 相對收斂，版本更新率高，但有自己的坑：每次 Xcode 大版本更新，XCUITest 的底層行為就可能悄悄變。

這意味著一支測試要在多組裝置上都穩定跑過，不能只靠一台模擬器驗收就放行。

## 原生元件與非同步渲染

Web 有 DOM，元件的存在與可見性是結構化、可查詢的。原生 App 沒有這層保障。

iOS UIKit 和 Android View 的渲染是非同步的——畫面還在組裝，accessibility tree 裡就已經有節點了，但它可能還不可互動。如果這時 `find_element` 找到了就立刻 `click()`，結果就是靜默失敗或 `ElementNotInteractableError`。

```python
# 踩過這個坑的人都知道：找到 ≠ 可以點
element = driver.find_element(AppiumBy.ACCESSIBILITY_ID, "submit-btn")
element.click()  # 畫面還沒穩，這裡可能丟 exception，或點了沒反應
```

這種行為在 Web 上少得多，在 App 上幾乎是家常便飯。`flaky test` 有很大比例的根因就在這裡——不是邏輯錯，是時機錯。

## 環境難搞、啟動成本高

起一個 Selenium session 大概一兩秒。起一個 Appium session，emulator 冷啟動要三十秒到一分鐘，安裝 APK 再加一段時間，加上 Appium server 本身的啟動——算下來光是「還沒跑任何測試」就過了快兩分鐘。

真機比 emulator 快，但真機有連線穩定性的問題，CI 環境更難管。雲端裝置農場解決了機器的問題，但多了網路延遲、帳號配額、計費壓力。每一種環境都有它的狀況。

這個啟動成本直接影響後面的設計決策：fixture 要怎麼設計才不會每個 test 都重啟 session？哪些 case 適合共用 session、哪些必須獨立？這不是「工程考量」，是「能不能跑完」的現實問題。

## 所以要工程化

正因為變數多，App 自動化比 Web 更沒本錢靠蠻力撐著。

裝置碎片化 → 測試本身要夠穩，不能讓裝置差異引爆 flaky。原生元件渲染時機不定 → 等待策略要到位，不能靠 `sleep`。啟動成本高 → fixture 要設計好，session 開銷要被控制在合理範圍內。多人協作 → 元件封裝（POM）讓定位邏輯集中在一處，改一次就好。

這門課的主軸就是這件事：用工程化的方法讓 App 自動化變得可維護。

後面幾篇會把環境搭起來、把 POM 建起來、把等待策略寫清楚——每一個設計決策，都可以對應回這裡說的某個難點。

## 帶得走

- App 自動化的難不在工具，在環境：裝置碎片化、原生渲染非同步、session 啟動成本高，三個難點互相疊加。
- 「找到元件」不等於「可以互動」——原生元件的渲染時機是 flaky 的主要根源，穩定的等待策略是必需品而不是加分項。
- session 啟動成本決定 fixture 設計，fixture 設計不好就是 CI 慢死的前因。
- 變數越多越需要工程化：POM、等待策略、好的 fixture——這不是選配，是 App 自動化能長期運作的基礎。
