## 測試債是什麼樣子的

根據 Capgemini《World Quality Report 2024-25》，測試活動佔企業 IT 預算的比例近年持續上升，然而超過六成的受訪組織仍承認測試覆蓋不足。花了更多預算，覆蓋卻依然有缺口——這個悖論的根源，就是無人追蹤、靜悄悄累積的測試債。

技術債大家都有概念：寫了一個 workaround，跑得動，但知道這不是正確的做法，遲早要回來修。

測試債的形式更隱形，因為沒有人把它記下來：

- 「這個功能先粗略測一下，下個 sprint 再認真補」
- 「這個邊界值情境先跳過，下次有時間再加」
- 「自動化案例一直沒時間寫，先靠手動撐著」
- 「這個模組的測試案例三年沒有更新，但功能已經改了很多次」

每一個這樣的決定，單獨看起來合理。但它們累積起來，就是測試債。

## 和技術債的差別

技術債有一個特點：它最終會讓開發速度變慢，讓 RD 感受到痛苦，讓主管看到效率下降。所以技術債到一定程度，工程團隊通常會主動推動 refactoring sprint 來還。

測試債不一樣，因為它的痛苦是延遲出現的，而且出現的時候不一定看起來像「測試債」的問題：

- Bug 逃逸率慢慢升高，但很難說清楚是哪個測試覆蓋缺口造成的
- 每次 release 手動回歸越來越長，因為自動化一直沒有補上
- 某個功能每次 release 都會出 bug，但沒有人知道測試案例其實沒有覆蓋那個邊界條件
- 測試環境越來越難維護，因為測試資料一直沒有清理

這些症狀分散在不同的地方，很難歸因到「測試債」。所以測試債可以累積很久，直到某次上線出了嚴重問題才被正視。

DORA 2024 的研究佐證了這個觀察：測試實踐偏弱的低效能交付團隊，其生產環境 bug 的修復時間中位數比高效能團隊長出數倍，而根本原因往往指向測試覆蓋的系統性缺口，而不是個別開發者的疏失。

## 我在 我們的 App 累積的債

那半年 我們的 App 的功能迭代很快。每個 sprint 都有新功能，我的優先級是測新功能，舊功能的測試案例就放著。

半年後，我回頭看：

**計時模組的測試案例是三個版本以前寫的。** 那個版本的計時邏輯只有一種計時模式，現在有四種。舊的測試案例沒有覆蓋新的模式，但因為核心邏輯沒動，大部分還是 pass 的。測試是綠的，但覆蓋是假的。

**訂閱相關的自動化測試一個都沒有。** 訂閱是最高風險的功能（涉及金錢），但因為 IAP 的自動化很難做，一直放著手動。每次 release 都要手跑一次完整的訂閱流程，花 40 分鐘。

**邊界值測試案例很零散。** 一開始想到哪個邊界就測哪個，沒有系統性地覆蓋。後來某次上線，「計時 0 秒完成」觸發了一個 bug，是我測試案例裡從來沒有測過的情境。

## 怎麼管理，不是怎麼避免

測試債無法完全避免，因為你永遠有優先級的取捨。問題是要讓它可見、可控。

**讓測試債可見**

我現在維護一個「已知測試覆蓋缺口」的清單。每次因為時間壓力跳過某個測試，我就把它記進去，加上風險評估：「這個缺口如果被觸發，影響是什麼？」

這個清單讓主管能看到當前的風險狀況，也讓 QA 自己在排優先級的時候有根據。

**還債要有策略**

不是每個缺口都要補。低風險、很少被觸發的情境，可以長期放在缺口清單裡。

高風險、或者已經因為沒有覆蓋而出過問題的情境，應該在下個 sprint 明確安排時間補。

我現在每個 sprint 保留 10–15% 的測試時間專門用來「還測試債」：補覆蓋缺口、更新過時的測試案例、把手動測試的高風險情境自動化。

**功能改動觸發測試案例 review**

規則是：任何功能的程式碼有改動，對應的測試案例要跟著 review，確認還是有效的。

這個習慣可以防止「測試是綠的但覆蓋是假的」的情況：功能改了，但測試案例還在測舊的邏輯，測試 pass 不代表現在的功能正確。

## 結尾

測試債最危險的特質是：它讓你對品質有一種虛假的安全感。測試都過了，看起來沒問題——但測試案例可能已經和實際功能脫節了。

還技術債有 refactoring sprint，還測試債有什麼？大部分的團隊沒有「測試債 sprint」，所以它就一直累積。

我認為可行的做法是：在每個 sprint 明確地分配一小塊時間還債，把它跟新功能測試一樣排進 sprint，不是「有時間再做」。有時間再做，就是永遠不做。
