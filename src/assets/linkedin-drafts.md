## 7｜同一個地方一直出 bug？
slug: qa-problem-solving-six-steps

修了又壞，壞了又修。

每個 sprint 都有人說「這個上次已經修過了」。

這不是 RD 不認真，是問題的根因從來沒有被找到。

我用的六步框架：

1. 還原現場（不只是重現 bug，是重建當時的狀態）
2. 分離變數（哪些條件必要，哪些是干擾）
3. 往上追一層（這個 bug 是哪個決策造成的？）
4. 往前追一層（在哪個環節可以更早發現？）
5. 找到共同根因（這個 bug 有沒有家族成員？）
6. 寫成文件（下次不用重新找）

## 43｜QA 說「測完了」，PM 說「還有這個沒測」

這對話重複了多少次？

根本原因：兩邊對「測完」的定義從來沒有對齊。

QA 的「測完」：所有 AC 都過了。
PM 的「還沒完」：邊界情況、跨瀏覽器、上版後的 smoke test。

這不是誰的問題，是 Definition of Done 沒有寫清楚。

**DoD 是團隊共識，不是 QA 的作業。**

每個 sprint 開始時把 DoD 攤開確認：

→ 哪些測試必須在 staging 上通過
→ 什麼叫「效能沒有退步」
→ 上版當天 QA 要做哪些確認

確認完才算對齊，否則「測完了」永遠是雙方各說各的。
