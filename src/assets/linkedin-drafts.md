## 1｜測試的問題每間公司都有，但為什麼每間公司都沒解決

「功能測試和效能測試混在一起跑」「QA 不懂 CI/CD」「大家以為 QA 等於測試」

這些問題，業界講了十幾年了。

為什麼還在重演？

不是因為大家不知道解法。而是因為——**這些問題的答案，從來沒有在對的時間出現在對的場合。**

功能的範疇在 PM 和 RD 討論完的時候就定了。
CI/CD 架構在 DevOps 設計完的時候就確定了。
QA 的職責在 JD 寫完的時候就被框住了。

QA 是在這些決策完成之後才出現的——然後被要求解決這些決策帶來的所有問題。

這是結構問題，不是能力問題。

## 2｜AI 時代，測試的 Domain Knowledge 比過去更重要

很多人以為 AI 出現後，QA 的價值會下降。

AI 可以生成測試案例，但它不知道：

## 13｜Playwright 進階：讓測試從能跑，變成團隊敢信任的東西

一個測試套件「能跑」和「讓人放心」是兩件完全不同的事。

- 每個測試都是一個大 monolith → Fixtures 拆乾淨
- 測試資料每次要手動準備 → APIRequestContext 直接打 API 建立

## 21｜測試債和技術債一樣可以累積，而且更難還

技術債大家都懂：code 寫快了，後面要花更多時間重構。

測試債很少人在說：

- 沒寫的測試
- 寫了但不可信的測試（永遠綠燈）
- 沒有維護的測試（只要系統一動就全壞）
- 沒有文件的測試（只有原作者知道在測什麼）

**測試債比技術債更難還，因為你甚至不知道虧了多少。**

技術債有 code review，測試債連度量都很難。

## 37｜探索測試怎麼說清楚「我測了什麼」？SBTM 完整指南

探索測試最常被質疑的一點：「你到底測了什麼？」

沒辦法回答，是因為缺少記錄方法，不是因為探索測試本身有問題。

SBTM（Session-Based Test Management）解決的就是這件事：

- 每個 session 有明確的 charter（我要測什麼）
- 時間盒（60–90 分鐘，專注不中斷）
- 結束後有 debrief（測了什麼、發現什麼、下一步）

讓探索測試從「說不清楚」變成「可追蹤、可討論」：[連結]

## 40｜Playwright 進階：storageState 讓 auth 只做一次

E2E 測試最浪費時間的事之一：每個測試都重新登入。
